# 第一部分：会议内容解析——老师说的 vs 我原来做的

## 1.1 老师明确说了什么（按主题归纳，尽量保留原话）

### A. 否定的点

**A1. 对话式为主体、满屏文字——明确否定，这是全场最重的一句话。**
【老师原话】"对话是不重要的……这个对话直接就生成语音了，你干嘛还要这么多文字打出来呢？"
【老师原话】"你先要进入角色，你作为张师傅现在就是在工厂……打字有没有可能？看一堆文字，现实吗？"
【老师原话】"以步骤式为主体，可以不用对话式为主体。"
【老师原话】"对话框只放右下角，小小的，它不作为主体。主体放你那个检修步骤。"
→ 判定：**对话式诊断台从"产品灵魂"降级为"右下角辅助入口"**。这直接推翻了历史 MVP V3 里"一切发生在对话流"的核心设计。

**A2. 诊断结果纯文字输出——否定。**
【老师原话】"它文字太多……你可以加点图标啊，加点这种实物的东西，生成一些模拟的图像。""加点三维模型的建模，能展现出来（故障）是在哪。"

**A3. 作业卡导出 PDF 打印这种"文字终点"——否定其局限性。**
【老师原话】"（导出 PDF 打印）还是局限了点儿，光想着整成这些文字的事儿了。你考虑的是要可视化呀。""交互式……一来一回，有点击有反馈，而且是多种形式的反馈。向导也不能全是文字。你拿着平板在现场，不可能（看大段文字）的。"

**A4. 瀑布式、文档先行、部署后置的开发方式——严厉否定。**
【老师原话】"你就生成一版，然后就放到龙芯上去运行一下，生成一版就运行一下。别等着生成了所有功能、那边都测得没问题了才跑龙芯上去试，结果一试怎么试都不行。"
【老师原话】"软工教给你的方法……当你的系统还很小的时候，就要把这些兼容性问题先解决掉。"
【老师原话】"别在这空想……赶紧动手做。拿到第一个可演示程序之后，你就自己知道下一个该弄什么了。""半天就一个迭代。""明天下午就能拿出一个东西来。"
【老师原话】"重设计轻实现（有整体设计是好事），但是不要停留在这个东西上边。"

**A5. "只做一个例子"的演示准备——否定。**
【老师原话】"怎么也准备个三五个（案例）嘛。现场的老师让你再换一个例子，你就说 OK，可以再换一个。""你就做了一个例子，现场专家说你再换一个，换不了了，拉倒了。"
【老师原话】"它不是只是一个演示 demo，它是感觉你做完了一个系统，把它做得真一点。答辩的时候，别说我这个东西就是简单做了。"

### B. 认可 / 保留的点

**B1. 工控设备方向本身——保留。** 老师全程沿用"工控机""工控设备"讨论，还亲自给系统命名：【老师原话】"系统你就（叫）工控设备异常检测与故障分析系统。"未提出换行业、换设备。

**B2. 知识库 + 历史案例回流——明确认可。**
学生复述"专家可以对这些记录进行修改……做成修改案例，下一次不仅检索知识库的内容资料，还有修改案例……形成知识库"，【老师原话】"啊，可以。是。"并在第二次谈话中确认"解决完了之后，不都进知识库吗"。

**B3. 多 Agent 分工——认可。** 学生提出大模型管理模块内含"接诊+知识检索 Agent、分析诊断 Agent、操作合规 Agent"，【老师原话】"行，可以。"

**B4. 整体功能规划——认可。**【老师原话】"你现在规划这些可以""行，这功能可以。"（附带要求：功能名称要用短语，不要一句话描述。）

**B5. 预置步骤/预置内容的做法——不但认可，而且是老师主动教的。**
【老师原话】"当然是要内置好的啊……让大模型给你想出来，然后做成内置好的。""肯定都是预制好的步骤流程，只不过你做成 3 个、5 个，看起来像是实时动态的。""你像华为他们开现场发布会，生成的结果不都是直接读的本地图片吗？"

### C. 新要求（老师希望换成什么）

**C1. 主体换成"检修作业向导"（步骤式）。**
【老师原话】"先做向导，先把那向导整出来。""主体第一步需要干哪些工作，第二步需要干哪些工作……每一个步骤具体要完成哪些任务，一项一项地打勾，一项一项地操作。"
向导是动态的（相对于老师发的固定五步视频案例）：【老师原话】"你做的是个动态的：第一步是用户描述现场的问题、以及读取工控机的一些信息，然后由大模型判断生成一个步骤（清单），用户确定一下这 4 步、5 步能不能解决当前问题；解决不了就让大模型再补一个步骤；解决得了就开始执行。"
步骤数要有变化：【老师原话】"步骤别都弄成 4 步 5 步的。有个 4 步的、5 步的、6 步的；4 步的还能加成 5 步，6 步的可以减成 5 步。"（→ 演示"动态增删步骤"的能力）

**C2. 可视化 / 三维定位是新的展示核心。**
【老师原话】（图书馆借书系统类比）"他会给你标示哪个书架、书的位置，甚至还有导航……你加到你那个展示里头，展示性就很强了。你找工控机，哪台工控机出问题了——这个显示效果太好了。"
【老师原话】"一个厂区的三维布局图，工控机的安装位置、那些工业设备的位置，详情展示、可视化。""这个不用你开发啥，大模型想办法给你生成的差不多的。"
【推断】老师要的不是真实三维引擎级建模，而是"看起来像三维厂区定位"的展示层——可以用大模型生成的图/建模素材 + 前端标注高亮实现。

**C3. 多模态输入输出升级：语音、视频进输入端；语音播报进输出端。**
【老师原话】"输入的地方，可以加入语音和视频。"
【老师原话】（蓝牙耳机播报）"现场带个蓝牙耳机过去……让评委老师带上你的蓝牙耳机，一步一步展示，走到哪一步，蓝牙耳机会有语音播报输出……语音提前预制好、生成好就是了。"

**C4. 连线专家（视频/语音会议）作为设计功能，模拟演示即可。**
【老师原话】"连线在线的专家、远程专家，能够发起视频会议或者语音会议，专家连进来。""那功能你可以设计上就是有个接口……做个演示就行了。他如果问你这个东西具体对接完了吗，你可以说这个是目前模拟的功能，可以对接第三方的在线语音即时消息接口（IM 接口）。"

**C5. 双端演示形态：平板（现场向导端）+ 龙芯（菜单式后台管理端）。**
【老师原话】"你去现场比赛也是一样的，要拿一个平板和龙芯去演示。""龙芯上主要展示那个菜单式的后台，然后拿你自己平板把那个（向导）打开，用平板演示更详细的功能。""跟谢文涛那个类似，你也得有个菜单式的后台。"

**C6. 功能分解图（三级树状）与七大模块——老师亲自定框架。**
系统名：**工控设备异常检测与故障分析系统**。模块（结合两次谈话）：
1. 异常检测模块——图像/视频/语音/文字输入；直接读取工控机数据信息。
2. 故障分析模块——知识库检索分析；内置专家 Agent 分析；视频/语音连线专家。
3. 工控机检修知识库——维修手册资料存放读取；知识图谱；历史维修案例（专家修改案例回流）。
4. 大模型管理模块——接诊与知识检索 Agent；分析诊断 Agent；操作合规 Agent；API 管理。
5. 可视化模块——诊断时部位/部件信息展示；维修时位置可视化；检索过程动态可视化。
6. 检修作业向导模块——平板上的步骤向导；语音提示。
7. 系统管理（后台）——用户管理、角色、部门、基础数据。
命名要求：【老师原话】"功能名字都是一些短语……你现在都是一句话的功能描述，显得不像功能点，再给它想个名字，用短语表示。"

**C7. 技术与部署要求。**

## 1.2 我原来做了什么（原方案基准 = 站控慧眼 MVP V3）

- **原产品定位**：油气管道场站工控设备多模态检修知识助手；"不是普通 AI 问答"的知识检索与作业辅助系统。
- **原设备方向**：研华 ACP-4000 / IPC-610 工控机整机（6 类故障方向），PLC 柜作关联节点。
- **原业务场景**：油气管道场站一线巡检/工控维护人员 + 专家审核员。
- **原核心闭环**：对话描述 → 槽位填充多轮追问 → 诊断上下文卡 → 一次 consult 出"双专家会诊"三段结论 → 生成作业卡 → 打印 PDF（主流程终点）；专家审核回流为附加分支。
- **原功能模块**：智能诊断台（对话式）、RAG 检索、多 Agent 会诊、知识图谱抽屉、标准化作业卡、专家审核回流。
- **原技术架构**：Vue/React 前端 + Flask 后端 + SQLite + ECharts 图谱 + 云大模型 API + 多模态识别 API + RAG；LoongArch + 银河麒麟适配；5 页面、9 接口、全流程大模型调用 ≤5 次；"一切皆内容，交互只剩按钮"；演示缓存（槽位哈希 key）。
- **原演示思路**：单浏览器两幕剧 4.5 分钟（一线闭环 → 专家回流金标），终点是打印 PDF。
- **原开发方式**：先把 PRD/MVP 文档打磨到 V3 定稿，再按 D1~D7 排期开发，最后适配龙芯（"龙芯空壳部署演练"虽列为第零优先级，但整体仍是文档先行）。

**十项重点判断逐条回答：**
1. 设备方向要换吗？——**不换**，仍是工控机。
2. 行业场景要换吗？——**不强制换**。老师说的是"工厂/厂区"，油气场站背景可保留【推断】，但演示话术宜说"工厂/场站"而非死扣油气管道。
3. 产品理念要换吗？——**要换**：从"检修知识助手（问答）"换成"异常检测与故障分析系统（向导+可视化）"。
4. 技术路线要换吗？——主干不换（前端 + Python 后端 + 云 API + RAG）；**交付方式换**（Node 开发态直跑、不打包）、**开发顺序换**（龙芯先行、迭代推进），前端主体从聊天流换成向导/可视化组件。
5. 只是表达和包装要换吗？——不止。交互主体、演示形态、开发方法都是实质改动。

## 2.1 老师为什么觉得现在方向不好

- **场景代入失真（核心）**：张师傅在现场戴着手套、站在机柜前，不会打字、不会读大段文字。对话式界面违背用户真实工况。（明确说）

- **开发节奏危险**：文档改了一版又一版，两天没有可运行程序；兼容性问题全部后置到龙芯，重蹈谢文涛队"兼容性搞三四周、到苏州当晚还在搞"的覆辙。（明确说）

## 2.2 老师真正想让我做什么（直接回答）

- **方向**：工控设备异常检测与故障分析系统——现场工程师用平板走"步骤式检修向导"，后台管理员在龙芯上用"菜单式管理系统"。
- **核心问题**：让一线人员在现场"看得见故障在哪、一步一步跟着做、听得见语音提示"，而不是"读 AI 写的文章"。
- **面向谁**：一线检修工程师（平板端）+ 后台/专家管理员（龙芯后台端）。
- **输入**：现场问题描述（文字/语音/拍照/视频）+ 工控机运行数据（读取或模拟接入）。
- **输出**：①三维/图示化的故障定位（哪台设备、哪个部件）；②动态生成（预置）的检修步骤清单，可增删步骤、逐项打勾执行；③每步的可视化图示 + 语音播报；④检修完成后案例入库；专家可修改案例反哺下次检索。
- **比赛亮点**：三维厂区定位、交互式向导、双端演示、蓝牙耳机语音播报、专家案例回流、龙芯真机运行。
- **架构体现**：七模块功能分解（三级树状图）、多 Agent 分工、知识库三件套（手册/图谱/案例）、云端大模型 API + 龙芯本地运行前后端。
- **怎么讲**：讲"完整系统"，不讲"简单 demo"；准备 3~5 个案例随时切换；模拟功能（连线专家）大方承认"预留第三方 IM 接口，当前为模拟演示"。
- **怎么应对评委**：评委要求换例子→立即切换；评委问某设备→"该厂区暂无此设备，但有 X 设备，我演示 X"；追问到底→承认模拟并说明接口设计。

## 2.3 理念上的更新（逐条标注依据）

1. **从"对话式 AI 助手"到"向导式作业系统"**——对话是喂给系统的输入方式之一，不是产品形态。（明确说）
2. **从"文字生成"到"多形态反馈"**——图标、实物图、三维定位、动画、语音播报；"交互式 = 有点击有反馈，且反馈是多种形式的"。（明确说）
3. **从"AI 实时生成"到"AI 离线生产 + 系统预置呈现"**——让大模型提前帮你生成步骤、图像、语音，内置到系统里；现场看起来是动态的。这是把 AI 从"演员"变成"编剧"。（明确说，华为发布会类比）
4. **从"单端网页"到"双端系统"**——平板前台 + 龙芯菜单式后台，后台承载知识库管理/用户角色/基础数据，撑起"系统感"。（明确说）
5. **从"瀑布文档驱动"到"迭代演示驱动"**——半天一迭代，每一版都上龙芯验证，兼容性问题前置。（明确说）
6. **从"AI 生成答案"到"专家审核 + 案例沉淀"**——这一条原方案已有，老师确认保留。（明确说）
7. 【推断】**从"技术叙事"到"体验叙事"**——答辩打动评委的第一层是"我看见了故障在厂区里的位置、我听见了语音指引"，第二层才是 RAG/Agent/图谱。

# 第四部分：重新整理技术架构

## 4.1 总体架构

- **前端（平板向导端）**：Vue 或 React 单页应用；**以 Node.js 开发态直接运行（npm run dev），不做编译打包部署**（老师明确指示）。主体组件：向导步骤引擎、厂区可视化视图（SVG/图像标注为主，ECharts/Three.js 视情况降级）、语音播放器、右下角对话助手、连线专家模拟窗。
- **前端（龙芯后台管理端）**：同技术栈的菜单式管理系统（侧边栏多级菜单）：知识库管理（手册/图谱/案例）、案例审核、用户/角色/部门、基础数据、大模型 API 配置。
- **后端**：Python（Flask 或 FastAPI，Flask 延续旧方案即可）；老师明确说 Python 容易且大模型代码生成质量高。
- **数据库**：SQLite（轻、免装、龙芯零负担）。
- **文件处理**：手册 PDF/文档解析入库（离线预处理为主）；图片上传压缩。
- **知识库**：三件套——手册 chunk（RAG 向量检索）、知识图谱（实体关系表 + 前端渲染）、维修案例库（结构化记录 + 专家修正标记）。
- **RAG**：云端 embedding API + SQLite 存向量（或轻量本地余弦计算）；检索结果带出处编号。
- **Agent/工作流**：三 Agent 按老师认可口径——接诊与知识检索 Agent、分析诊断 Agent、操作合规 Agent；实现上仍可"一次重调用输出多字段"以控成本【推断：老师未干预实现方式】。
- **专家审核**：后台案例审核页（列表 → 对照修改 → 通过入库 → 下次检索命中带"专家修正"标记）。
- **演示缓存**：所有大模型调用带缓存层；预置案例的步骤、图像、语音全部离线生成入库——现场演示可完全不依赖实时调用（网络兜底）。
- **部署适配（LoongArch + 银河麒麟）**：核心纪律 = **每完成一个迭代就上龙芯跑一遍**；首个上龙芯的版本是"只有主界面的空壳"；前端 Node 开发态运行、后端 Python 直跑、SQLite 免编译；避免需要源码编译的重型依赖（提前查龙芯镜像源可装清单）。

```text
 → POST /api/detect：异常检测归类（语音先转文字；图片走识别 API）
 → POST /api/analyze：检索知识库（手册 chunk + 历史案例，案例命中带修正标记）
    + LLM 分析（缓存优先）→ 返回 {故障判断、原因、定位坐标、安全项、推荐步骤方案ID}
 → 前端展示：厂区图上高亮故障工控机位置 + 分析结果 + "生成检修向导"按钮
 → GET /api/guide/:planId：取步骤方案 → 向导界面逐步执行
    （每步：部位图示高亮 + 说明 + 播放预制语音 + 打勾；

```

## 4.4 多模态还要不要

- **图片识别**：保留，作为异常检测输入之一（拍面板/机柜照片）；不再是唯一多模态卖点。
- **语音**：**地位大幅上升**——输入端语音描述（ASR API 或按住说话转文字）、输出端每步语音播报（**预制 TTS，蓝牙耳机演示，这是老师点名的演示桥段**）。
- **视频**：输入端做上传占位 + "连线专家"模拟视频窗即可，不做真实视频分析。
- **文档解析**：属于离线知识生产线，重要但不上演示台面。
- **主次判断**：多模态从"主卖点"变为"向导体验的组成部分"；**主演示功能 = 三维定位 + 步骤向导 + 语音播报**。
- **时间紧的降级序**：视频输入 → 语音输入（ASR）→ 图片识别，逐级砍；**语音播报（预制音频）成本极低，永不砍**。

## 4.5 知识图谱还要不要

- 定位：**辅助展示，不是核心亮点**——老师只把它列为知识库的组成部分，未展开任何要求。
- 程度：后台知识库管理下的一个图谱页（ECharts 力导图），节点=设备/部件/故障/案例；专家审核通过后加一条边（保留旧方案的"回流加边"演示点）。
- 真实 vs 静态：**半真实**——数据存表、前端真渲染，但实体关系为人工整理 + 大模型离线辅助抽取，不做运行时自动抽取。
- 防装饰化：把图谱嵌进两个真实用途——①分析结果页"影响范围"来自图谱查询；②审核通过后图谱可见地长出新边。

## 4.6 Agent 还要不要

- 老师要的形态：**工作流 Agent 分工**（接诊检索 / 分析诊断 / 操作合规），外加"连线真人专家"这个人机结合位——不是让 Agent 之间演对手戏。
- 角色设计（沿用老师认可的三分法，短语命名）：
  1. **接诊检索 Agent**——接收输入、抽取要素、检索知识库；
  2. **诊断分析 Agent**——故障判断、原因排序、步骤方案推荐；
  3. **操作合规 Agent**——安全项校验（仅引用安全规则库）。
- 实现：一次重调用输出三段结构化字段 + 前端按 Agent 身份分区展示（成本与旧方案一致）【推断：老师关心的是功能分解图上有这三块，不是真多进程】。
- 比赛展示：分析过程页用三个 Agent 头像分栏 + 检索动画（"哗哗哗"），后台大模型管理模块展示 Agent 调用日志——"看得见的分工"。

## 6.2 评委最可能问什么（18 问）

1. **和普通 RAG 问答有什么区别？**——答：交互形态上我们不是问答，是向导：输出的是可执行、可打勾、可增删的步骤和厂区定位，不是一段文字；机制上检索源包含专家修正过的案例库，且每条结论带出处编号。
2. **数据从哪来？**——答：公开的工控机维修手册与技术资料，人工整理 + 大模型辅助结构化；案例库为团队按真实故障模式构造并经指导教师审阅；全部脱敏、无企业涉密数据。
3. **怎么保证答案可信？**——答：三层——检索结论强制带出处；安全项只能引用安全规则库、模型无权生成；关键内容（步骤方案）走"大模型离线生产 + 人工审定入库"而非现场即兴生成。
4. **怎么解决幻觉？**——答：约束生成（仅基于编号证据）、结构化输出校验、预置审定内容托底；讲"幻觉不是靠 prompt 消除，是靠流程隔离"。
5. **为什么选这个场景？**——答：工控设备故障直接影响生产与安全；现场工况决定了"免打字、可视化、分步指引"的刚需，这正是大模型+可视化能补的空档。
6. **能不能真正落地？**——答：架构上已按落地设计——知识库可替换为企业手册、设备数据接口预留、后台含用户角色权限；当前为验证系统，落地差的是企业数据接入与现场硬件对接。
7. **有没有接入真实设备？**——答：当前用模拟数据面板演示"读取工控机信息"，接口按真实点位数据格式设计（诚实 + 显示设计能力）。
8. **知识图谱是真的还是展示用的？**——答：数据真实存储、前端真实渲染，且有两个真实用途：影响范围查询、审核回流后图谱新增边（可现场演示长出一条边）；实体关系为人工+离线辅助构建，不做运行时抽取——如实说明。
9. **Agent 是真协同还是提示词包装？**——答：坦诚定位为工作流分工而非自主智能体：三个 Agent 有各自输入约束和输出契约（合规 Agent 只能引用安全规则库），后台可查每次调用日志；这种可控分工恰是工业场景需要的。
10. **怎么适配 LoongArch？**——答：讲方法论——第一天空壳上机，每迭代必部署，兼容性问题前置；技术选型全部避开重编译（Node 开发态、Python、SQLite）；出示逐迭代部署记录。
11. **为什么不用本地大模型？**——答：实测龙芯单核跑本地模型速度不可用（有同学验证）；架构上模型层已抽象，算力允许时可切换本地模型，当前云 API + 全量缓存保证演示不依赖网络。
12. **现场网络不好怎么办？**——答：预置案例全部内容（分析、步骤、图、语音）离线入库，主流程零网络依赖；这是设计目标不是补丁。
13. **怎么保证安全（只读不控）？**——答：系统对设备只读不控，检修操作由人执行、系统只做指引；安全项由规则库硬性注入。
14. **企业资料涉密怎么处理？**——答：演示数据全脱敏；落地形态支持私有化部署（后端与知识库全在内网，仅模型调用可替换为内网模型）。
15. **专家回流有效性怎么证明？**——答：现场演示——专家修正某案例后，重发同类故障，结果命中带"专家修正"标记的内容，前后对比可见。
16. **换一个故障演示一下？**——答：好的（切换预置案例，这正是准备 3~5 个的原因）。
17. **连线专家是真的吗？**——答：老师给的标准话术——"当前为模拟功能，系统预留了对接第三方在线音视频/IM 即时通信接口的设计。"
18. **步骤是实时生成的吗？**——答：步骤方案由大模型生产、人工审定后预置入库，现场按诊断结果匹配调用并支持动态增删——工业指令类内容必须经过审定，这是可靠性设计而非能力妥协。
