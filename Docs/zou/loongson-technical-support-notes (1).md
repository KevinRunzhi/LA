# 龙芯（LoongArch）技术支持与部署总结

> 日期：2026-07-06
> 来源：两次老师谈话转写（老师给出的技术指示）+ 历史对话中的部署相关内容 + 据此整理的实操建议。
> 标注：【老师指示】= 老师明确说的；【实操建议】= 据此推导的做法，需在真机上验证。

---

## 一、老师给出的技术指示（原话级）

1. **前端不打包，开发态直跑**：
   【老师指示】"前端项目你直接用 Node（开发）环境去启动就好了，别弄成部署的 web。工程完整的项目要部署的话，都是编译成 JavaScript/HTML/CSS 打成包才能部署——你就别考虑了，又多一步。"
   → 结论：龙芯上装 Node.js，`npm run dev` 方式运行 Vue/React 项目即可。

2. **后端选 Python**：
   【老师指示】"后端都行，Python 会容易点，大模型写 Python 也写得相对好一点。Python 配环境比较简单。"

3. **大模型不做本地部署**：
   【老师指示】"（龙芯上）本地部署不行。"佐证：杨慧威在龙芯本地跑通过大模型，"很慢、特别慢、单核"，仅证明部署可行、性能不可用。
   → 结论：云端 API（学生提到质谱 GLM，老师未反对）+ 全量缓存 + 预置内容兜底。

4. **迭代式部署纪律（最重要的一条）**：
   【老师指示】"你就生成一版，然后就放到龙芯上去运行一下，生成一版就运行一下。别等着所有功能都测得没问题了才跑龙芯上试，结果怎么试都不行。"
   【老师指示】"先生成一个带主界面、啥功能都没有的演示程序，挪到龙芯上让它跑起来，跟你这边效果一模一样，再接着做下一步，一个模块一个模块推进。"
   【老师指示】"当你的系统还很小的时候，就要把兼容性问题先解决掉；知道可能有哪些兼容性问题之后，开发的时候就直接避免这些问题。"

5. **排错方法**：
   【老师指示】跑不起来时看终端报错（"肯定有报错的，终端那里"），前端用 Node 环境启动便于看到报错，不要用 IDE 一键部署方式。

6. **技术选型原则**：
   【老师指示（转述学生计划获认可）】开发前先查 LoongArch 大概支持什么技术栈；设计阶段就避开龙芯镜像源装不了、需要源码编译的包（学生自述不熟源码编译，老师未要求学）。

7. **开发环境路径**：学生计划"先在 WSL Ubuntu 上开发，再放到龙芯虚拟机适配"，老师认可，但强调**不要等全部做完才适配**——每一步都同步验证。

## 二、推荐技术栈（兼容性优先）

| 层 | 选型 | 理由 | 龙芯风险 |
|---|---|---|---|
| 前端框架 | Vue 3（或 React）+ Vite 开发态 | Node 生态纯 JS，无本地二进制编译依赖为佳 | 注意个别依赖含原生模块（见第三节黑名单思路） |
| 前端运行 | `npm run dev`（Vite/webpack dev server） | 老师指示，免打包 | dev server 内存占用需观察 |
| 可视化 | SVG 标注 + CSS 动画为主；ECharts 备用 | 纯 JS 渲染，无 WebGL 硬依赖 | Three.js/WebGL 在龙芯浏览器上性能未知——**先做 SVG 方案，Three.js 仅在真机验证后考虑** |
| 后端 | Python 3 + Flask | 老师认可；依赖轻 | 优选纯 Python 包，避开需编译的 C 扩展 |
| 数据库 | SQLite | 免安装服务、Python 内置驱动 | 无 |
| 向量检索 | 云 embedding API + 自实现余弦（numpy 或纯 Python） | 避开 faiss 等需编译库 | numpy 在龙芯源通常有包，若无则退纯 Python 实现（数据量小可行） |
| 大模型 | 云 API（GLM 等）+ 本地缓存层 | 老师认可路线 | 现场网络不可靠 → 预置内容全覆盖 |
| TTS 语音 | 开发机上预生成 mp3/wav 文件，龙芯只做播放 | 零龙芯侧依赖 | 平板播放则连龙芯音频都不依赖 |
| ASR 语音输入 | 云 API 或浏览器 Web Speech（平板端） | 在平板浏览器执行，不占龙芯 | 需平板浏览器实测 |

**总原则：凡是需要"编译"的东西都不要出现在龙芯上——前端不编译打包、Python 不装 C 扩展、模型不本地推理、音频不现场合成。**

## 三、部署实操清单（按迭代排）

### 迭代 0（拿到虚拟机当天~48h 内）：空壳上机
- [ ] 登录龙芯虚拟机，确认系统版本：`uname -a`（应为 loongarch64）、`cat /etc/os-release`
- [ ] 确认/安装 Node.js：`node -v`（记录龙芯源可装的版本号）
- [ ] 确认/安装 Python3 + pip：`python3 -V`
- [ ] 起一个 Hello 级 Vite 项目 `npm run dev --host` → 局域网内平板浏览器访问验证
- [ ] 起一个 Hello 级 Flask `app.py` → curl 验证
- [ ] 记录：镜像源地址、可装包清单、遇到的每个报错与解法（这份记录就是答辩材料）

### 迭代 1~N：每完成一个功能模块
- [ ] 代码同步到龙芯（git pull 或 scp）
- [ ] 前端 dev server 重启、后端重启，走一遍当次新增功能
- [ ] 与 WSL 开发机效果比对，"跟这边效果一模一样"才进入下一模块（老师标准）
- [ ] 新增依赖当天就在龙芯上 `pip install` / `npm install` 验证，装不上立即换库

### 演示前专项
- [ ] 龙芯 ↔ 平板同网段联通与 IP 固定（平板浏览器访问龙芯上的前端 dev server 或直接访问后端）
- [ ] 断网演练：拔网线走完整主流程（验证预置内容兜底）
- [ ] 蓝牙耳机连接平板播放预制语音实测
- [ ] 开机自启动或一键启动脚本（避免现场敲命令）
- [ ] 一键重置演示数据脚本

## 四、已知坑与规避（谈话与历史对话汇总）

| 坑 | 来源 | 规避 |
|---|---|---|
| 兼容性问题堆到最后爆发（往届三四周、决赛前夜还在修） | 谢文涛队教训 | 迭代 0 空壳先行 + 每迭代必上机 |
| 龙芯镜像源缺包 → 被迫源码编译（团队不会） | 学生自述担忧 | 选型阶段查源；装不上立即换等价纯 Python/纯 JS 库 |
| 本地大模型单核极慢 | 杨慧威实测 | 云 API + 缓存 + 预置 |
| 前端打包部署链路复杂 | 老师指出（课程未教部署） | dev 态直跑 |
| 跑不起来无报错提示 | 学生困惑 | 终端启动看 stderr；前端用 node 起而非 IDE |
| 现场网络不可靠 | 常识 + 历史对话兜底设计 | 预置案例全内容离线；缓存 key 按状态哈希 |
| 平板与龙芯联通未演练 | 新增双端形态的风险 | 演示前专项第 1 条，尽早演练 |
| 龙芯浏览器 WebGL/性能未知 | 新增可视化的风险 | 可视化走 SVG/CSS；重展示放平板端，龙芯只跑后台页面 |

## 五、与老师约定的开发节奏

- 第 1 周：**检修向导**（含 1 个完整案例）——老师钦点先做；期间完成迭代 0 空壳上机。
- 第 2 周：**菜单式后台**（案例审核回流 + 知识库管理框架）。
- 节奏纪律：半天一迭代；"明天下午就能拿出一个东西"是老师的期望速度；每版发老师看，不攒一周。
- 求助纪律：拿不准就随时问老师，"别等着一周攒一周"。
