# 工程师检修完成记录与案例回流合并 Spec

版本：V1.0
日期：2026-07-13
状态：已实施（WSL），龙芯待验证
适用范围：工程师侧检修完成、案例上传、专家待审核衔接

> 本文档是工程师侧案例回流页面组织的最新执行基线。与 `backend-case-feedback-development-spec.md` 中“工程师案例确认页”“案例回流工作台”冲突时，以本文档为准。专家三步审核、SQLite 知识发布和工程师知识同步继续沿用原 Spec。

## 1. 本次修订结论

本次只冻结两项产品调整：

1. 工程师侧不再保留独立的“案例回流工作台”。“案例回流”不再是一张页面，而是检修完成记录上的最终业务动作和状态。
2. 将原案例回流所需的确认、证据、恢复数据、上传和状态反馈全部合并进“检修完成记录”页面。点击“上传至专家知识库”成功后，即视为工程师侧案例回流完成，案例进入专家账号的待审核案例库。

最终主线固定为：

```text
现场接诊
  -> 诊断与检修预方案
  -> 执行检修与恢复验证
  -> 检修完成记录
  -> 上传至专家知识库
  -> 工程师侧完成
  -> 专家独立登录审核
  -> 修订案例、知识与图谱
  -> 发布正式知识
```

## 2. 修订原因

当前“检修完成记录”只有设备、故障、步骤数量和专家状态四项摘要，存在以下问题：

- 更像流程结束提示，不像可供专家审核的完整案例。
- 没有展开最终原因、实际处理、恢复验证和遗留风险。
- 没有形成“现场事实—诊断依据—执行步骤—恢复结果”的证据链。
- 前面上传的图片、视频和音频没有在最终记录中回看。
- “完成 5 项”只显示数量，无法判断实际执行内容。
- 上传前就显示“待审核”，状态语义错误。
- 工程师需要进入另一套案例回流页面重复确认，主线被割裂。

## 3. 角色边界

### 3.1 工程师

工程师只负责：

- 查看系统自动归集的现场事实和诊断结论。
- 核对或修正最终故障原因、实际处理和恢复结果。
- 检查现场材料、完成步骤和恢复参数。
- 点击“上传至专家知识库”。
- 在原页面查看上传结果和待审核状态。

工程师页面不得提供“进入专家页面”“切换为专家”或“替专家审核”的入口。

### 3.2 专家

专家必须退出工程师账号后，使用独立 `expert` 账号登录。专家继续负责：

- 查看待审核案例及其完整证据包。
- 修正案例结论。
- 修改一条或多条结构化知识。
- 确认知识图谱关系。
- 发布正式知识版本。

### 3.3 术语说明

按钮文案保留“上传至专家知识库”，但其业务含义是：

> 将工程师检修记录上传到专家待审核案例库，而不是直接写入正式检修知识库。

只有专家审核并发布后，内容才成为正式知识并进入知识图谱。

## 4. 信息架构调整

### 4.1 删除的工程师页面

以下页面不再作为工程师主流程页面：

- 独立“案例回流工作台”。
- 独立“工程师案例确认页”。
- 独立“上传成功页”。

对应能力全部在“检修完成记录”中原位完成。

### 4.2 导航调整

- 工程师主导航删除“案例回流”入口。
- “检修记录”保留，负责查看本次记录和历史检修记录。
- 当前流程刚完成时，直接展示本次“检修完成记录”详情。
- 从导航进入“检修记录”时，无论是否存在当前记录，均先进入分类台账列表。
- 当前记录作为台账中的首条记录展示；工程师点击该条后，再进入完整“检修完成记录”详情。
- 历史记录继续保留独立详情页，并支持按检修类型分类和按关键词筛选。
- 专家端的“专家工作台、全部案例、检修知识库、知识图谱”保持不变。

### 4.3 页面职责

“检修完成记录”同时承担三种职责：

1. 检修结果归档预览。
2. 专家待审核案例的上传前确认。
3. 上传后的状态回执。

## 5. 页面状态模型

| 页面状态 | 后端案例状态 | 页面文案 | 可编辑 | 主按钮 |
| --- | --- | --- | --- | --- |
| 未生成 | 无当前记录 | 检修记录尚未生成 | 否 | 返回检修向导 |
| 待确认 | `awaiting_engineer_confirmation` | 尚未上传 | 是 | 上传至专家知识库 |
| 上传中 | 请求处理中 | 正在整理案例并上传 | 否 | 加载状态 |
| 待专家审核 | `pending_expert_review` | 已进入专家待审核库 | 否 | 已上传，不可重复提交 |
| 已发布 | `archived_with_knowledge` | 专家已审核并发布知识 | 否 | 查看知识更新 |
| 上传失败 | 状态未改变 | 上传失败，记录仍保留 | 是 | 重新上传 |

强制规则：

- 上传前不得显示“专家状态：待审核”，必须显示“尚未上传”。
- 只有接口成功返回后才能切换为“待专家审核”。
- 重复点击或刷新不得生成重复案例。
- 上传失败不得清空工程师已填写的数据。

## 6. 检修完成记录页面结构

页面采用“结果主线 + 证据辅助”的双栏结构，避免重新变成杂乱的后台工作台。

### 6.1 顶部记录头

展示：

- `recordId`：例如 `REC-ACP4000-001`。
- 记录标题：例如“站控柜 A01 工控机散热异常检修”。
- 一句话结论。
- 当前状态标签：尚未上传 / 上传中 / 待专家审核 / 已发布。
- 检修完成时间、工程师、场站和设备位置。

### 6.2 核心摘要卡

保留四张卡，但内容调整为：

1. 设备与位置。
2. 故障与告警。
3. 已完成步骤数量。
4. 案例回流状态。

第四张卡在上传前显示“尚未上传”，上传成功后才显示“待专家审核”。

### 6.3 现场事实与诊断依据

以只读方式展示：

- 发生时间。
- 发生地点。
- 发生事件及其他现象。
- 告警与处理前参数。
- Agent 诊断方向。
- 关键判断依据及来源。

这一块用于回答专家的第一个问题：“为什么当时会形成这个判断？”

### 6.4 实际检修结果

上传前允许工程师核对和修正：

- 最终故障原因 `finalCause`。
- 实际处理 `actualResolution`。
- 恢复结果 `recoveryResult`。
- 遗留风险 `residualRisk`。

提供“一键采用现场结果”，但不得自动提交。上传后全部锁定为只读。

### 6.5 已完成步骤

不再只显示“5 项”，而是展开简洁步骤清单：

```text
1. 断电、挂牌和防静电确认
2. 清理滤网与风道
3. 检查 FAN1/FAN2 接线
4. 更换老化风扇
5. 恢复上电并观察 15 分钟
```

每一步展示完成状态；安全强制项保留锁定标识。

### 6.6 恢复验证

使用处理前后对比表，至少包含：

| 参数 | 处理前 | 处理后 | 结果 |
| --- | ---: | ---: | --- |
| 风扇转速 | 420 rpm | 1280 rpm | 恢复 |
| 系统温度 | 58℃ | 42℃ | 下降 |
| CPU 温度 | 74℃ | 58℃ | 下降 |
| TEMP/FAN 告警 | 告警 | 已解除 | 通过 |
| 连续观察 | — | 15 分钟 | 稳定 |

恢复验证是页面的重点内容，不能折叠成一句话。

### 6.7 现场材料

展示本次会话中已经上传的：

- 故障图片缩略图。
- 视频封面和时长。
- 音频名称和时长。
- 材料总数。

展示方式继续使用紧凑宫格，点击可预览，不允许大图挤压结果区。

第一版限制：前端本地材料尚未上传服务器，因此提交 SQLite 时只保存材料元数据；二进制文件上传不属于本次范围。页面必须明确区分“本地会话材料”和“已持久化附件”。

### 6.8 来源与可追溯信息

在页面底部用轻量信息区展示：

- 来源诊断编号。
- 来源检修预方案版本。
- 工程师确认时间。
- 目标专家待审核案例编号。
- 目标知识条目，例如 `KB-008`。

该区域不作为视觉主体，但必须让专家能够追溯案例来源。

### 6.9 底部操作区

上传前：

- 次按钮：打印作业卡。
- 主按钮：上传至专家知识库。
- 辅助说明：上传后由专家独立审核、修正，再决定是否发布正式知识。

上传成功后不跳页，原位替换为：

```text
上传成功
案例 CASE-ACP4000-001 已进入专家待审核库
当前状态：等待专家审核
```

同时主按钮进入不可重复提交状态，可提供“查看已上传内容”，但不得跳转专家页面。

## 7. 数据结构

前端提交对象统一为 `feedbackPackage`：

```json
{
  "caseId": "CASE-ACP4000-001",
  "recordId": "REC-ACP4000-001",
  "engineerId": "lishifu",
  "incident": {
    "time": "2026-07-10 10:25",
    "site": "某输气场站",
    "location": "站控柜 A01",
    "equipment": "研华 ACP-4000 / IPC-610",
    "description": "站控柜内工控机温度告警，风扇声音异常"
  },
  "diagnosis": {
    "conclusion": "工控机散热异常",
    "evidence": ["TEMP/FAN 告警", "风扇 420 rpm", "CPU 74℃"]
  },
  "maintenanceResult": {
    "finalCause": "滤网积尘叠加风扇老化",
    "actualResolution": "清理滤网并更换老化风扇，核对 FAN1/FAN2 接线",
    "recoveryResult": "TEMP/FAN 告警解除，设备恢复正常",
    "residualRisk": "无明显遗留风险"
  },
  "completedSteps": [],
  "recoveryMetrics": {
    "before": {},
    "after": {},
    "observationMinutes": 15
  },
  "materials": [],
  "targetKnowledgeIds": ["KB-008"]
}
```

要求：

- `feedbackPackage` 是页面渲染与上传的同一份数据，禁止页面展示一套、接口提交另一套。
- `finalCause`、`actualResolution`、`recoveryResult` 和恢复参数为上传必填项。
- `materials` 第一版只保存名称、类型、大小、时长和本地会话标识，不保存 Blob URL。

## 8. API 与 SQLite 处理

第一版优先复用现有接口：

```http
POST /api/admin/cases/CASE-ACP4000-001/submit
```

前端按钮文案和工程师语义改变，但后端可先保持兼容，避免扩大修改范围。请求体由现有 `engineerResult` 扩展为能够接收完整 `feedbackPackage`；过渡期继续兼容旧字段。

成功事务必须完成：

1. 保存或更新工程师检修结果。
2. 保存记录与案例的关联。
3. 保存材料元数据。
4. 将案例状态更新为 `pending_expert_review`。
5. 写入提交时间和提交人。
6. 保证同一 `caseId + recordId` 幂等。

上传时不得：

- 创建正式知识版本。
- 修改正式知识图谱。
- 自动标记专家通过。

正式知识和图谱仍只能由专家发布事务写入。

## 9. 与专家端的联动

上传成功后，专家账号的工作台必须出现：

- 案例编号和记录编号。
- 工程师与场站。
- 最终原因、实际处理和恢复结果。
- 已完成步骤。
- 处理前后参数。
- 材料元数据和可用预览。
- 来源诊断和目标知识条目。

专家端原有三步流程保持不变：

```text
审核并修正案例
  -> 修正知识与知识图谱
  -> 确认并发布
```

## 10. 错误与恢复

- 网络或后端失败：保留表单，显示“上传失败，可重新上传”。
- SQLite 写入失败：事务回滚，状态仍为 `awaiting_engineer_confirmation`。
- 重复点击：上传按钮在请求期间禁用，后端同时做幂等保护。
- 页面刷新：从 SQLite 恢复状态；已上传案例显示只读回执。
- 服务重启：状态不得丢失。
- 演示重置：页面状态和 SQLite 同时恢复到“尚未上传 / V1.0”。

## 13. 验收标准

### 13.1 工程师流程

- 工程师主导航不再出现独立“案例回流”页面入口。
- 检修完成后直接进入信息完整的检修记录。
- 页面能够说明发生了什么、为什么判断、做了什么以及是否恢复。
- 完成步骤必须展开，不能只显示数量。
- 恢复参数必须有处理前后对比。
- 现场材料以紧凑缩略图展示。
- 上传前状态为“尚未上传”。
- 点击上传后不跳页。
- 上传成功后显示案例编号和“待专家审核”。
- 页面不存在跳转专家端按钮。

### 13.2 数据与专家联动

- SQLite 中只生成一条对应案例。
- 刷新和服务重启后待审核状态仍在。
- 专家账号能看到工程师上传的完整数据。
- 工程师上传不会直接修改正式知识或图谱。
- 专家发布后才生成新知识版本和正式图谱关系。
- 演示重置同时恢复页面与数据库。

### 13.3 技术验证

- Node.js `20.19.4` 下 `npm run build` 通过。
- 后端单元测试覆盖提交幂等、重启恢复和重置。
- 浏览器验证 1366×768 与 1920×1080 主路径无横向溢出。
- 上传、刷新、专家读取和重置路径无控制台错误。
- 龙芯 Node.js `20.19.4` 开发态完成同模块验证后，才可标记最终完成。

## 14. 本次不做

- 真实附件二进制上传与对象存储。
- 真实大模型自动生成案例。
- 工程师与专家即时消息。
- 工程师直接修改正式知识。
- 工程师页面内切换专家身份。
- 自动专家审批。
- PDF 导出；仅保留现有打印能力。

## 15. 完成定义

只有同时满足以下条件，本次调整才算完成：

1. 独立工程师案例回流页面不再参与主流程。
2. 检修完成记录具备完整案例证据与恢复信息。
3. 工程师可在本页直接上传并原位看到结果。
4. SQLite 正确进入 `pending_expert_review`。
5. 专家账号读取到同一份案例并可继续三步审核。
6. 原有知识发布、图谱更新和工程师同步链路没有回归。
7. WSL 构建、后端测试和浏览器主路径通过。

## 16. 2026-07-13 实施结果

- 工程师主导航已移除独立“案例回流”。
- “检修记录”在存在当前记录或 SQLite 已提交记录时，直接打开完整检修完成记录；没有当前记录时保留历史记录列表。
- 检修完成记录已合并事实与诊断、工程师实际结果、完整步骤、恢复前后对比、现场材料和来源追溯。
- 上传前状态为“尚未上传”；上传后原页切换为“待专家审核”，字段锁定且按钮不可重复提交。
- 后端兼容原 `engineerResult`，同时接收并持久化完整 `feedbackPackage`；同一 `recordId` 重复提交按幂等成功返回。
