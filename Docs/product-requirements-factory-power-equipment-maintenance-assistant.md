# 产品需求文档：工控设备异常检测与故障分析系统

## 1. 产品概述

### 1.1 产品名称

工控设备异常检测与故障分析系统

### 1.2 产品定位

面向油气管道场站一线巡检与工控维护人员的“异常定位 + 故障分析 + 步骤式检修向导”系统。

系统不以传统关键词搜索为核心，也不以连续对话聊天作为主体体验。巡检人员可以先用自然语言描述现场现象，或上传工控机、PLC 控制柜、模块指示灯、告警界面、机柜环境等现场图片；系统识别出设备、现象、风险和缺失信息后，应快速切换为步骤式检修向导，用一组明确的步骤卡引导用户完成异常定位、故障分析、检查操作、安全确认、恢复验证和专家回流。

对话式交互定位为辅助能力：用于识别问题、补齐信息、解释某一步骤和回答追问；但主流程不应让用户一直在聊天框里等待答案，而应让用户按照可见、可勾选、可追溯的步骤推进。MVP 阶段采用预设流程：大模型总结、步骤选项、步骤内容、图片占位区和语音/专家等入口均可使用预置数据与缓存，重点展示产品交互理念和完整闭环。图片插入、上传、管理和真实图片展示先不做，只预留位置与字段。

系统引入“多 Agent 协同会诊 + 经验记忆回流”机制，将老师傅经验抽象为工控设备专家、PLC 系统专家和安全合规专家角色。首期通过接诊、检索、领域专家、安全合规等角色协同完成诊断，让系统输出更接近真实场站值守与维护班组的会诊建议。

系统边界明确为“只读不控”：系统只用于检修知识检索、故障辅助判断、作业指引和经验沉淀，不向任何工控设备、PLC、SCADA 或现场执行机构下发控制指令，不涉及工控网络安全攻防能力。

### 1.3 场景定位

场景限定为：

> 某大型油气管网企业输气场站站控核心设备检修

场景统一采用匿名化表达，使用“某大型油气管网企业”“某输气场站”“站控机柜”“PLC 控制柜”等通用表述。所有现场照片、设备照片和资料均需脱敏处理，不出现企业名称、真实站点名称、人员信息、敏感铭牌和内部网络信息。

油气管道场站通常具备少人值守、连续运行、远程监控和安全要求高等特点。站控系统中的工控机、PLC 控制柜、通信模块、电源模块和 I/O 模块承担现场数据采集、状态监测、联锁逻辑执行和远程监控支撑。一旦工控机高温、PLC 模块通信中断、电源模块异常或告警监控失效，可能造成监测数据上传中断、远程监控失明、部分监测点失联或现场处置依赖人工切换，进而影响场站运行安全。

## 2. 背景与痛点

### 2.1 业务背景

油气管道场站承担输气、计量、调压、监测和远程管理等任务，站控系统是场站安全运行的核心支撑。工控机和 PLC 控制柜虽然不是管道本体设备，但它们承担数据采集、状态监测、报警展示、控制逻辑执行和远程上传等关键能力，是场站运行的“眼睛”和“大脑”。

传统检修知识分散在设备手册、PLC 模块手册、巡检记录、维修日志、告警记录、历史工单、Excel 表格和老师傅经验中。一线人员遇到工控机温度告警、PLC 模块指示灯异常、通信中断或电源模块异常时，往往需要翻手册、查灯态对照表、问专家，才能形成初步判断。

### 2.2 主要痛点

1. 知识分散：工控机手册、PLC 模块手册、灯态对照表、巡检规程、历史案例和专家经验分散存放。
2. 检索低效：巡检人员需要知道准确设备型号、模块名称或指示灯含义，才能搜到相关内容。
3. 现场信息利用不足：工控柜照片、模块指示灯状态、告警截图、机柜环境照片、温度读数等信息无法直接参与检索。
4. 影响范围难判断：模块通信中断或电源异常后，一线人员难以快速判断会影响哪些监测点、上传链路或现场功能。
5. 作业不规范：断电、静电防护、禁止热插拔、通知调控中心、恢复运行确认等安全检查容易遗漏。
6. 新人上手慢：工控设备型号多、模块灯态多，依赖老师傅口头经验，经验传承困难。
7. 大模型回答不可直接信任：需要知识来源、专家修正和审核机制。
8. 经验难回流：现场修正和优秀案例不能及时沉淀为可复用知识。

## 3. 产品目标

### 3.1 用户目标

1. 一线巡检人员能够用最少输入获得工控设备故障处理方向。
2. 一线巡检人员能够通过照片快速理解指示灯、告警和机柜环境异常。
3. 一线巡检人员能够看到清晰的检查步骤、安全提醒和恢复运行确认条件。
4. 专家能够审核、修正并沉淀工控设备检修经验。
5. 新员工能够通过系统复用历史案例和专家知识。

### 3.2 业务目标

1. 缩短工控设备检修知识查找时间。
2. 提高工控机、PLC 控制柜和模块故障初判效率。
3. 降低不规范作业、误插拔、误复位和安全检查遗漏风险。
4. 帮助一线人员快速判断故障影响范围，支持调控中心和现场人员协同处置。
5. 提升检修经验沉淀和复用能力。
6. 形成持续更新的场站工控设备检修知识体系。

### 3.3 比赛目标

1. 展示多模态知识检索能力。
2. 展示标准化作业指引能力。
3. 展示知识沉淀与更新能力。
4. 展示知识图谱与专家修正闭环。
5. 展示比传统搜索更易上手的检修辅助体验。
6. 展示系统可通过配置和知识库扩展新设备、新故障和新检修场景，而不是依赖代码改造。
7. 展示多 Agent 协同会诊能力，将老师傅经验从静态知识条目升级为可参与诊断、可修正、可持续沉淀的专家视角。
8. 展示工控设备指示灯状态识别、灯态对照检索和影响范围评估等场站特色能力。

### 3.4 首期创新点

首期聚焦“可视化 + 步骤式向导”的演示闭环，不扩展到过重的多模态文档理解、复杂三维建模或复杂 Agent 框架。

1. 步骤式检修向导
   - 用户可以先描述现场现象，系统识别问题后自动生成可执行步骤。
   - 每一步展示目标、操作、依据、安全提醒、完成条件和下一步入口。
   - 避免用户在长对话中寻找答案，让检修过程以任务卡片推进。

2. 图片占位式步骤定位
   - 首期不做真实三维建模、可旋转模型或设备热点算法。
   - MVP 阶段只在每个向导步骤预留固定尺寸图片区块。
   - 占位区显示“图片待补充”或“后续添加步骤示意图”等文案。
   - 后续可把预先标注好的厂区、站控柜、工控机或风道位置图片接入这些占位区。

3. 对话辅助识别
   - 对话用于识别问题、补齐设备型号、检修等级、灯态等缺失信息；图片信息作为后续扩展项。
   - 用户进入步骤式向导后，可在任一步骤中发起追问，系统只解释当前步骤或补充依据。

4. 预设流程模拟动态生成
   - 首期演示流程预先设计，覆盖一个设备和一个核心故障。
   - 对话结束后，大模型总结结果并弹出若干步骤选项；这些选项和后续步骤可使用预置数据。
   - 用户逐步选择、确认和完成步骤，形成“动态生成并推进”的演示效果。

5. 指示灯状态多模态识别
   - 将 PLC 模块、通信模块、电源模块等设备的 LED 颜色、亮灭和闪烁状态结构化。
   - 将指示灯状态作为精确检索键，匹配手册灯态对照表和历史案例。
   - 首期可作为辅助能力或备用演示，不作为第一版核心流程。

6. 场站级影响范围评估
   - 在知识图谱中建立“模块故障 -> 影响监测点/上传链路/人工监视要求”等关系。
   - 让系统不仅解释故障原因，也能说明对场站运行的影响。

7. 安全合规 Guardrails Agent
   - 对检修向导步骤和检修记录进行安全合规检查。
   - 检查断电、静电防护、禁止热插拔、双人作业、通知调控中心、恢复运行确认、验收、异常升级等关键项。

8. 专家 Agent 经验记忆回流
   - 专家修正后可写入对应专家 Agent 的经验记录。
   - 后续同类诊断可引用已审核经验。

9. 证据链式 RAG
   - 检修建议必须展示知识来源。
   - 回答、图谱、检修向导和导出的作业卡都能追溯到知识条目、案例或专家修正记录。

10. 模拟能力与预留接口
   - 连线专家、语音播报和视频输入首期只做入口、状态标识或模拟界面。
   - 文档中明确这些能力预留第三方 IM/RTC、TTS、视频识别或设备采集接口。

11. 诊断轨迹 / 可审计面板
   - 记录每次诊断的输入、识别、检索、会诊、安全校验、向导执行、检修记录和修正过程。
   - 用于演示、测试报告和后续复盘。

首期不做完整视觉文档 RAG，不做 PDF 页面截图、高亮区域、扫描件 OCR、图表识别和多向量页面检索；不做工控网络安全攻防分析，不接入真实控制链路，不向任何现场设备下发控制指令。

## 4. 目标用户

### 4.1 一线巡检与检修人员

典型用户：

- 新入职场站巡检员
- 工控设备维护人员
- 自动化仪表与站控维护班组成员

核心诉求：

- 快速判断故障方向
- 少翻手册、少查灯态表、少问人
- 获得明确的下一步检查建议
- 不遗漏安全注意事项
- 明确故障可能影响哪些监测点或上传链路
- 能把现场经验反馈给系统

### 4.2 专家审核员

典型用户：

- 班组长
- 自动化工程师
- 工控系统工程师
- 场站技术负责人

核心诉求：

- 审核 AI 输出是否可靠
- 修正错误或不完整答案
- 沉淀高质量案例
- 维护企业知识库质量

### 4.3 专家 Agent

专家 Agent 是对老师傅和专业岗位经验的数字化抽象，不等同于真实员工复制，而是代表某一类专业经验参与诊断。

首期设置：

1. 工控机设备专家 Agent
   - 关注工控机高温、风扇、滤网、供电、硬盘和监控告警等故障。
   - 擅长结合面板状态、机柜环境和设备手册判断故障原因和检查顺序。

2. PLC 系统专家 Agent
   - 关注 PLC 控制柜、通信模块、电源模块、I/O 模块、端子和指示灯状态。
   - 擅长结合灯态对照表、告警截图和现场现象给出排查路径。

3. 安全合规专家 Agent
   - 关注断电、静电防护、禁止热插拔、双人作业、通知调控中心、恢复运行确认和验收要求。
   - 专门检查检修向导和检修记录是否遗漏关键安全步骤。

核心价值：

1. 让老师傅经验以角色形式参与诊断。
2. 让系统输出不只是单一 AI 答案，而是多视角会诊结论。
3. 让专家修正不仅进入知识库，也能进入对应专家 Agent 的经验记录。

## 5. 产品核心闭环

系统主流程：

```text
现场输入
  -> 识别设备、模块、指示灯状态与故障现象
  -> 确认故障类型、检修等级和风险等级
  -> 检索相关知识和历史案例
  -> 多 Agent 协同会诊
  -> 大模型总结问题并给出步骤选项
  -> 用户选择步骤并进入步骤式检修向导
  -> 每一步展示图片占位区、操作说明、安全提醒和完成条件
  -> 按步骤完成安全确认、风道/滤网/风扇检查、处理建议和恢复验证
  -> 生成检修完成记录，可附带导出作业卡
  -> 一线人员反馈
  -> 专家审核修正
  -> 知识库、知识图谱和专家经验记忆更新
```

产品体验目标：

```text
看到什么 -> 上传什么
灯态/告警/环境 -> 自动结构化
系统识别 -> 自动关联知识和影响范围
生成步骤 -> 按卡执行
专家修正 -> 下次更准
```

智能接诊与轻量会诊流程：

```text
用户描述现场现象
  -> 系统判断信息是否足够
  -> 通过对话或表单补齐场站位置、设备类型、型号、模块类型、检修等级或图片
  -> 形成结构化检修上下文
  -> 检索手册、灯态对照、规程、案例和图谱证据
  -> 领域专家视角分析
  -> 安全合规专家视角检查
  -> 评估监测点、上传链路或人工监视要求等影响范围
  -> 系统汇总会诊意见并识别当前问题
  -> 生成几个步骤选项供用户选择
  -> 用户按预设步骤执行、确认、补充或提交专家审核
```

## 6. 首期设备与故障范围

### 6.1 设备范围

首期核心演示设备收敛为 1 类：

1. 研华 ACP-4000 / IPC-610 工控机整机

PLC 控制柜、通信交换机、传感器、现场仪表、电动执行机构、SCADA 监控平台等对象首期作为关联知识条目、影响范围节点、备用演示素材或知识图谱节点保留，不作为主要页面展示和主演示对象。后续可通过配置和知识库扩展为完整场站设备类型。

### 6.2 故障范围

首期第一版只实现 1 个主演示故障，确保流程完整、可控、可录制。其他故障作为后续扩展或备用演示数据保留。

| 设备 | 故障 |
|---|---|
| 研华 ACP-4000 / IPC-610 工控机整机 | 高温告警 / 风道堵塞 / 散热异常（第一版主演示） |
| 研华 ACP-4000 / IPC-610 工控机整机 | 风扇故障/滤网堵塞/风道不畅（与主演示合并处理） |
| 研华 ACP-4000 / IPC-610 工控机整机 | 供电异常/冗余电源单路异常 |
| 研华 ACP-4000 / IPC-610 工控机整机 | 硬盘故障/存储异常 |
| 研华 ACP-4000 / IPC-610 工控机整机 | 指示灯/蜂鸣器/监控告警异常 |
| 研华 ACP-4000 / IPC-610 工控机整机 | 系统运行异常/死机/数据上传中断 |
| PLC 控制柜（备用扩展） | 模块通信中断/电源模块异常/I/O 指示灯报警 |

PLC 控制柜故障保留在配置和数据结构中，首期不进入第一版主演示；如果时间紧，PLC 只作为站控系统关联设备和影响范围节点出现。

### 6.3 首期演示数据规模

首期数据以支撑演示闭环为目标，不追求大规模覆盖。

建议数据规模：

1. 知识条目：45-60 条。
2. 工控机指示灯、蜂鸣器和告警状态对照条目：8-12 条。
3. 历史案例：6-10 条。
4. 工控安全规则：8-12 条。
5. 图谱实体：45-60 个。
6. 图谱关系：70-90 条。
7. 图片占位：5 个步骤占位 key；真实演示图片后续补充。
8. 作业卡模板：3-4 个。
9. AI 调用缓存：覆盖主演示和备用演示。
10. 预埋专家修正演示对：1 组主线、1 组备用。

主演示固定为“工控机高温告警 / 风道堵塞 / 散热异常”。备用演示可保留“工控机供电异常”或“硬盘/存储异常”，但不作为第一版必须完成内容。

## 7. 功能需求

### 7.1 工作台

工作台用于让用户快速进入检修任务。

功能点：

1. 显示智能诊断入口。
2. 显示常见故障快捷入口。
3. 显示最近知识更新。
4. 显示专家待审核数量。
5. 显示最近诊断记录。

### 7.2 异常检测与步骤式检修向导

异常检测与步骤式检修向导是系统核心页面。它不是让用户一直与系统聊天，而是先接收用户的一句话描述、图片或预设异常事件，再由系统完成问题总结，随后切换为步骤式向导。

输入方式：

1. 输入现场现象，例如“站控柜内工控机温度告警，风扇声音异常”或“工控机面板硬盘指示灯异常闪烁，数据上传间歇中断”。
2. 系统根据描述总结设备、故障现象、风险点和缺失信息。
3. 用户可选择或确认设备类型。
4. 用户可输入设备型号。
5. 用户可选择检修等级。
6. 用户可上传现场照片、模块指示灯照片、告警截图或机柜环境照片作为辅助信息。

输出内容：

1. 故障描述结构化结果。
2. 图片辅助识别结果。
3. 设备、模块、指示灯状态和环境风险识别结果。
4. 可能故障类型。
5. 影响范围提示，例如监测点失联、上传链路中断或需转人工监视。
6. 推荐检查方向。
7. 相关知识条目。
8. 相似历史案例。
9. 步骤选项，例如“定位异常设备”“查看故障分析”“检查风道和滤网”“执行安全确认”“完成检修记录”。
10. 进入步骤式检修向导的入口。

交互要求：

1. 首期以文字描述或预设异常事件作为第一入口，场站位置、设备类型、设备型号、模块类型和检修等级通过少量追问、表单或预设数据补齐。
2. 图片作为辅助取证，不作为主流程强依赖。
3. 指示灯识别结果不确定时，允许人工通过灯态勾选面板确认或修改。
4. 输出不能是一整段文本，应拆分为诊断摘要、步骤选项、步骤卡、图片占位区和来源。
5. 首期步骤流允许预设，不要求真实泛化生成；演示中应表现为“大模型总结后生成步骤选项”。

#### 7.2.1 诊断与向导状态流转

诊断页面采用“步骤式向导主区域 + 辅助对话小窗 + 流程状态栏”的交互方式。主区域展示当前步骤、图片占位区、操作要求和完成按钮；对话小窗只用于解释当前步骤、补充信息或回答追问；流程状态栏展示阶段进度。

诊断会话状态：

```text
idle
  -> describing
  -> info_collecting
  -> analyzing
  -> step_options
  -> guiding
  -> completed
  -> record_created
  -> submitted_for_review
  -> reviewed
```

状态说明：

1. `idle`：用户尚未开始诊断，停留在工作台或诊断台空状态。
2. `describing`：用户已输入现场现象或触发预设异常事件，系统正在识别问题类型。
3. `info_collecting`：系统正在引导用户补充场站位置、设备型号、检修等级、图片等信息。
4. `analyzing`：系统进行知识检索、Agent 会诊和安全检查，并汇总故障判断。
5. `step_options`：系统展示大模型总结结果和若干步骤选项，用户选择下一步。
6. `guiding`：用户进入步骤式检修向导，按步骤查看图片占位区、执行操作、勾选完成。
7. `completed`：步骤向导执行完成，系统汇总检修过程和结果。
8. `record_created`：系统生成检修完成记录，可导出作业卡或提交专家审核。
9. `submitted_for_review`：维修请求已提交专家审核，等待专家处理。
10. `reviewed`：专家已完成审核，修正内容已回流或记录。

右侧流程栏展示 6 个节点：

1. 现场描述。
2. 信息补充。
3. 知识检索与会诊。
4. 步骤选项生成。
5. 步骤式检修向导。
6. 检修记录与专家回流。

节点状态包括：

1. 未开始。
2. 进行中。
3. 已完成。
4. 需补充。

按钮出现规则：

1. `info_collecting` 状态下显示“继续补充信息”；图片上传入口只保留为后续能力标识，MVP 不实现上传。
2. `analyzing` 状态下显示“正在分析”和检索/会诊过程动画。
3. `step_options` 状态下显示“定位异常设备”“查看故障分析”“进入检修步骤”等步骤入口。
4. `guiding` 状态下显示“上一步”“下一步”“标记完成”“查看依据”“辅助提问”。
5. `completed` 状态下显示“生成检修记录”“导出作业卡”“提交专家审核”。
6. `submitted_for_review` 状态下显示“查看审核状态”。
7. `reviewed` 状态下显示“查看专家修正”“再次发起相似诊断”。

异常状态处理：

1. 图片能力未接入或识别失败时，诊断会话不终止，系统提示人工确认设备和异常现象。
2. RAG 无可靠证据时，诊断结果中标记“不确定，需人工确认”。
3. 云 API 调用失败时，优先使用演示缓存或本地兜底数据，并展示可理解提示。

### 7.3 多模态知识检索

支持输入：

1. 文本描述。
2. 设备型号。
3. 设备类型。
4. 检修等级。
5. 模块类型和指示灯状态。
6. 故障图片、告警截图或机柜环境照片作为后续辅助输入，MVP 只预留入口。

支持检索：

1. 设备手册。
2. PLC 模块手册和指示灯状态对照表。
3. 检修规程。
4. 历史案例。
5. 安全规范。
6. 专家经验。

输出要求：

1. 返回相关知识摘要。
2. 返回引用来源。
3. 返回相似案例。
4. 返回指示灯状态对应含义。
5. 返回影响范围提示。
6. 返回建议追问方向。

### 7.4 辅助问答

辅助问答不是主流程，只作为步骤式向导中的解释入口。用户可以围绕当前故障、当前步骤、当前图片或安全要求继续提问，系统回答应尽量短，并回到当前步骤继续执行。

示例问题：

1. “工控机温度告警应该先检查什么？”
2. “PLC 通信模块 ERR 红灯常亮、COMM 灯熄灭可能是什么原因？”
3. “更换通信模块前需要哪些安全确认？”
4. “这个模块通信中断会影响哪些监测点？”

回答结构：

1. 判断结论。
2. 可能原因。
3. 检查方法。
4. 处理建议。
5. 安全注意事项。
6. 影响范围。
7. 引用来源。

#### 7.4.1 诊断结论结果结构

用户完成现象描述或触发预设异常后，系统生成诊断摘要卡片，并提供步骤选项。诊断摘要不是最终产出，最终产出是步骤式检修向导和检修完成记录。

诊断结论卡包含：

1. 结论摘要。
2. 当前诊断上下文。
3. 故障判断。
4. 可靠性提示。
5. 可能原因 Top N。
6. 建议检查步骤。
7. 影响范围。
8. 安全提醒。
9. RAG 证据卡片。
10. Agent 会诊意见。
11. 下一步操作按钮。

字段说明：

| 字段 | 说明 |
|---|---|
| 结论摘要 | 用 1-2 句话说明当前最可能的问题 |
| 当前诊断上下文 | 场站、机柜、设备型号、故障现象、灯态和图片占位信息 |
| 故障判断 | 例如“工控机高温/散热异常” |
| 可靠性提示 | 高/中/低，或“需人工确认” |
| 可能原因 Top N | 按优先级列出滤网堵塞、风扇故障、风道不畅等原因 |
| 建议检查步骤 | 给出下一步应检查的项目 |
| 影响范围 | 展示可能影响监控告警、数据上传、历史记录或人工监视要求 |
| 安全提醒 | 展示断电、静电防护、禁止热插拔等关键项 |
| RAG 证据卡片 | 每条关键结论绑定来源文档、章节、引用片段或案例 |
| Agent 会诊意见 | 展示工控设备专家、安全合规专家等意见摘要 |
| 下一步操作 | 定位异常设备、查看故障分析、进入检修步骤、提交专家审核 |

诊断结论生成规则：

1. 故障判断、可能原因、处理建议和安全提醒必须尽量绑定证据来源。
2. 没有证据来源的结论不得作为确定结论展示，应标记为“不确定，需人工确认”。
3. 专家已审核的经验命中时，应展示“专家修正 · 已审核”标签，并显示审核编号或来源记录。
4. 诊断结论不直接替代人工检修判断，应提示“现场执行前需按企业规程确认”。

诊断结论示例：

```text
诊断结论：当前优先判断为工控机高温/散热异常。

可能原因 Top 3：
1. 前面板滤网堵塞。
2. 风扇转速异常或风扇轴承老化。
3. 机柜进出风通道不畅。

影响范围：
- 可能影响监控告警可靠性。
- 长时间高温可能影响数据上传稳定性和历史记录完整性。

安全提醒：
- 检查前做好静电防护。
- 禁止带电拆卸非热插拔部件。
- 必要时通知值班负责人。
```

#### 7.4.2 图片占位式步骤展示

MVP 阶段不实现图片插入、图片上传、图片管理和真实图片展示，只采用“步骤图片占位区”的方式保留后续扩展位置。

实现边界：

1. 不做真实三维建模。
2. 不做可旋转模型。
3. 不做自动热点识别或自动坐标标注。
4. 不做复杂设备部件联动动画。
5. 每个步骤对应一个固定尺寸图片占位区。
6. 图片由团队后续手动标注后，可通过预留素材字段接入，不影响当前步骤流程。

页面表现：

1. 异常定位步骤显示“场站 / 站控柜 / 工控机位置图片待补充”占位区。
2. 风道检查步骤显示“风道、进风口或出风口图片待补充”占位区。
3. 滤网检查步骤显示“滤网位置和积尘风险图片待补充”占位区。
4. 风扇检查步骤显示“风扇位置和异常检查点图片待补充”占位区。
5. 恢复验证步骤显示“温度告警、数据上传或运行状态确认图片待补充”占位区。

图片占位与步骤关系示例：

| 步骤 | 图片占位 | 说明 |
|---|---|---|
| 定位异常设备 | `step-01-location` | 后续用于补充发生高温告警的站控柜和工控机示意图 |
| 确认安全条件 | `step-02-safety` | 后续用于补充电源、值班通知和防护要求示意图 |
| 检查风道 | `step-03-airflow` | 后续用于补充进风口、出风口和可能堵塞位置示意图 |
| 检查滤网和风扇 | `step-04-filter-fan` | 后续用于补充滤网、风扇和清理/检查位置示意图 |
| 恢复验证 | `step-05-verify` | 后续用于补充温度、告警解除和数据上传稳定确认示意图 |

MVP 阶段图片占位的价值在于保证步骤向导布局稳定，后续补图时不需要重做流程；第一版重点仍是“下一步做什么”和“完成条件是什么”，不是展示复杂三维技术。

### 7.5 多 Agent 协同会诊

系统应支持在一次诊断中展示多个 Agent 协同过程，从接诊、知识检索、专业判断和安全合规多个方向生成会诊意见。

首期不引入重型 Agent SDK。系统通过后端编排器组织多个 Agent 角色的输入、输出和上下文，其中部分流程 Agent 可由规则和代码实现，领域专家与安全合规 Agent 通过云端大模型 API 实现。

会诊输入：

1. 现场图片识别结果作为后续扩展项，MVP 显示占位或模拟结果。
2. 设备类型和型号。
3. 模块类型和指示灯状态。
4. 故障描述。
5. 检修等级。
6. 检索到的知识条目。
7. 知识图谱中关联的场站、系统、设备、模块、故障、原因、措施和影响范围。
8. 历史案例。

会诊输出：

1. 每个专家 Agent 的独立判断。
2. 每个判断引用的知识来源。
3. 专家 Agent 之间的差异点。
4. 系统汇总后的推荐结论。
5. 需要人工确认的问题。

首期 Agent 角色：

| Agent | 类型 | 关注范围 | 输出内容 |
|---|---|---|---|
| 接诊 Agent | 流程 Agent | 用户描述、场站位置、设备、型号、模块、检修等级、图片 | 标准化现场输入、补充追问 |
| 知识检索 Agent | 工具 Agent | 工控机手册、PLC 模块手册、灯态对照、规程、案例、图谱 | 证据列表、来源、相似案例 |
| 工控设备专家 Agent | 领域专家 Agent | 工控机、PLC 控制柜及关联检修知识 | 故障判断、原因分析、检查建议 |
| 安全合规 Guardrails Agent | 规则/专家 Agent | 工控检修安全、作业规范、恢复确认 | 安全提醒、合规检查、风险提示 |

页面展示要求：

1. 会诊结果以多专家卡片展示。
2. 每个专家 Agent 的结论应有来源依据。
3. 汇总结论应清楚标明采用了哪些专家意见。
4. 用户可以将某个专家意见加入检修向导或检修记录。
5. 专家审核员可以修正专家意见，并将修正内容回流到知识库或专家经验记录。

### 7.6 PDF 文本证据卡片

系统支持文本型 PDF 文档导入，并在检修问答和作业卡中展示可追溯证据。

该功能的定位是“可信来源展示”，不是完整视觉文档理解。系统只要求能够从文本型 PDF 中抽取文字、记录页码，并在 RAG 命中后展示对应证据。

证据卡片字段：

1. 来源文档名。
2. 页码。
3. 章节标题。
4. 引用片段。
5. 相关设备。
6. 相关故障。

示例：

```text
来源文档：工控机维护手册.pdf
页码：第 18 页
章节：散热与滤网维护
引用片段：出现温度告警时，应检查风扇运行状态、进风口滤网、机柜通风条件和环境温度……
相关设备：工控机
相关故障：高温/散热异常
```

首期不支持：

1. 扫描版 PDF OCR。
2. PDF 页面截图。
3. 页面区域高亮。
4. 图表识别。
5. 完整视觉文档 RAG。

用户提示：

1. 如果 PDF 为扫描件或图片型 PDF，系统可提示“暂不支持自动解析，请上传文本型 PDF 或使用结构化录入”。
2. 如果章节无法自动识别，系统可回退显示“第 N 页相关内容”。
3. 证据卡片只承诺展示文档名、页码和引用片段，不承诺定位页面高亮区域。

### 7.7 知识图谱

知识图谱用于展示检修知识关系，帮助用户理解故障链路。

核心关系：

```text
设备 -> 部件
部件 -> 故障
故障 -> 原因
故障 -> 检测方法
故障 -> 处理措施
处理措施 -> 安全要求
故障 -> 历史案例
故障 -> 指示灯状态
模块 -> 影响 -> 监测点/上传链路/人工监视要求
```

页面能力：

1. 根据当前故障展示相关图谱。
2. 高亮当前诊断链路。
3. 点击节点查看说明、案例和来源。
4. 从图谱节点跳转到知识条目、检修步骤或检修记录。
5. 展示模块故障对监测点、上传链路或人工监视要求的影响范围。
6. 展示专家修正后新增或更新的关系。

### 7.8 检修完成记录与作业卡导出

首期核心演示不围绕“生成一张作业卡 PDF”，而是围绕“步骤式检修向导”。作业卡定位为检修向导完成后的记录沉淀与导出结果，用于留痕、复盘和专家审核。

检修完成记录内容：

1. 设备信息。
2. 场站位置和机柜位置。
3. 故障现象。
4. 检修等级。
5. 风险等级。
6. 影响范围。
7. 已执行的向导步骤。
8. 每一步的完成状态。
9. 每一步对应的图片占位、证据或后续素材字段。
10. 作业前安全检查结果。
11. 合规提醒。
12. 恢复运行确认条件。
13. 验收标准。
14. 异常升级条件。
15. 引用来源。
16. 参与会诊的专家 Agent。
17. 采纳的专家意见。
18. 安全合规校验结果。

交互要求：

1. 步骤勾选发生在检修向导中，完成后自动汇总为检修记录。
2. 安全项应在对应向导步骤中明显展示。
3. 高风险步骤应突出提醒。
4. 作业完成后可填写反馈。
5. 若安全合规校验未通过，系统应阻止进入“完成检修”状态，或标记“需专家确认”。
6. 检修记录支持导出为作业卡 PDF。

导出交互：

1. 页面提供“导出作业卡”按钮。
   - 点击后打开检修记录打印版页面。
   - 页面自动或手动触发浏览器打印。
   - 用户通过浏览器“另存为 PDF”完成导出。

2. 打印版页面应隐藏导航、按钮、侧栏等非记录内容，只保留检修记录和作业卡正文。
3. 首期不提供 Word 导出，避免增加部署依赖和排版工作量。

#### 7.8.1 检修记录状态流转

检修记录从向导执行到归档采用明确状态流转，方便前端控制按钮、专家审核和演示。

检修记录状态：

```text
not_started
  -> guiding
  -> guardrails_checking
  -> completed
  -> record_created
  -> submitted_for_review
  -> diagnosis_reviewed
  -> record_reviewed
  -> approved
  -> rejected
  -> archived
```

状态说明：

1. `not_started`：诊断摘要已生成，但用户尚未进入检修向导。
2. `guiding`：用户正在按步骤执行检修向导。
3. `guardrails_checking`：安全合规 Guardrails 正在检查已执行步骤。
4. `completed`：向导步骤已完成，等待生成检修记录。
5. `record_created`：检修记录已生成，可查看、导出或提交专家审核。
6. `submitted_for_review`：检修记录随维修请求提交专家审核。
7. `diagnosis_reviewed`：专家已完成 AI 诊断详情审核，等待记录详情审核。
8. `record_reviewed`：专家已完成检修记录详情审核。
9. `approved`：整条维修请求已审核通过，记录标记为专家审核通过。
10. `rejected`：专家驳回维修请求或检修记录。
11. `archived`：检修执行反馈已保存，进入归档状态。

一线人员权限：

1. 可以进入步骤式检修向导。
2. 可以逐步查看图片占位区、操作说明和安全提醒。
3. 可以勾选完成步骤。
4. 可以生成检修记录。
5. 可以导出 PDF。
6. 可以提交专家审核。
7. 可以填写执行反馈和现场备注。
8. 首期不允许一线人员修改核心检修步骤、安全项和验收标准。

专家审核员权限：

1. 可以查看检修记录和安全合规检查结果。
2. 可以用“修正文本 + 回流选项”方式补充或纠正作业要求。
3. 可以将修正内容更新到当前检修记录或后续作业卡导出内容。
4. 可以通过、驳回或退回补充。

检修记录按钮规则：

1. `not_started` 状态下显示“进入检修向导”。
2. `guiding` 状态下显示“上一步”“下一步”“标记完成”。
3. `completed` 或 `guardrails_checking` 状态下显示“正在检查安全项”。
4. `record_created` 状态下显示“查看检修记录”“导出作业卡”“提交专家审核”。
5. `submitted_for_review` 状态下显示“查看审核状态”。
6. `approved` 状态下显示“导出专家审核版 PDF”“填写执行反馈”。
7. `rejected` 状态下显示“查看驳回原因”“补充信息后重新提交”。

记录版本规则：

1. 每次生成检修记录形成一个版本号，例如 `MR-001-v1`。
2. 专家修改后通过形成新版本，例如 `MR-001-v2`。
3. 页面应能显示当前版本是否为“AI 生成草稿”或“专家审核通过版”。
4. 导出 PDF 时优先导出最新已审核版本；若尚未审核，应显示“待专家确认”水印。

### 7.9 安全合规 Guardrails

步骤式检修向导执行过程中和检修记录生成前，必须经过安全合规 Guardrails 检查。

检查内容：

1. 是否包含作业前通知调控中心或值班负责人。
2. 是否包含断电或安全隔离要求。
3. 是否包含静电防护要求。
4. 是否包含禁止热插拔或热插拔前提说明。
5. 是否包含双人作业或旁站确认要求。
6. 是否包含拍照留档和告警记录要求。
7. 是否包含恢复运行确认，例如数据上传恢复、指示灯恢复、监测点稳定。
8. 是否包含验收标准。
9. 是否包含异常升级条件。

检查结果：

1. 通过。
2. 未通过。
3. 缺失项清单。
4. 修正建议。

### 7.10 专家审核与修正

专家可以审核和修正：

1. 识别结果。
2. 问答内容。
3. 检修向导步骤。
4. 安全注意事项。
5. 故障原因。
6. 处理措施。
7. 历史案例。
8. 专家 Agent 的经验记录。
9. 某个专家 Agent 的会诊结论。

审核结果：

1. 通过。
2. 驳回。
3. 修改后通过。

审核通过后，内容进入知识沉淀流程。

#### 7.10.1 专家审核入口

专家审核员登录后，可从以下入口进入审核页面：

1. 工作台“待审核”卡片。
2. 步骤式检修向导流程栏中的“进入专家审核”按钮。
3. 检修记录详情页中的“提交专家审核 / 查看审核状态”入口。
4. 专家审核与知识回流页面中的待审核列表。

待审核记录以“一次维修请求”为单位生成。一条待审核记录包含该维修请求下的 AI 诊断详情、检修向导步骤和检修记录，不把诊断结论、记录、图谱关系拆成多条独立待审核任务。

待审核请求来源包括：

1. 一线人员完成诊断并点击“提交专家审核”。
2. 一线人员完成检修向导并点击“提交专家审核”。
3. AI 生成结果被标记为“需专家确认”的高风险维修请求。
4. 安全合规 Guardrails 未完全通过但一线人员请求专家判断的维修请求。
5. 一线人员上传检修案例或经验总结后形成的待审核请求。

#### 7.10.2 专家审核页面布局

专家审核页面采用“待审核维修请求列表 + 请求详情 + 分步修正区”的结构。

页面左侧为待审核列表，展示：

1. 维修请求编号。
2. 提交时间。
3. 提交人。
4. 设备型号。
5. 故障类型。
6. 风险等级。
7. 当前审核阶段：待审核诊断、待审核检修记录、已完成、已驳回、退回补充。
8. 当前状态：待审核、修改中、已通过、已驳回。

页面主区域展示本次维修请求详情：

1. 用户原始输入。
2. 图片入口和识别结果占位。
3. 结构化诊断上下文。
4. RAG 检索证据卡片。
5. Agent 会诊意见。
6. 知识图谱路径和影响范围。
7. 检修向导步骤和检修记录内容。
8. 安全合规检查结果。

页面右侧或下方为分步专家修正区，包含两个审核步骤：

1. 第一步：审核 AI 诊断详情。
2. 第二步：审核检修记录详情。

每一步都展示：

1. AI 原始内容。
2. 专家修正文本。
3. 修改原因。
4. 回流选项。
5. 审核操作按钮。

#### 7.10.3 专家修正方式

首期不做复杂的结构化字段编辑器。专家修正采用“AI 原文 + 专家修正文本 + 回流选项”的方式。

AI 诊断详情审核时，页面展示：

1. AI 故障判断。
2. 可能原因。
3. 检查建议。
4. 处理建议。
5. 影响范围。
6. 风险等级。
7. 引用证据。

专家在修正文本框中补充或纠正内容，例如：

```text
沙尘环境下，站控柜工控机滤网维护周期应从季度检查缩短为月度检查；若温度告警伴随风扇转速低，应优先检查滤网堵塞和风扇轴承老化。
```

检修记录详情审核时，页面展示：

1. 作业前安全检查。
2. 操作步骤。
3. 工具和备件要求。
4. 恢复运行确认条件。
5. 验收标准。
6. 异常升级条件。

专家在修正文本框中补充或纠正作业要求，例如：

```text
恢复运行后需观察数据上传稳定不少于 10 分钟，并由值班负责人确认温度告警解除后再关闭工单。
```

系统不要求专家逐字段修改。修正文本作为审核意见、知识库条目、专家经验或检修记录补充内容进入回流流程。

#### 7.10.4 审核操作流程

专家审核的标准流程如下：

```text
进入待审核列表
  -> 打开某条维修请求
  -> 查看用户输入、AI 诊断、证据、图谱、检修向导和检修记录
  -> 第一步审核 AI 诊断详情
  -> 在诊断修正区填写修正文本和修改原因
  -> 选择诊断回流目标：知识库 / 图谱 / 专家 Agent 经验 / 仅记录意见
  -> 点击“诊断审核通过”或“诊断退回补充”
  -> 第二步审核检修记录详情
  -> 在检修记录修正区填写修正文本和修改原因
  -> 选择记录回流目标：更新当前检修记录 / 知识库 / 专家 Agent 经验 / 仅记录意见
  -> 点击“检修记录审核通过”或“检修记录退回补充”
  -> 整条维修请求进入已审核状态
  -> 后续同类诊断优先引用已审核内容
```

审核操作按钮包括：

1. 保存草稿：仅保存专家修改，不生效。
2. 诊断审核通过：完成 AI 诊断详情审核。
3. 检修记录审核通过：完成检修记录详情审核。
4. 修改后通过：当前步骤的修正内容生效，并进入回流流程。
5. 直接通过：当前步骤原 AI 内容生效，并记录审核人。
6. 驳回维修请求：整条维修请求不进入知识库，必须填写驳回原因。
7. 退回补充：要求一线人员补充照片、设备型号、告警截图或现场现象。

#### 7.10.5 回流选项

专家点击“修改后通过”时，系统应按当前审核步骤提供回流选项。

诊断详情回流选项：

1. 写入知识库：生成或更新一条知识条目，后续 RAG 可检索。
2. 更新知识图谱：新增或更新设备、部件、故障、原因、措施、影响范围关系。
3. 写入专家 Agent 经验记忆：作为工控设备专家 Agent 后续会诊上下文。
4. 仅记录审核意见：只保留审计记录，不进入知识库和图谱。

检修记录详情回流选项：

1. 更新当前检修记录：将专家修正作为记录补充或替换内容，并标记为“专家审核通过”。
2. 写入知识库：将作业步骤、安全项或恢复确认经验沉淀为知识条目。
3. 写入专家 Agent 经验记忆：作为后续检修向导和作业卡导出的参考经验。
4. 仅记录审核意见：只保留审计记录，不进入知识库。

默认建议：

1. 对诊断详情的关键修正，默认勾选“写入知识库”和“写入专家 Agent 经验记忆”。
2. 对影响范围的修正，默认勾选“更新知识图谱”。
3. 对检修步骤、安全项和恢复确认的修正，默认勾选“更新当前检修记录”。
4. 对不确定或一次性现场判断，默认只记录审核意见。

#### 7.10.6 审核状态流转

审核状态包括：

```text
待审核
  -> 诊断审核中
  -> 检修记录审核中
  -> 已审核
  -> 已驳回
  -> 退回补充
```

状态说明：

1. 待审核：一线人员或系统提交后等待专家处理。
2. 诊断审核中：专家正在审核 AI 诊断详情。
3. 检修记录审核中：诊断详情已审核，专家正在审核检修记录详情。
4. 已审核：AI 诊断详情和检修记录详情均已审核完成。
5. 已驳回：整条维修请求不进入知识库，需保留驳回原因。
6. 退回补充：需要一线人员补充信息后再次提交。

#### 7.10.7 演示用专家审核场景

主演示建议使用“工控机高温/散热异常”作为专家审核回流场景：

1. 第一次诊断时，系统给出“滤网堵塞、风扇故障、风道不畅”等原因。
2. 专家先审核 AI 诊断详情，发现系统没有体现现场经验：“沙尘环境下滤网维护周期应缩短”。
3. 专家在诊断修正区补充该经验，并选择：
   - 写入知识库。
   - 写入工控设备专家 Agent 经验记忆。
   - 更新图谱中的“沙尘环境 -> 滤网堵塞风险 -> 高温告警”关系。
4. 专家继续审核检修记录详情，补充“恢复后观察数据上传稳定不少于 10 分钟”。
5. 专家选择更新当前检修记录，并完成整条维修请求审核。
6. 审核通过后，回到异常检测入口再次提出类似问题。
7. 系统在新诊断结论中展示“专家修正 · 已审核”标签，并引用刚才的经验记录。

### 7.11 专家 Agent 经验记忆回流

专家 Agent 的经验记忆用于记录某一类专家角色在实际诊断中积累的修正经验。

回流来源：

1. 专家审核员对会诊意见的修正。
2. 专家审核员对作业卡步骤的修正。
3. 一线人员提交并审核通过的案例。
4. 后续诊断中被多次采纳的高质量经验。

回流内容：

1. 适用设备。
2. 适用故障。
3. 经验结论。
4. 触发条件。
5. 不适用条件。
6. 来源案例。
7. 审核人。
8. 生效状态。

示例：

```text
专家 Agent：工控机设备专家
适用设备：工控机
适用故障：高温/散热异常
经验记录：若夏季高温告警同时伴随机柜进风口积尘，应优先检查滤网堵塞和风扇运行状态；沙尘环境下滤网维护周期应缩短。
来源：专家修正案例
```

产品要求：

1. 专家修正后，可选择写入某个专家 Agent 的经验记录。
2. 后续同类问题会优先参考已审核经验。
3. 页面应能展示“本结论参考了某专家 Agent 的经验记录”。
4. 专家经验记录应支持启用、停用和编辑。

### 7.12 知识沉淀与更新

支持沉淀内容：

1. 一线人员上传的检修案例。
2. 一线人员填写的经验总结。
3. 专家修正后的答案。
4. 专家修正后的作业卡。
5. 新增设备、故障、原因、措施关系。
6. 专家 Agent 经验记录。
7. 文档入库流水线生成的知识条目、图谱关系建议和作业卡草稿。

要求：

1. 内容需经审核后生效。
2. 生效后应能被后续检索命中。
3. 生效后应能在知识图谱中体现。
4. 内容应保留来源和审核记录。
5. 经验记录应能绑定到具体专家 Agent。

### 7.13 Agentic 文档入库流水线

系统支持将上传的检修案例、规程片段或手册内容处理为可审核的结构化知识。

文档入库流程：

```text
文档上传
  -> 文档分类
  -> 抽取设备、部件、故障、原因、措施、安全要求
  -> 生成知识条目草稿
  -> 生成图谱关系建议
  -> 生成作业卡步骤草稿
  -> 专家审核
  -> 入库生效
```

首期支持：

1. 结构化表单录入。
2. Markdown/TXT 文本导入。
3. 文本型 PDF 内容抽取。

首期不支持：

1. 扫描版 PDF OCR。
2. 复杂版面理解。
3. 自动无审核入库。

### 7.14 诊断轨迹与可审计面板

系统应记录每次诊断过程，用于复盘、演示和测试报告。

诊断轨迹包括：

1. 用户输入。
2. 图片识别结果占位。
3. 诊断计划。
4. 检索来源。
5. 调用的专家 Agent。
6. 各专家 Agent 的结论。
7. 安全合规校验结果。
8. 最终作业卡。
9. 专家修正记录。
10. 知识回流记录。

页面展示：

1. 时间线。
2. Agent 调用顺序。
3. 证据来源。
4. 安全校验状态。
5. 修正前后对比。

### 7.15 知识库管理

支持管理：

1. 检修规程。
2. 设备手册。
3. 故障案例。
4. 专家经验。
5. 安全规范。

首期优先支持：

1. 结构化表单录入。
2. Markdown/TXT 文档导入。
3. PDF 文本内容导入作为增强。

### 7.16 设备与故障配置管理

系统应支持通过配置方式扩展设备和故障，不应把设备类型、故障类型、作业卡内容写死在代码中。

可配置内容：

1. 设备类型。
2. 设备部件。
3. 常见故障。
4. 故障现象关键词。
5. 风险等级。
6. 推荐检修等级。
7. 作业卡模板。
8. 关联知识条目。
9. 关联图谱节点和关系。

新增设备流程：

```text
新增设备类型
  -> 配置部件和常见故障
  -> 导入设备手册/规程/案例
  -> 维护图谱关系
  -> 绑定作业卡模板
  -> 生效为新的检修场景
```

产品目标：

> 后期新增设备或故障时，主要通过配置和知识库维护完成，尽量不修改代码。

首期范围：

1. 保留新增设备类型、故障类型、作业卡模板的接口设计。
2. 可在文档和演示数据中展示设备配置可扩展性。
3. 首期不单独做完整设备类型管理页面。

---

## 8. 页面规划

首期页面控制在 5 个：

1. 工作台
2. 异常检测入口
3. 步骤式检修向导
4. 检修记录与导出
5. 专家审核与知识回流

### 8.1 专家审核与知识回流页面

该页面面向专家审核员，用于完成“看得见、改得动、能回流”的审核闭环。

核心模块：

1. 待审核列表：展示待审核诊断、检修记录、案例、图谱关系和专家经验。
2. 审核详情：展示用户输入、图片占位或后续识别结果、诊断结论、证据、Agent 会诊、向导步骤和检修记录。
3. 修正对比：左侧展示 AI 原文，右侧展示专家修正内容。
4. 回流选项：选择写入知识库、更新图谱、写入专家 Agent 经验、更新检修记录或仅记录意见。
5. 审核操作：保存草稿、直接通过、修改后通过、驳回、退回补充。
6. 回流结果：显示本次修改影响了哪些知识条目、图谱关系和专家经验记录。

首期交互要求：

1. 专家至少能审核一条诊断结论和一条检修记录。
2. 专家至少能修改一条故障原因或现场经验。
3. 审核通过后，页面应提示“已写入知识库 / 已更新图谱 / 已写入专家经验”。
4. 再次诊断时，系统应能命中该专家修正内容，并展示“专家修正 · 已审核”标签。

## 9. 非功能需求

### 9.1 易用性

1. 首页直接提供诊断入口。
2. 图片上传和自动识别不作为 MVP 功能，只保留后续入口或占位标识。
3. 结果以结构化卡片展示。
4. 检修向导支持逐步执行。
5. 专家修正入口明显。
6. 异常情况给出明确提示。
7. 支持基础登录和角色区分。

### 9.1.1 MVP 界面风格

MVP 视觉风格参考“站控慧眼 / 智能诊断台”示意图。

1. 整体采用浅色工作台背景、白色内容卡片、细边框和轻阴影。
2. 主强调色固定为冷静工业蓝 `#2563EB`，悬停色 `#1D4ED8`，弱强调背景 `#EEF4FF`；如确需轻微紫调，只允许使用辅助高亮 `#6366F1`，不使用大面积蓝紫渐变。
3. 基础色固定为页面背景 `#F7F9FC`、卡片背景 `#FFFFFF`、边框 `#DDE4F0`、次级文字 `#64748B`、正文 `#0F172A`。
4. 页面结构优先采用左侧窄导航、中间诊断/向导主区域、右侧诊断流程与快捷操作。
5. 页面应呈现专业诊断台风格，不做深色工业大屏，不做复杂三维看板。
6. 图片占位区需要固定尺寸和统一样式，后续补充图片时不改变页面布局。

### 9.2 稳定性

1. 外部服务失败时显示可理解的错误。
2. 识别失败时允许人工选择。
3. 检索无结果时提示补充设备型号或故障描述。
4. 演示数据应可兜底展示。

### 9.3 可维护性

1. 知识条目应可新增、编辑、审核。
2. 故障分类应可扩展。
3. 检修步骤模板和作业卡导出模板应可维护。
4. 专家修正应可追溯。

## 10. 成功标准

### 10.1 功能成功标准

1. 用户能输入故障描述、设备类型和设备型号，并进入异常检测与故障分析流程。
2. 系统保留图片辅助输入入口和数据字段，但 MVP 不要求真实上传、管理或识别图片。
3. 系统能识别或人工确认指示灯状态，并基于灯态对照条目参与检索。
4. 故障分析结果能显示引用来源。
5. 系统能展示步骤对应的图片占位区、后续素材字段和影响范围。
6. 系统能生成步骤式检修向导。
7. 专家能修正诊断结论、检修步骤或检修记录。
8. 修正内容能影响后续问答或图谱展示。
9. 系统保留设备和故障配置能力，后续可通过配置扩展更多设备和故障。
10. 系统能展示多 Agent 协同会诊结果。
11. 专家修正能写入对应专家 Agent 的经验记录，并在后续同类诊断中被引用。
12. 系统能展示 PDF 文本证据卡片。
13. 检修向导和检修记录能经过安全合规 Guardrails 检查。
14. 系统能展示一次诊断与向导执行的关键轨迹。
15. 系统全流程保持只读不控，不向任何工控设备下发控制指令。

### 10.2 演示成功标准

1. 主流程 3-5 分钟内完整走通。
2. 每个页面有明确业务价值。
3. 输出不是普通聊天文本，而是异常定位、故障分析和步骤式检修向导。
4. 能证明系统不是传统搜索，而是“诊断摘要 + 步骤式检修”的场站工控设备检修系统。
5. 能展示“对话/异常输入 -> 大模型总结 -> 步骤选项 -> 图片占位式步骤向导 -> 检修记录 -> 专家回流”的特色闭环。

## 11. 暂不做内容

首期不做：

1. 移动 App。
2. 复杂权限体系。
3. 复杂工单系统。
4. 真实实时传感器接入。
5. 预测性维护模型训练。
6. 工控网络安全攻防分析。
7. 大规模多企业知识管理。
8. 向 PLC、SCADA、现场仪表或执行机构下发任何控制指令。
9. 接入真实生产控制网络。
10. 真实三维建模、可旋转三维模型、自动热点识别或复杂设备部件联动动画。
11. 首期真实实现连线专家、语音播报、视频识别；这些能力只保留模拟入口和预留接口。

## 12. 待确认问题

1. 需要准备第一版主演示所需的预标注步骤图片：异常定位、安全确认、风道检查、滤网/风扇检查、恢复验证。
2. 需要确认可用于演示的现场照片或 AI 生成示意图范围，并完成企业名称、站点名称、人员、人脸、敏感铭牌和内部网络信息脱敏。
3. 需要确认研华 ACP-4000 / IPC-610 工控机公开手册版本和可引用资料范围。
4. 场景化叙事已确定继续使用“某大型油气管网企业”“某输气场站”，不引用项目报告正文内容。
5. 需要确认首期是否加入官方摩托车维修手册作为通用性证明；若时间紧，可放入后续增强，不影响主线闭环。
6. 若老师建议保留 PLC 次演示，需要确认照片中 PLC 控制柜的具体品牌和型号，以锁定公开手册、指示灯状态对照表和模块命名。

## 13. 已确定产品决策

1. 核心演示已确定为步骤式检修向导；作业卡降级为检修完成后的记录与导出结果。
2. 已确定需要基础登录和角色区分，首期至少包含一线人员、专家审核员两个角色。
3. 首期保留设备类型、故障类型和作业卡模板的配置接口，但不单独做完整设备管理页面。
4. 首期采用多 Agent 协同会诊，但不引入重型 Agent SDK。
5. 首期展示四类 Agent 角色：
   - 接诊 Agent
   - 知识检索 Agent
   - 工控设备专家 Agent
   - 安全合规 Guardrails Agent
6. 用户入口采用对话/异常输入辅助识别方式：用户先描述现场现象，系统总结问题并生成步骤选项，随后进入步骤式向导。
7. 首期深度演示设备限定为研华 ACP-4000 / IPC-610 工控机整机，第一版故障固定为工控机高温告警 / 风道堵塞 / 散热异常。PLC 控制柜、通信交换机、传感器、现场仪表、电动执行机构、SCADA 监控平台等对象保留为关联知识条目、影响范围节点、备用演示素材和后续扩展。
8. 系统定位为只读不控的检修知识助手，不做工控网络安全攻防，不接入真实生产控制链路。
9. MVP 可视化方案已调整为“图片占位区 + 后续素材字段”：每个步骤先保留固定图片区域，不做图片插入、上传、真实展示、三维建模、自动热点识别或复杂三维交互。
10. 连线专家、语音播报、视频输入首期只做模拟能力和预留接口，不作为第一版真实实现范围。
