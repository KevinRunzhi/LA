# 2026-07-10 老师修改意见：动态接诊工作区

> 文档性质：最新评审意见与下一轮页面规划输入
> 文档版本：V1.2
> 反馈日期：2026-07-10
> 整理日期：2026-07-11
> 适用范围：异常接入、现场接诊工作区及其信息生成交互
> 当前状态：老师要求已按 V1.2 重新统一，已形成对应设计方案与开发 Spec；V1.1 试验页面不再作为基线，V1.2 尚未实施

## 1. 老师提出的修改意见

1. 接诊工作区的各个卡片在出现时增加动画效果，例如淡入、轻微缩放或位移进入。
2. 信息出现顺序调整为：发生时间、发生地点、发生事件及其他现象。
3. 上一步已经输入或确认的已知信息，在左上角或页面顶部缩小展示，不再作为中央主体内容。
4. 将发生地点作为接诊工作区的视觉中心：根据故障类型展示站场布局图、设备位置图或设备拓扑图。
5. 围绕中央地点图，通过动画依次出现发生时间、发生事件、其他现象、依据标准、操作手册等卡片，并与用户交互完成信息补充和确认。

### 1.1 V1.2 最终确认口径

1. 页面内容丰富后已经出现杂乱感，必须用“发生时间 → 发生地点 → 发生事件 → 其他现象 → 依据与手册”归纳出唯一业务主线。
2. 上一步已经确认的信息需要持续可见，但不能继续作为中央主体；应缩成左上角或顶部的紧凑已知信息区。
3. 中央工作区只负责当前任务的生成、补充和确认；同一时刻不允许多张完整任务卡争夺注意力。
4. 卡片动画不是装饰，而是 Agent 生成过程的主要产品表达，必须覆盖地图进入、故障点识别、卡片生成、字段逐项出现、确认收缩和归入已知信息区。
5. “接入摘要”名称不准确，最终完整结果页面统一改为“现场异常事件全景”。
6. 必须明确主次：中间工作区是与工作人员交互、完成业务确认的主要空间；右侧区域只是 Agent 思考过程、依据解释和对话的辅助空间。

## 2. 对完整产品意图的理解

这次反馈不是单纯要求“给现有卡片增加动画”，而是要求重新组织接诊工作区的信息层级和交互叙事。

接诊页面应从纵向表单式流程，调整为以故障地点为中心的动态现场态势工作区。Agent 围绕中央场站图或设备拓扑图，按照异常事件形成的逻辑顺序，逐步识别、生成并请求用户确认信息，使用户能够直观看到系统如何理解现场，而不是一次看到所有表单。

目标体验可以概括为：

> 以地点图或拓扑图为中心，以“时间—地点—事件—现象—依据”为信息主线，通过渐进动画和用户确认，形成完整、可解释、可修改的异常事件画像。

## 3. 新的信息组织顺序

接诊信息统一按照以下顺序生成：

```text
发生时间
  → 发生地点
  → 发生事件
  → 其他现象
  → 依据标准与操作手册
  → 接入信息汇总
```

各阶段含义如下：

| 阶段 | 需要表达的信息 | 典型交互 |
| --- | --- | --- |
| 发生时间 | 首次发现时间、持续时长、是否反复出现 | 自动提取后由用户确认或修改 |
| 发生地点 | 场站、区域、机柜、设备或拓扑节点 | 在中央图中标点、选择或修正节点 |
| 发生事件 | 温度告警、停机、通信中断、异响等主要异常 | Agent 分类，用户确认事件类型 |
| 其他现象 | 指示灯、转速、温度、声音、环境和现场材料 | 动态补充字段、图片、视频与音频 |
| 依据与资料 | 告警标准、设备手册、操作规程、历史案例 | 展示当前判断来源，并允许查看详情 |
| 异常事件确认 | 已确认的时间、地点、事件、现象和依据 | 形成“现场异常事件确认”，确认后进入诊断 |

## 4. 页面空间结构

### 4.1 左上角或顶部：已知信息区

已确认信息采用紧凑信息胶囊或小卡片持续展示：

```text
已知信息
时间  10:25
地点  某输气场站 · 站控柜 A01
事件  工控机温度告警
现象  风扇低速 · 异响 · TEMP/FAN 告警
```

规则：

- 只显示已经确认的内容，未生成信息不占空位；
- 单项保持一行短摘要，禁止在顶部放大段描述或完整表单；
- 新确认项从中央任务卡收缩并进入已知信息区，进入后短暂高亮；
- 点击已知信息项可以返回修改；
- 顶部已知信息区用于过程上下文，最终仍进入独立“现场异常事件全景”进行完整核对。

### 4.2 中央：地点图或设备拓扑图

中央区域始终用于展示“故障发生在哪里”，并根据案例切换表达方式：

- 场站级异常：站场布局图；
- 机柜或设备异常：设备位置图；
- 通信、控制或数据链路异常：设备拓扑图。

中央图需要明确展示故障点、当前选中节点和位置标题，并允许用户修正识别结果。任何浮层、标签或卡片都不得遮挡故障点和关键图像内容。

### 4.3 中央图周围：动态信息卡

周边卡片围绕地点图依次出现，而不是全部纵向堆叠：

```text
                   发生时间

       其他现象    地点图/拓扑图    发生事件

              依据标准    操作手册
```

该结构表达的是信息与故障点之间的关系，不要求机械地做成完全对称的环形布局。实际设计需要优先保证中央图足够大、文字可读、现场材料可查看，并适配常用屏幕尺寸。

## 5. 卡片状态与动画原则

每张卡片应具有明确的生成状态：

```text
尚未生成 → Agent 生成中 → 等待用户确认 → 已确认并缩小
```

建议的动画语言：

- 新卡片出现：淡入、轻微上移或从关联方向进入，同时进行小幅缩放；
- 当前交互卡：清晰高亮，成为周边信息中的视觉焦点；
- 确认完成：轻微收缩、降低强调度，保留摘要；
- 下一张卡片：在上一项确认后再出现，避免同时抢占注意力；
- 中央故障点变化：使用短促高亮或脉冲提示，不遮挡原图；
- `prefers-reduced-motion` 环境下关闭非必要动画。

动画必须用于说明“信息刚生成、当前需要确认、已经完成”，不能成为循环播放的装饰，也不能拖慢接诊操作。

## 6. Agent 与用户交互方式

周边卡片不是静态信息展示，而是 Agent 发起的确认任务：

- 已识别内容：展示系统识别值和来源；
- 存在歧义：展示候选项，请用户选择；
- 信息缺失：提供输入、选择或材料上传入口；
- 用户修改：标记为工程师确认值，并保留原建议及来源；
- 确认完成：当前卡缩小并移动到左上角或顶部已知信息区，新信息胶囊短暂高亮；全部业务步骤结束后再汇入“现场异常事件确认”。

右侧辅助对话继续承担解释职责，说明为什么生成当前卡片、使用了哪些已知信息、依据了什么标准，以及确认后将生成什么内容。右侧不能承载业务必填字段、主要确认按钮或决定流程是否可以继续的操作。

## 7. 页面主次关系

### 中间工作区：业务主空间

中间区域负责工作人员真正需要完成的任务：查看地点图或拓扑图、确认故障点、补充时间和事件、核对其他现象、查看材料并完成业务确认。页面标题、主要按钮、当前任务状态和错误恢复全部属于中间区域。

### 右侧区域：Agent 辅助空间

右侧只负责展示 Agent 思考过程、生成原因、字段来源、依据解释、风险提醒和追问对话。它应使用更低的视觉强调度，宽度稳定且可收缩；右侧内容再多也不能挤压中央图或改变业务卡片位置。

两者的关系是：

> 中间区域负责“做什么和确认什么”，右侧区域负责“为什么这样做”。

## 8. 与当前实现的关系

当前动态接诊状态机、字段来源、条件分支、材料上传预览、地点图和最终信息确认逻辑继续保留。下一轮主要调整信息顺序、页面空间结构、主次关系和生成动画，不应推翻已经验证的业务规则。

需要重点重新设计的部分：

- 补充“发生时间”任务；
- 建立紧凑的左上角或顶部已知信息区，只沉淀已经确认的时间、地点、事件和现象；
- 将地点图提升为中央长期主视觉；
- 将设备、事件、现象、依据和手册改为围绕中央图生成的任务卡；
- 重新定义卡片生成、确认和收缩动画；
- 增加“任务卡收缩并归入已知信息区”的连续动画；
- 保证图片、视频和音频仍有清晰、可用的展示空间；
- 将“接入摘要”统一改为“现场异常事件确认”，并按业务主线重新组织内容；
- 降低右侧 Agent 区域视觉权重，禁止在右侧放置核心业务操作。

- 动画时长、触发条件和并发规则；
