# R0.1【P0】中心态势式动态接诊工作区改进方案

> 版本：V1.2
> 日期：2026-07-11
> 需求来源：`teacher-feedback-2026-07-10-dynamic-intake-workspace.md`
> 设计目标：把现有纵向动态卡片接诊，升级为以故障地点图或设备拓扑图为中心的动态现场态势工作区
> 当前状态：V1.1 第一版布局未达到老师要求；V1.2 已重新确认“顶部沉淀已知信息＋中央动态生成任务”的方向，等待重新实施

## 1. 本轮改进结论

本轮不推翻已经完成的 R0 状态机、条件分支、字段来源、材料预览和最终信息确认逻辑，而是重新组织接诊页面的信息顺序、空间关系和视觉主次。

页面从“中央堆叠任务卡”调整为：

```text
顶部已知信息区持续沉淀已确认内容
        ↓
地点图 / 设备拓扑图长期作为主视觉
        ↓
周边按照时间、事件、现象、依据与手册依次生成交互卡片
        ↓
确认完成的卡片收缩并归入顶部已知信息区
        ↓
最后形成独立“现场异常事件确认”并启动诊断
```

核心产品表达是：Agent 正在围绕一个真实故障点逐步形成异常事件画像，而不是要求工程师填写一套已经暴露的固定表单。

## 2. 信息架构

### 2.1 一级信息层级

| 层级 | 内容 | 页面职责 |
| --- | --- | --- |
| 已知上下文 | 首页描述及已经确认的时间、地点、事件、现象 | 左上角或顶部紧凑展示，不占据中央主体 |
| 空间主上下文 | 场站布局、机柜位置、设备节点或拓扑链路 | 中央长期展示，是页面视觉中心 |
| 当前确认任务 | 当前待确认的时间、事件或现象卡 | 周边高亮展示，承载主要交互 |
| 已确认任务 | 已完成的时间、事件、现象和依据 | 周边收缩为摘要，不再抢占视觉焦点 |
| 解释上下文 | 生成原因、字段来源、规则和下一步 | 右侧辅助对话持续展示 |

### 2.2 接诊生成顺序

```text
首页已知描述
  → 发生时间
  → 发生地点
  → 发生事件与设备对象
  → 其他现象与现场材料
  → 依据标准与操作手册
  → 现场异常事件确认
```

其中“设备对象”并入发生事件，用于回答“哪台设备发生了什么”；告警值、温度、转速、指示灯和音视频材料归入“其他现象”。

## 3. 桌面端页面结构

页面继续保留左侧诊断流程和右侧辅助对话，但必须明确中央主、右侧辅。已知信息以紧凑条位于中央业务区顶部，中央主体负责当前任务，右侧只解释 Agent 的思考与依据。

```text
┌───────┬────────────────────────────────────────────┬─────────────┐
│ 诊断  │ 已知信息：时间 · 地点 · 事件 · 现象（紧凑） │ Agent 辅助   │
│ 流程  ├────────────────────────────────────────────┤ 思考与对话   │
│       │ 当前业务：确认异常发生时间                  │ 生成原因     │
│ 默认  │              发生时间卡                    │ 当前建议     │
│ 收缩  │ 其他现象卡   地点图 / 设备拓扑   发生事件卡 │ 字段来源     │
│       │                                            │ 依据检索     │
│       │          依据 / 手册（按顺序生成）          │ 下一步建议   │
└───────┴────────────────────────────────────────────┴─────────────┘
```

### 3.1 顶部已知信息区与唯一业务主线

顶部已知信息区位于中央业务工作区内部，高度控制在 `52–64px`。采用横向信息胶囊或紧凑小卡，只展示已经确认的内容：

- 初始报障：截断为一行；
- 发生时间：时间和持续时长；
- 发生地点：场站、区域和机柜；
- 发生事件：设备对象和主要告警；
- 其他现象：运行值和材料数量的短摘要。

未确认项不显示空占位。每次用户确认后，中央任务卡通过收缩与位移动画归入顶部，并生成对应信息胶囊。点击胶囊可以返回修改。

页面唯一业务主线为：

```text
顶部查看已知内容
  → 中央处理当前任务
  → 确认后归入顶部
  → 中央生成下一项任务
```

### 3.2 中央空间主视觉

中央主视觉至少占中间业务工作区可用面积的 50%，并在整个接诊阶段保持尺寸稳定。中央业务工作区占除左侧流程外可用宽度的约 70%–76%，右侧 Agent 区域控制在约 24%–30%。

根据事件类型选择内容：

| 场景 | 中央内容 |
| --- | --- |
| 场站或区域尚未确定 | 场站布局图 |
| 已定位到站控柜或设备 | 带故障点的设备位置图 |
| 通信中断或数据不上送 | 设备拓扑图及疑似异常链路 |

中央图只保留位置标题、故障点、当前节点和必要图例。识别解释、匹配结果和长文字继续放在右侧辅助对话，禁止覆盖图像。

### 3.3 周边卡片槽位

使用固定 CSS Grid 槽位，不使用随机坐标和自由漂浮：

| 槽位 | 默认内容 | 说明 |
| --- | --- | --- |
| `north` | 发生时间 | 横向紧凑卡，最先出现 |
| `east` | 发生事件 | 设备、告警分类和事件类型 |
| `west` | 其他现象 | 灯态、温度、转速和紧凑材料宫格 |
| `south-west` | 依据标准 | 告警阈值、规则与历史案例 |
| `south-east` | 操作手册 | 设备手册、规程和后续可用资料 |
| `center` | 地点图 / 设备拓扑 | 始终保留，不被周边卡片覆盖 |

当前任务卡可以适度扩大。为避免信息再次变乱，同一时刻只展示一张完整任务卡；已确认卡不再留在中央，而是全部归入顶部已知信息区。卡片变化不能改变中央图尺寸。

### 3.4 中央与右侧的职责边界

| 中央业务工作区（主） | 右侧 Agent 区域（辅） |
| --- | --- |
| 当前业务任务和状态 | Agent 正在分析什么 |
| 场站图、设备图或拓扑图 | 为什么生成当前任务 |
| 时间、地点、事件和现象输入 | 字段建议来自哪里 |
| 确认、修改、继续、重试按钮 | 规则、手册和风险解释 |
| 材料查看与业务错误恢复 | 追问与辅助建议 |

右侧禁止出现业务必填字段、唯一的继续按钮或必须点击后才能推进的操作。右侧内容溢出时独立滚动或收缩，不得压缩中央图。

## 4. 各卡片内容设计

### 4.1 发生时间卡

字段：

- 首次发现日期与时间；
- 持续时间；
- 是否反复发生。

默认从当前演示时间或描述中提取建议值，但必须标记“系统建议”，由工程师确认。第一版不做真实自然语言时间解析。

### 4.2 发生地点卡与中央图

地点确认主要发生在中央图内，周边只显示轻量操作：

- 当前识别路径；
- “确认位置”；
- “修改位置”；
- 候选节点选择。

确认后中央图继续保留，地点卡收缩为位置短摘要；最终统一进入“现场异常事件确认”。

### 4.3 发生事件卡

字段：

- 设备型号；
- 设备角色；
- 主要事件类型；
- 关联告警。

继续沿用逐字段生成和来源标签。用户修改设备或告警后，重新计算散热、无灯态高温和设备/告警不一致分支。

### 4.4 其他现象卡

字段与内容：

- 指示灯与蜂鸣器；
- 风扇转速、系统温度、CPU 温度；
- 环境与其他可见现象；
- 图片、视频和音频紧凑宫格；
- 添加、删除、替换和点击预览入口。

现场材料不再作为中央主视觉，但必须保持清晰可见。默认显示缩略图，点击后使用现有独立预览层放大。

### 4.5 依据标准卡

只展示与当前判断直接相关的 2–4 条依据：

- 告警阈值；
- 设备台账；
- 现场描述；
- 历史案例或知识图谱节点。

每条依据必须能说明“它支持哪个字段或哪项判断”，不能只列资料名称。

### 4.6 操作手册卡

展示已命中的手册和规程摘要：

- 文档名称；
- 适用设备；
- 命中章节或检查主题；
- 后续诊断或检修阶段将如何使用。

第一版使用现有静态数据，不实现真实文档检索和全文阅读。

## 5. 卡片状态与视觉优先级

统一状态：

| 状态 | 视觉 | 行为 |
| --- | --- | --- |
| `hidden` | 完全不展示 | 尚未生成 |
| `generating` | 骨架或 Agent 生成提示 | 禁止交互 |
| `active` | 高对比边框、完整字段 | 等待用户确认 |
| `confirmed` | 收缩、降低强调度 | 展示摘要，可重新编辑 |
| `stale` | 琥珀提示 | 上游修改后等待重新评估 |
| `blocked` | 明确原因与恢复动作 | 当前分支不能继续 |

同一时刻只能有一张主要 `active` 卡片。中央地点图可以保持激活标记，但不得与周边当前任务竞争主要按钮。

## 6. 动画编排方案

动画用于表达状态变化，不使用循环装饰。

| 场景 | 动画 | 建议时长 |
| --- | --- | --- |
| 地图进入 | 淡入 + `0.94 → 1` 缩放，可配一次轻量扫描 | 500–700ms |
| 故障点识别 | 标记缩放弹出 + 两次短促脉冲 + 位置名淡入 | 700–1000ms |
| 卡片容器生成 | 淡入 + 16px 位移 + `0.88 → 1` 缩放 | 360–480ms |
| 标题与说明出现 | 标题、生成原因分两段淡入 | 间隔 160–240ms |
| 字段逐项推荐 | 字段上移淡入并高亮来源标签 | 每项间隔 300–450ms |
| 确认按钮出现 | 最后淡入并轻微上移 | 220–280ms |
| 用户确认 | 边框转完成色，完整卡收缩为摘要 | 240–320ms |
| 归入顶部 | 摘要向已知信息区移动，新胶囊放大后回落 | 360–520ms |
| 下一卡片出现 | 上一项归档后短暂停顿再生成 | 220–300ms |
| 上游修改 | 受影响胶囊和当前卡短促琥珀高亮 | 240–320ms |

约束：

- 单个业务步骤的首次生成动画控制在 1.2–2.5 秒；
- 不使用大幅旋转、弹跳或无限呼吸；
- 按钮在动画结束前不得发生位置跳变；
- 支持 `prefers-reduced-motion: reduce`；
- 龙芯开发态需要验证动画不造成明显掉帧。

## 7. 交互流程与业务主线

```text
1. 首页提交描述和现场材料
2. 顶部已知信息区先显示初始报障短摘要
3. 发生时间卡在中央生成，字段逐项出现；确认后收缩并归入顶部
4. 中央地点图进入识别状态，用户确认故障点
5. 发生事件卡生成，逐字段补充设备与告警
6. 其他现象卡生成，核对运行值并补充材料
7. 依据标准和操作手册按需展开，解释当前判断
8. Agent 形成完整异常事件信息
9. 进入独立“现场异常事件确认”页面
10. 用户核对最终信息后启动多 Agent 诊断
```

当用户返回修改：

- 修改时间：只更新对应收缩卡和最终事件信息；
- 修改地点：设备、事件、现象和依据标记为 `stale`，重新匹配台账；
- 修改设备或事件：现象字段、条件分支、依据和手册重新生成；
- 修改告警或运行值：重新评估诊断方向和最终事件信息，不强制重新确认地点。

## 8. 响应式策略

### 宽屏（≥ 1440px）

完整展示中央图和五个固定周边槽位。

### 常规桌面（1180–1439px）

中央图保持主视觉；时间卡位于顶部，事件与现象分列，依据与手册进入底部横向栏。

### 窄屏（< 1180px）

不强行维持环绕布局。改为：顶部已知信息区 → 中央地点图 → 当前任务卡；右侧辅助对话改为抽屉。

本项目主要验收桌面端，但窄屏不能出现卡片覆盖中央图、横向溢出或无法点击。

## 9. 保留、替换与暂缓

### 继续保留

- 首页材料选择与本地预览；
- 三条预制条件分支；
- 设备字段逐项推荐及来源；
- 独立最终信息确认逻辑；
- 右侧 Agent 生成原因和依据；
- 左侧可收缩诊断流程；
- 后续多 Agent 诊断和 R2 检修预方案。

### 本轮替换

- 原始描述大卡 → 顶部已知信息区中的一行起始事实；
- 中央材料宫格 → 中央地点图 / 拓扑图；
- 位置独立小卡 → 中央图内的位置确认状态；
- 纵向任务堆叠 → 固定周边槽位；
- 设备/告警命名 → 事件/其他现象信息架构。
- “接入摘要”页面 → “现场异常事件确认”页面；
- 右侧与中央近似同权 → 中央业务主空间、右侧 Agent 辅助空间。
- 底部“已确认步骤”轨迹 → 顶部已知信息区。

### 暂缓

- 自由拖拽卡片；
- 图与依据之间的动态连线；
- 真实拓扑自动生成；
- 真实图片、视频和语音识别；
- 真实手册全文检索；
- 复杂时间语义解析。

## 10. 下一步开发顺序

1. 调整接诊数据结构和状态顺序；
2. 建立顶部已知信息区、当前业务标题和中央空间主视觉；
3. 将现有设备、告警和材料逻辑迁入周边槽位；
4. 新增发生时间卡；
5. 新增依据标准与操作手册卡；
6. 接入地图、故障点、卡片、字段、确认和归入顶部的完整动画编排；
7. 建立“现场异常事件确认”并更新左侧流程；
8. 完成桌面端、窄屏、减弱动画和龙芯性能验证。
