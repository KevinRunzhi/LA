# 正式 MVP 开发计划与功能边界

## 1. 文档目的

本文档用于明确“站控慧眼：工控设备检修助手”从当前高保真演示骨架进入正式 MVP 开发时的范围、功能边界、实现顺序和验收标准。

正式 MVP 的目标不是一次性做完整产品，而是先做出一个可真实运行、可演示闭环、可在龙芯环境继续部署迭代的最小版本。

第一版开发原则是“演示流程优先”。能做真实功能的地方做真实功能；短期不适合完全实现的地方，可以做半真实功能或模拟功能，但必须让演示链路完整、稳定、可信。

## 2. 当前状态判断

当前项目已经具备高保真演示型 MVP 原型基础：

- 已完成首页智能诊断入口。
- 已完成账号密码登录页。
- 已完成异常接入、分析诊断、检修向导、检修记录、知识图谱、设置等主要页面。
- 已完成固定故障场景：油气管道场站内站控柜工控机高温、风扇异常、风道/滤网堵塞相关故障。
- 已完成步骤式检修向导骨架，并接入多张演示图片。
- 已完成辅助对话框的模拟大模型回复效果。
- 已完成分析诊断阶段的 Agent 动态执行感。
- 已完成场站图定位提示和蓝色故障点标记。

因此，下一阶段重点不再是继续堆页面，而是把关键链路接成可运行的最小真实功能。

## 3. 正式 MVP 核心目标

正式 MVP 要跑通一条完整闭环：

1. 用户登录系统。
2. 用户在首页输入现场异常现象。
3. 系统进入异常接入阶段，补充设备型号、位置、角色、告警与阈值信息。
4. 用户触发分析诊断。
5. 后端调用云端大模型 API，生成结构化诊断结论。
6. 系统根据诊断结论进入预设检修向导。
7. 工程师逐步勾选检修检查项。
8. 系统生成检修记录。
9. 检修记录可以本地保存并在记录页查看。

第一版只服务一个主演示场景：站控柜内 ACP-4000 / IPC-610 工控机散热异常。

## 4. 第一版真实功能范围

### 4.1 登录

第一版只做账号和密码登录。

需要实现：

- 登录表单。
- 基础账号密码校验。
- 登录后进入工作台。
- 退出登录。

建议实现方式：

- 前期可以使用后端固定账号。
- 不做复杂权限系统。
- 不做手机号、验证码、厂站选择、身份认证。

### 4.2 首页诊断入口

需要实现：

- 用户输入现场现象。
- 支持使用默认演示问题快速进入流程。
- 提交后进入异常接入阶段。

第一版不需要真实上传图片、语音、视频，但 UI 可以保留入口。

### 4.3 异常接入

需要实现：

- 描述现场现象。
- 补充设备型号。
- 补充设备位置。
- 补充设备角色。
- 选择关联告警。
- 修改灯值和阈值信息。
- 接入信息确认。

实现方式：

- 设备型号、位置、角色、关联告警采用下拉选项。
- 选项第一版可以使用固定演示数据。
- 阈值支持页面内编辑，不要求接入真实采集系统。

### 4.4 分析诊断

需要实现：

- 点击“触发诊断”。
- 展示多 Agent 动态执行过程。
- 调用后端诊断接口。
- 后端调用云端大模型 API。
- 返回结构化诊断结论。
- 前端按结构化字段展示结论。

需要保留的演示效果：

- 分析诊断 Agent、操作合规 Agent、知识检索 Agent 逐个执行。
- 每个 Agent 有“运行中 / 已完成”状态。
- 最终诊断结论使用流式或近似流式输出。

### 4.5 检修向导

需要实现：

- 按步骤展示检修流程。
- 每个步骤包含多个检查项。
- 检查项支持勾选完成。
- 点击右侧步骤可以跳转。
- 中间区域显示对应步骤图片。
- 图片切换保留淡入淡出效果。

第一版检修步骤以当前固定流程为准：

1. 确认告警与定位设备。
2. 安全准备。
3. 检查机柜环境与风道。
4. 检查滤网、风扇和接线。
5. 恢复运行与验证。

### 4.6 检修记录

需要实现：

- 根据诊断结论和检修向导完成状态生成记录。
- 检修记录页以列表形式展示。
- 支持点击记录查看详情。
- 支持按故障关键词搜索。

第一版不做 Word 或 PDF 文件导出，优先实现“导出为打印视图”。

打印视图定位：

- 不是简单截图，也不是空模板。
- 是从本次诊断会话、异常接入信息、诊断结论、检修步骤完成状态中自动汇总出来的作业卡页面。
- 用户点击“导出作业卡”后，进入一个适合浏览器打印的作业卡详情页。
- 后续可以基于同一份结构化数据扩展 Word/PDF 导出。

打印视图应包含：

- 作业卡编号。
- 场站和设备位置。
- 设备型号和设备角色。
- 检修人员和生成时间。
- 异常现象描述。
- 关联告警与阈值。
- 诊断结论。
- 疑似故障原因。
- 安全措施。
- 检修步骤和检查项完成状态。
- 恢复验证结果。
- 处理结论。
- 专家审核意见占位。

### 4.7 知识图谱

需要保留。

第一版实现目标：

- 展示节点图样式。
- 展示设备、告警、原因、知识条目、检修步骤之间的关系。
- 作为演示和赛题要求体现，不做复杂图数据库推理。

## 5. 第一版模拟能力范围

以下能力第一版保留 UI 和状态标识，但不做真实能力：

- 图片上传识别。
- 视频输入。
- 语音输入。
- 专家连线。
- 真实三维建模。
- 实时设备采集。
- 真实工控通信。
- 真实知识图谱推理。
- 复杂多设备、多故障自动扩展。

这些能力可以在 UI 上体现“入口已预留”或“后续接入”，但不要影响主流程演示。

## 6. 大模型 API 接入边界

第一版只接一个后端诊断接口。

建议接口：

```text
POST /api/diagnosis
```

请求字段建议：

```json
{
  "symptom": "站控柜内工控机温度告警，风扇声音异常，前面板风扇转速很低。",
  "device": {
    "model": "研华 ACP-4000 / IPC-610",
    "location": "某输气场站 · 站控柜 A01",
    "role": "站控工控机"
  },
  "alarms": ["TEMP/FAN 告警", "风扇转速偏低"],
  "thresholds": {
    "fanRpm": "< 500 rpm",
    "systemTemperature": "> 55°C",
    "cpuTemperature": "> 70°C"
  }
}
```

返回字段建议：

```json
{
  "summary": "疑似工控机散热系统异常。",
  "faultType": "散热异常 / 风道堵塞 / 滤网积尘",
  "confidence": "高",
  "evidence": [
    "TEMP/FAN 告警",
    "风扇转速偏低",
    "系统温度和 CPU 温度超过阈值"
  ],
  "possibleCauses": [
    "滤网积尘",
    "风道受阻",
    "风扇低速或异常",
    "机柜环境温度偏高"
  ],
  "recommendedGuide": "ACP4000_IPC610_cooling_fault_repair",
  "safetyNotes": [
    "检修前正常关机并断开所有电源",
    "执行防静电措施",
    "恢复运行后连续观察不少于 10 分钟"
  ]
}
```

前端必须按结构化字段展示，不依赖大模型自由文本排版。

## 7. 数据存储建议

第一版建议使用 SQLite。

原因：

- 部署简单。
- 不依赖 Docker。
- 适合龙芯虚拟机环境。
- 足够支持登录、诊断记录、检修记录和步骤状态保存。

### 7.1 数据库设计原则

数据库设计要服务两个目标：

1. 支撑第一版演示闭环：能保存一次完整诊断和检修过程。
2. 为后续真实功能留接口：后续接入大模型、导出 Word/PDF、专家审核、知识回流时，不需要推翻表结构。

第一版不追求复杂范式，但要把“本次检修过程如何发生”记录下来，而不仅是保存一条标题和时间。

建议记录粒度：

- 一次诊断会话。
- 用户输入的异常接入信息。
- Agent 分析过程。
- 诊断结论。
- 使用的检修向导。
- 每个步骤和检查项的完成状态。
- 最终生成的作业卡。
- 专家审核或回流意见。

### 7.2 数据表初稿

#### `users`

用途：保存登录用户。

核心字段：

- `id`：用户 ID。
- `username`：登录账号。
- `password_hash`：密码哈希。第一版可先用固定账号，后续再替换。
- `display_name`：展示名称，例如“李师傅”。
- `role`：角色，例如“一线检修人员”。
- `team`：班组。
- `created_at`：创建时间。

第一版实现建议：

- 可以先预置一个用户。
- 不做复杂权限。
- 登录通过后将 `user_id` 关联到诊断会话和检修记录。

#### `diagnosis_sessions`

用途：记录一次从首页输入到检修闭环的完整会话。

核心字段：

- `id`：会话 ID。
- `session_no`：会话编号，便于展示。
- `user_id`：发起人。
- `status`：状态，例如 `intake`、`diagnosing`、`guiding`、`completed`。
- `scenario_code`：场景编码，例如 `ipc_cooling_fault_demo`。
- `started_at`：开始时间。
- `completed_at`：完成时间。
- `created_at`：创建时间。
- `updated_at`：更新时间。

说明：

- 这是主表。
- 后续诊断输入、诊断结果、检修步骤、作业卡都挂到该会话下。

#### `diagnosis_inputs`

用途：保存异常接入阶段的信息。

核心字段：

- `id`：输入记录 ID。
- `session_id`：关联诊断会话。
- `symptom_text`：用户描述的现场现象。
- `device_model`：设备型号。
- `device_location`：设备位置。
- `device_role`：设备角色。
- `alarms_json`：关联告警列表，JSON 存储。
- `thresholds_json`：阈值信息，JSON 存储。
- `attachments_json`：图片、视频、语音等材料占位，第一版可为空。
- `created_at`：创建时间。

说明：

- 第一版图片、语音、视频不做真实上传，但字段先保留。
- 后续接入材料上传时，不需要修改主流程。

#### `agent_runs`

用途：保存分析诊断阶段各 Agent 的运行状态。

核心字段：

- `id`：Agent 运行记录 ID。
- `session_id`：关联诊断会话。
- `agent_name`：Agent 名称，例如“分析诊断 Agent”。
- `agent_type`：类型，例如 `analysis`、`safety`、`knowledge`。
- `status`：`pending`、`running`、`completed`、`failed`。
- `summary`：本 Agent 的简短输出。
- `evidence_json`：命中的证据或知识条目。
- `started_at`：开始时间。
- `completed_at`：完成时间。

第一版实现建议：

- 可以先写入模拟 Agent 运行记录。
- 后续接大模型后，Agent 输出可由真实接口填充。

#### `diagnosis_results`

用途：保存诊断结论。

核心字段：

- `id`：诊断结论 ID。
- `session_id`：关联诊断会话。
- `summary`：诊断摘要。
- `fault_type`：故障类型。
- `confidence`：置信度，例如“高”。
- `evidence_json`：证据列表。
- `possible_causes_json`：疑似原因列表。
- `recommended_guide_code`：推荐检修向导编码。
- `safety_notes_json`：安全提醒。
- `raw_model_output`：大模型原始输出，便于调试。
- `created_at`：创建时间。

说明：

- 前端应主要读取结构化字段展示。
- `raw_model_output` 只作为调试和追溯，不直接作为页面排版来源。

#### `maintenance_guides`

用途：保存检修向导模板。

核心字段：

- `id`：向导 ID。
- `guide_code`：向导编码，例如 `ACP4000_IPC610_COOLING_FAULT`。
- `title`：向导标题。
- `device_model`：适用设备。
- `fault_type`：适用故障类型。
- `version`：版本。
- `description`：说明。
- `created_at`：创建时间。

第一版实现建议：

- 可以从现有 JSON seed data 初始化。
- 第一版只有一个主演示向导。

#### `maintenance_steps`

用途：保存检修向导中的大步骤模板。

核心字段：

- `id`：步骤 ID。
- `guide_id`：关联向导。
- `step_code`：步骤编码。
- `order_no`：排序。
- `title`：步骤标题。
- `description`：步骤说明。
- `safety_note`：安全提醒。
- `default_image`：默认图片路径。

说明：

- 这是模板表，不记录某次检修是否完成。

#### `maintenance_check_items`

用途：保存每个大步骤下的检查项模板。

核心字段：

- `id`：检查项 ID。
- `step_id`：关联大步骤。
- `item_code`：检查项编码。
- `order_no`：排序。
- `label`：检查项名称。
- `instruction`：操作说明。
- `image`：对应图片路径。
- `required`：是否必选。

说明：

- 例如“TEMP/FAN 告警解除”“风扇 > 500 rpm”“FAN1/FAN2 顺序”。

#### `maintenance_records`

用途：保存某次会话实际进入检修向导后的检修记录。

核心字段：

- `id`：检修记录 ID。
- `record_no`：检修记录编号。
- `session_id`：关联诊断会话。
- `guide_id`：使用的检修向导。
- `user_id`：检修人员。
- `equipment_name`：设备名称。
- `equipment_model`：设备型号。
- `fault_summary`：故障摘要。
- `status`：`in_progress`、`completed`、`reviewed`。
- `started_at`：开始时间。
- `completed_at`：完成时间。
- `conclusion`：处理结论。
- `created_at`：创建时间。

说明：

- 检修记录页主要读取此表。
- 点击详情后，再读取步骤和检查项完成情况。

#### `maintenance_step_records`

用途：保存某次检修中每个大步骤的完成状态。

核心字段：

- `id`：记录 ID。
- `maintenance_record_id`：关联检修记录。
- `step_id`：关联模板步骤。
- `status`：`pending`、`active`、`completed`。
- `started_at`：开始时间。
- `completed_at`：完成时间。
- `note`：本步骤备注。

#### `maintenance_check_records`

用途：保存某次检修中每个检查项的勾选状态。

核心字段：

- `id`：记录 ID。
- `maintenance_record_id`：关联检修记录。
- `step_record_id`：关联步骤记录。
- `check_item_id`：关联检查项模板。
- `checked`：是否完成。
- `checked_at`：完成时间。
- `checked_by`：操作人。
- `value_text`：现场记录值，例如“告警解除”“风扇恢复正常”。
- `note`：备注。

说明：

- 这是作业卡生成的关键表。
- 打印视图中的“检查项完成状态”从这里生成。

#### `job_cards`

用途：保存最终生成的作业卡。

核心字段：

- `id`：作业卡 ID。
- `job_card_no`：作业卡编号。
- `maintenance_record_id`：关联检修记录。
- `session_id`：关联诊断会话。
- `title`：作业卡标题。
- `status`：`draft`、`generated`、`printed`、`reviewed`。
- `print_snapshot_json`：打印视图快照数据。
- `generated_at`：生成时间。
- `printed_at`：打印时间。

说明：

- 第一版点击“导出作业卡”时，可以生成或更新该表。
- `print_snapshot_json` 用于保存当时打印页面的数据快照，避免后续记录被修改后影响已生成作业卡。

#### `expert_reviews`

用途：保存专家审核或模拟专家意见。

核心字段：

- `id`：审核 ID。
- `session_id`：关联诊断会话。
- `maintenance_record_id`：关联检修记录。
- `reviewer_name`：专家名称。
- `review_status`：`pending`、`approved`、`needs_revision`。
- `comment`：审核意见。
- `created_at`：创建时间。

第一版实现建议：

- 可以先写入模拟专家意见。
- UI 上保留“专家审核/回流”标识。

### 7.3 第一版可简化实现

如果第一版开发时间紧，可以先用以下简化表落地：

- `users`：用户账号。
- `diagnosis_sessions`：诊断会话。
- `diagnosis_inputs`：异常接入输入。
- `diagnosis_results`：诊断结论。
- `maintenance_records`：检修记录。
- `maintenance_check_records`：检查项完成状态。
- `job_cards`：打印作业卡快照。

模板类数据可以暂时保留在 JSON 中，后续再迁移到 `maintenance_guides`、`maintenance_steps`、`maintenance_check_items`。

如果开发时间非常紧，也可以先使用本地 JSON 文件模拟数据库，但正式 MVP 更建议直接上 SQLite。

## 8. 作业卡打印视图设计

### 8.1 功能边界

第一版只做“浏览器打印视图”，不做 Word 或 PDF 文件导出。

用户路径：

1. 完成检修向导。
2. 点击“生成作业卡”或“导出作业卡”。
3. 系统生成作业卡数据快照。
4. 打开作业卡打印视图页面。
5. 用户使用浏览器打印功能保存为纸质文件或 PDF。

### 8.2 打印视图内容结构

打印视图建议分为以下区域：

- 标题区：作业卡编号、设备名称、故障类型、生成时间。
- 基本信息：场站、位置、设备型号、检修人员、班组。
- 异常接入：现场现象、关联告警、阈值信息。
- 诊断结论：故障摘要、疑似原因、诊断证据、安全提醒。
- 检修过程：按大步骤展示检查项，显示已完成/未完成。
- 恢复验证：告警、风扇、温度、数据上传状态。
- 处理结论：最终处理结果和备注。
- 审核意见：专家或负责人审核意见占位。

### 8.3 打印视图实现建议

- 前端单独做一个 `JobCardPrintView` 组件。
- 通过 `job_card_id` 或 `maintenance_record_id` 读取数据。
- 页面样式使用白底、黑字、弱边框，适合打印。
- 使用 CSS `@media print` 隐藏导航、按钮和辅助对话。
- 打印按钮调用 `window.print()`。
- 后续 Word/PDF 导出可以复用同一份 `print_snapshot_json`。

### 8.4 第一版不做

- 不做复杂排版模板编辑。
- 不做 Word 导出。
- 不做后端 PDF 生成。
- 不做电子签章。
- 不做正式审批流。

## 9. 真功能、半真功能与模拟功能对照表

- 模拟功能：只做 UI、状态和演示效果，不接真实能力。

## 10. 演示优先边界

第一版必须始终服务答辩演示和主流程闭环。

优先级从高到低：

1. 主流程完整、稳定、可复现。
2. 页面展示可信，有工程检修步骤感。
3. 数据能在关键节点保存和回看。
4. 大模型能力能产生结构化诊断结论。
5. 模拟能力不穿帮、不影响主流程。

可以接受的半真实处理：

- Agent 运行过程可以是预设步骤，但输出要符合当前故障上下文。
- 知识图谱可以是预设关系，但要和当前设备、告警、原因、步骤一致。
- 图片、语音、视频可以只保留入口，但不要在主流程中强依赖。
- 作业卡可以先做打印视图，不做真实文件导出。
- 专家审核可以先做模拟意见，不接真实专家系统。

不建议第一版投入过多精力的内容：

- 做复杂通用规则引擎。
- 做多场景自动泛化。
- 做复杂 3D 建模。
- 做完整权限审批系统。
- 做正式电子签章和文件归档。

## 11. 第一版演示验收脚本

演示时按固定流程执行，保证每次都能稳定复现。

### 11.1 登录

操作：

1. 打开系统。
2. 输入账号密码。
3. 进入工作台。

展示重点：

- 系统是面向一线检修人员的工作台，不是静态网页。

### 11.2 首页输入异常

操作：

1. 在首页输入“站控柜内工控机温度告警，风扇声音异常，前面板风扇转速很低”。
2. 点击发送。

展示重点：

- 对话入口只是开始，后续会转为步骤式诊断。

### 11.3 异常接入

操作：

1. 查看场站图和蓝色故障点。
2. 选择设备型号、位置、角色、关联告警。
3. 修改或确认阈值。
4. 确认接入信息。

展示重点：

- 系统将现场描述整理为结构化诊断输入。
- 场站图用于辅助定位，不做复杂三维建模。

### 11.4 分析诊断

操作：

1. 点击触发诊断。
2. 等待 Agent 逐个运行。
3. 查看诊断结论。

展示重点：

- 展示分析诊断 Agent、操作合规 Agent、知识检索 Agent 的协同过程。
- 结论基于设备、告警、阈值和知识条目生成。

### 11.5 检修向导

操作：

1. 进入检修向导。
2. 逐步点击每个大步骤。
3. 勾选检查项。
4. 查看图片随检查项切换。

展示重点：

- 从文字结论转为工程师可执行的步骤式检修。
- 每一步都有安全提醒、检查项和可视化辅助。

### 11.6 恢复验证

操作：

1. 勾选告警解除、风扇恢复、温度正常、数据上传稳定。
2. 完成检修向导。

展示重点：

- 不是只给建议，而是形成闭环验证。

### 11.7 生成作业卡

操作：

1. 点击生成或导出作业卡。
2. 打开打印视图。
3. 展示本次检修记录的完整汇总。

展示重点：

- 作业卡来自本次流程数据，不是手写文档。
- 第一版先支持打印视图，Word/PDF 后续扩展。

### 11.8 查看检修记录和知识图谱

操作：

1. 进入检修记录页。
2. 点击本次记录详情。
3. 进入知识图谱页查看设备、告警、原因、步骤关系。

展示重点：

- 检修过程可沉淀。
- 知识图谱满足赛题要求并支撑后续知识回流。

## 12. 技术栈边界

当前技术栈保持不变：

- 前端：React + Vite。
- 后端：Python + Flask。
- 数据：SQLite，前期可保留 JSON seed data。
- 大模型：云端 API。
- 部署：不使用 Docker。
- Node.js：建议使用 `v20.19.4`。

后续不建议在第一版中引入过多新依赖。

## 13. 开发阶段规划

### 阶段 1：稳定当前演示骨架

目标：

- 固定主流程。
- 固定页面跳转。
- 固定演示图片和文字。
- 修正明显 UI 问题。

验收：

- 从首页输入到检修记录生成，流程能完整走通。
- 页面不出现明显遮挡、错位、空白断层。

### 阶段 2：接入后端状态和 SQLite

目标：

- 登录状态由后端或本地会话管理。
- 异常接入信息提交到后端。
- 检修步骤勾选状态可保存。
- 关键流程数据写入 SQLite。

验收：

- 刷新页面后关键记录不完全丢失。
- 后端能接收并返回当前流程数据。
- 检修记录页能读取 SQLite 中的记录。

### 阶段 3：接入大模型 API

目标：

- 后端提供 `/api/diagnosis`。
- 后端按固定 prompt 和固定 JSON schema 请求大模型。
- 前端展示结构化诊断结论。

验收：

- 无 API Key 时有降级演示数据。
- API 可用时能生成真实诊断结论。
- 大模型输出异常时不会导致页面崩溃。

### 阶段 4：作业卡打印视图

目标：

- 根据诊断记录、检修记录和检查项状态生成作业卡快照。
- 提供适合打印的作业卡详情页。
- 支持浏览器打印。

验收：

- 作业卡内容来自当前会话数据。
- 打印页面不显示导航、辅助对话和无关按钮。
- 能通过浏览器打印或另存为 PDF。

### 阶段 5：准备龙芯部署

目标：

- 明确依赖包。
- 明确启动命令。
- 明确无 Docker 部署方式。
- 明确云 API 配置方式。

验收：

- 在龙芯虚拟机中可以安装依赖、启动后端、访问页面。
- 前后端主流程可跑通。
