# 正式 MVP 开发计划

更新时间：2026-07-10

需求总纲：`meeting-analysis-and-dev-plan.md`

进度真相：`current-development-status.md`

当前实施 Spec：`r0-dynamic-intake-workbench-spec.md`

## 1. MVP 目标

在龙芯平台稳定演示一条工业设备智能接诊闭环：

```text
工作台简要接诊
  -> Agent 动态补齐现场信息
  -> 生成并确认可编辑检修预方案
  -> 步骤式检修向导
  -> 恢复验证与检修记录
  -> 上传、审核、归档并回流知识图谱
```

系统通过本地审定案例保证离线稳定性；所有模拟能力明确标注边界。

## 2. 冻结技术基线

- 前端：React 18 + Vite 6。
- 后端：Python 3 + Flask 3。
- Node.js：`20.19.4`。
- npm：`10.8.2`。
- 数据：本阶段案例 JSON + React 状态 + 现有 Flask 演示 API。
- 部署验证：龙芯 Node 开发态，逐模块同步验证。
- 不使用 Docker，不在本轮接真实大模型、图数据库或真实多模态识别。

## 3. 开发阶段

### R0 / P0：单页动态接诊工作区（当前进行中）

- 用工作区状态机替换固定四页跳转。
- 首页提供故障图片/视频入口，中央证据画布持续展示用户选择的材料与位置上下文。
- 接诊 Agent 自动流式分析。
- 每张卡片出现前说明信息缺口、参考依据和生成原因。
- 动态生成设备卡、告警卡和接入摘要。
- 支持少量条件分支、逐字段推荐和来源标识。
- 工程师逐项确认并可重新编辑。
- 接入摘要按原始描述、系统整理和诊断任务分层展示。
- 稳定触发现有分析诊断流程。

完成标准以 `r0-dynamic-intake-workbench-spec.md` 第 14 节为准。

### R1 / P0：统一案例数据与详细补充能力

- 定义统一案例 Schema。
- 将现有研华数据迁为 `case-01`。
- 补充品牌、型号、位置、灯态、读数、资料和思考文案字段。
- 页面、Agent 文案和摘要从同一案例数据源读取。

### R2 / P0：可编辑预方案确认

- 方案生成过渡。
- 步骤提纲与检查项展示。
- 步骤和检查项增、删、改。
- 确认结果真实传给现有检修向导。

### R3 / P0：多案例与行业品牌

- 加入罗克韦尔案例和第三品牌案例。
- 三个案例具有不同步骤数量。
- 品牌库覆盖罗克韦尔、浙大中控、霍尼韦尔、施耐德、西门子和研华。
- 二、三案例由人工逐条核验资料、阈值、安全步骤和恢复判据。

### R4 / P0：知识回流闭环

- 上传预置检修案例总结。
- 状态流转：待审核 → 已审核·已归档。
- 图谱采用添加前/添加后两份 JSON。
- 新节点显示 `new` 和近期时间，并支持定位新增知识。

### P1：演示增强

- 图谱扩容到约 25–40 节点并增加属性。
- 专家会诊模拟界面。
- 预制步骤语音。
- 首页辅助区折叠和采集样例。
- pad 多模态演示准备。

## 4. 保留但不提前实施

- SQLite 与真实会话持久化。
- 作业卡正式打印视图。
- 真实大模型 API 与真实 RAG。
- 真实图片、视频、语音识别。
- 真实专家音视频通话。

这些功能不是被取消，而是在最新 Meeting 的演示闭环稳定后再评估。

## 5. 每阶段共同验收

每个 R 模块必须：

1. 对应用户路径完整可操作。
2. 页面无阻断性控制台错误。
3. `npm run build` 在 Node `20.19.4` 下通过。
4. Flask 健康检查通过。
5. 相同代码在龙芯以 Node 开发态启动并完成新路径验证。
6. 结果写入 `current-development-status.md`。

## 6. 当前下一步

只推进 R0：先建立状态机和单页稳定布局，再迁移设备卡、告警卡、摘要卡，最后修复并验证触发诊断。R0 完成前不并行开发真实 API、数据库和 P1 动画功能。
