# 正式 MVP 开发计划与功能边界

## 1. 文档目的

本文档用于明确“站控慧眼：工控设备检修助手”从当前高保真演示骨架进入正式 MVP 开发时的范围、功能边界、实现顺序和验收标准。

正式 MVP 的目标不是一次性做完整产品，而是先做出一个可真实运行、可演示闭环、可在龙芯环境继续部署迭代的最小版本。

## 2. 当前状态判断

当前项目已经具备高保真演示型 MVP 原型基础：

- 已完成首页智能诊断入口。
- 已完成账号密码登录页。
- 已完成异常接入、分析诊断、检修向导、检修记录、知识图谱、设置等主要页面。
- 已完成固定故障场景：油气管道场站内站控柜工控机高温、风扇异常、风道/滤网堵塞相关故障。
- 已完成步骤式检修向导骨架，并接入多张演示图片。
- 已完成辅助对话框的模拟大模型回复效果。
- 已完成分析诊断阶段的 Agent 动态执行感。
- 已完成场站图定位提示和蓝色故障点标记。

因此，下一阶段重点不再是继续堆页面，而是把关键链路接成可运行的最小真实功能。

## 3. 正式 MVP 核心目标

正式 MVP 要跑通一条完整闭环：

1. 用户登录系统。
2. 用户在首页输入现场异常现象。
3. 系统进入异常接入阶段，补充设备型号、位置、角色、告警与阈值信息。
4. 用户触发分析诊断。
5. 后端调用云端大模型 API，生成结构化诊断结论。
6. 系统根据诊断结论进入预设检修向导。
7. 工程师逐步勾选检修检查项。
8. 系统生成检修记录。
9. 检修记录可以本地保存并在记录页查看。

第一版只服务一个主演示场景：站控柜内 ACP-4000 / IPC-610 工控机散热异常。

## 4. 第一版真实功能范围

### 4.1 登录

第一版只做账号和密码登录。

需要实现：

- 登录表单。
- 基础账号密码校验。
- 登录后进入工作台。
- 退出登录。

建议实现方式：

- 前期可以使用后端固定账号。
- 不做复杂权限系统。
- 不做手机号、验证码、厂站选择、身份认证。

### 4.2 首页诊断入口

需要实现：

- 用户输入现场现象。
- 支持使用默认演示问题快速进入流程。
- 提交后进入异常接入阶段。

第一版不需要真实上传图片、语音、视频，但 UI 可以保留入口。

### 4.3 异常接入

需要实现：

- 描述现场现象。
- 补充设备型号。
- 补充设备位置。
- 补充设备角色。
- 选择关联告警。
- 修改灯值和阈值信息。
- 接入信息确认。

实现方式：

- 设备型号、位置、角色、关联告警采用下拉选项。
- 选项第一版可以使用固定演示数据。
- 阈值支持页面内编辑，不要求接入真实采集系统。

### 4.4 分析诊断

需要实现：

- 点击“触发诊断”。
- 展示多 Agent 动态执行过程。
- 调用后端诊断接口。
- 后端调用云端大模型 API。
- 返回结构化诊断结论。
- 前端按结构化字段展示结论。

需要保留的演示效果：

- 分析诊断 Agent、操作合规 Agent、知识检索 Agent 逐个执行。
- 每个 Agent 有“运行中 / 已完成”状态。
- 最终诊断结论使用流式或近似流式输出。

### 4.5 检修向导

需要实现：

- 按步骤展示检修流程。
- 每个步骤包含多个检查项。
- 检查项支持勾选完成。
- 点击右侧步骤可以跳转。
- 中间区域显示对应步骤图片。
- 图片切换保留淡入淡出效果。

第一版检修步骤以当前固定流程为准：

1. 确认告警与定位设备。
2. 安全准备。
3. 检查机柜环境与风道。
4. 检查滤网、风扇和接线。
5. 恢复运行与验证。

### 4.6 检修记录

需要实现：

- 根据诊断结论和检修向导完成状态生成记录。
- 检修记录页以列表形式展示。
- 支持点击记录查看详情。
- 支持按故障关键词搜索。

第一版不强制实现真实导出文件，但可以保留“导出”按钮。

### 4.7 知识图谱

需要保留。

第一版实现目标：

- 展示节点图样式。
- 展示设备、告警、原因、知识条目、检修步骤之间的关系。
- 作为演示和赛题要求体现，不做复杂图数据库推理。

## 5. 第一版模拟能力范围

以下能力第一版保留 UI 和状态标识，但不做真实能力：

- 图片上传识别。
- 视频输入。
- 语音输入。
- 专家连线。
- 真实三维建模。
- 实时设备采集。
- 真实工控通信。
- 真实知识图谱推理。
- 复杂多设备、多故障自动扩展。

这些能力可以在 UI 上体现“入口已预留”或“后续接入”，但不要影响主流程演示。

## 6. 大模型 API 接入边界

第一版只接一个后端诊断接口。

建议接口：

```text
POST /api/diagnosis
```

请求字段建议：

```json
{
  "symptom": "站控柜内工控机温度告警，风扇声音异常，前面板风扇转速很低。",
  "device": {
    "model": "研华 ACP-4000 / IPC-610",
    "location": "某输气场站 · 站控柜 A01",
    "role": "站控工控机"
  },
  "alarms": ["TEMP/FAN 告警", "风扇转速偏低"],
  "thresholds": {
    "fanRpm": "< 500 rpm",
    "systemTemperature": "> 55°C",
    "cpuTemperature": "> 70°C"
  }
}
```

返回字段建议：

```json
{
  "summary": "疑似工控机散热系统异常。",
  "faultType": "散热异常 / 风道堵塞 / 滤网积尘",
  "confidence": "高",
  "evidence": [
    "TEMP/FAN 告警",
    "风扇转速偏低",
    "系统温度和 CPU 温度超过阈值"
  ],
  "possibleCauses": [
    "滤网积尘",
    "风道受阻",
    "风扇低速或异常",
    "机柜环境温度偏高"
  ],
  "recommendedGuide": "ACP4000_IPC610_cooling_fault_repair",
  "safetyNotes": [
    "检修前正常关机并断开所有电源",
    "执行防静电措施",
    "恢复运行后连续观察不少于 10 分钟"
  ]
}
```

前端必须按结构化字段展示，不依赖大模型自由文本排版。

## 7. 数据存储建议

第一版建议使用 SQLite。

原因：

- 部署简单。
- 不依赖 Docker。
- 适合龙芯虚拟机环境。
- 足够支持登录、诊断记录、检修记录和步骤状态保存。

建议最小数据表：

- `users`：用户账号。
- `diagnosis_records`：诊断输入、诊断结论、创建时间。
- `maintenance_records`：检修记录、检修结果、审核状态。
- `maintenance_check_items`：每条记录对应的检查项完成状态。

如果开发时间紧，也可以先使用本地 JSON 文件模拟数据库，但正式 MVP 更建议直接上 SQLite。

## 8. 技术栈边界

当前技术栈保持不变：

- 前端：React + Vite。
- 后端：Python + Flask。
- 数据：SQLite，前期可保留 JSON seed data。
- 大模型：云端 API。
- 部署：不使用 Docker。
- Node.js：建议使用 `v20.19.4`。

后续不建议在第一版中引入过多新依赖。

## 9. 开发阶段规划

### 阶段 1：稳定当前演示骨架

目标：

- 固定主流程。
- 固定页面跳转。
- 固定演示图片和文字。
- 修正明显 UI 问题。

验收：

- 从首页输入到检修记录生成，流程能完整走通。
- 页面不出现明显遮挡、错位、空白断层。

### 阶段 2：接入后端状态

目标：

- 登录状态由后端或本地会话管理。
- 异常接入信息提交到后端。
- 检修步骤勾选状态可保存。

验收：

- 刷新页面后关键记录不完全丢失。
- 后端能接收并返回当前流程数据。

### 阶段 3：接入大模型 API

目标：

- 后端提供 `/api/diagnosis`。
- 后端按固定 prompt 和固定 JSON schema 请求大模型。
- 前端展示结构化诊断结论。

验收：

- 无 API Key 时有降级演示数据。
- API 可用时能生成真实诊断结论。
- 大模型输出异常时不会导致页面崩溃。

### 阶段 4：接入 SQLite

目标：

- 保存诊断记录。
- 保存检修记录。
- 保存检查项完成情况。

验收：

- 检修记录页展示真实保存的数据。
- 点击记录可以查看详情。
- 支持按故障关键词搜索。

### 阶段 5：准备龙芯部署

目标：

- 明确依赖包。
- 明确启动命令。
- 明确无 Docker 部署方式。
- 明确云 API 配置方式。

验收：

- 在龙芯虚拟机中可以安装依赖、启动后端、访问页面。
- 前后端主流程可跑通。

## 10. 验收标准

正式 MVP 完成时，需要满足：

- 可以登录。
- 可以从首页输入异常描述。
- 可以完成异常接入信息补充。
- 可以触发诊断并得到结构化诊断结论。
- 可以进入检修向导。
- 可以逐项勾选检修步骤。
- 可以生成并保存检修记录。
- 可以在检修记录页查看历史记录。
- 可以展示知识图谱页。
- 可以在 WSL 环境稳定启动。
- 可以按文档迁移到龙芯虚拟机部署。

## 11. 第一版明确不做

以下内容不进入正式 MVP 第一版：

- 多设备、多场景完整泛化。
- 真实视频识别。
- 真实语音识别。
- 真实专家连线。
- 真实三维渲染。
- 复杂权限体系。
- 复杂审批流。
- 真实工控协议采集。
- Docker 部署。
- 图数据库。
- 多租户系统。

## 12. 下一步建议

建议立即进入“阶段 1：稳定当前演示骨架”，然后按顺序推进：

1. 检查当前主流程是否还有明显 UI 问题。
2. 固定正式 MVP 的诊断 JSON schema。
3. 增加后端 `/api/diagnosis` 接口。
4. 增加 SQLite 数据库和检修记录保存。
5. 整理龙芯部署启动文档。

从开发风险看，最关键的不是继续扩展页面，而是尽早把“诊断结果生成”和“检修记录保存”接成真实链路。
