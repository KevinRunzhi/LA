# MVP 功能清单与开发优先级：工控设备异常检测与故障分析系统

## 1. 文档目标

本文档用于冻结第一版 MVP 范围，目标是尽快做出一个能跑、能演示、能持续迭代的版本。

第一版不追求完整企业系统，不追求真实泛化能力，不追求复杂三维可视化。第一版只做一个设备、一个故障、一个完整闭环。

MVP 主演示场景：

```text
某输气场站站控柜内研华 ACP-4000 / IPC-610 工控机出现温度告警，
现场判断为风道堵塞 / 滤网积尘 / 散热异常方向，
系统引导一线人员按步骤完成检查、处理和恢复验证。
```

主演示维修依据来自《研华 ACP-4000 / IPC-610 工控机散热异常：滤网堵塞与风扇异常检修流程》。第一版不把 18 个维修动作全部做成页面步骤，而是压缩为 5 个适合演示的主步骤，每个主步骤内部保留关键判断标准、风险提示和来源依据。

关键阈值：

| 项目 | 阈值 / 判断 | 用途 |
|---|---|---|
| 系统风扇转速 | `< 500 rpm` 判定异常 | 判断风扇低速或停转 |
| 系统温度 | `> 55°C` 判定系统温度过高 | 判断机箱散热异常 |
| CPU 温度 | `> 70°C` 判定 CPU 温度过高 | 判断高温风险升级 |
| 环境温度 | `≤ 40°C` 为官方工作温度范围 | 判断机柜环境是否越限 |
| 恢复观察 | `≥ 10 分钟` | 验证温度、告警和数据上传是否稳定 |

MVP 核心闭环：

```text
用户输入或触发预设异常
  -> 系统生成诊断摘要和步骤选项
  -> 用户进入步骤式检修向导
  -> 每一步展示图片占位区、操作说明、安全提醒和完成按钮
  -> 用户完成风道 / 滤网 / 风扇 / 恢复验证步骤
  -> 系统生成检修记录
  -> 专家修正一条经验
  -> 修正内容写入知识库 / 图谱 / 专家经验
  -> 再次触发同类异常时命中专家修正
```

首期深度演示设备：

| 设备 | 范围 |
|---|---|
| 研华 ACP-4000 / IPC-610 工控机整机 | 第一版唯一主演示设备 |

首期主演示故障：

| 设备 | 故障 |
|---|---|
| 研华 ACP-4000 / IPC-610 工控机整机 | 高温告警 / 风道堵塞 / 散热异常 |

暂不进入第一版主演示：

1. PLC 控制柜完整诊断。
2. 通信交换机、传感器、现场仪表、电动执行机构。
3. 供电异常、硬盘异常、系统死机等其他工控机故障。
4. 真实视频识别、真实语音播报、真实连线专家。

## 2. 开发优先级定义

### P0：必须完成

P0 是第一版演示闭环必需功能。缺少任何一项，MVP 就不完整。

### P1：建议完成

P1 能增强演示可信度，但如果时间紧，可以静态化或延后。

### P2：后期增强

P2 不影响第一版演示。只在 P0 稳定后考虑。

## 3. P0 必做功能

### 3.1 龙芯空壳运行验证

目标：

- 尽早解决龙芯兼容性问题。
- 不等所有功能做完再上龙芯。

功能：

1. 前端空壳页面能在龙芯上通过 Node/Vite 开发态启动。
2. Flask Hello API 能在龙芯上启动。
3. 前端能请求后端 Hello API。
4. 能展示一个空的工作台页面。
5. 记录 Node、npm、Python、Flask 的版本和启动命令。

验收：

```text
龙芯本机打开前端页面
  -> 页面显示系统名称
  -> 点击测试按钮
  -> 后端返回 hello
```

不做：

- 平板访问龙芯 IP。
- Nginx 正式部署。
- Docker。
- 云 API 龙芯实时联调。

---

### 3.2 基础页面骨架

目标：

- 先做出可演示的页面结构。
- 为后续功能填充留入口。

页面：

1. 工作台。
2. 异常检测入口。
3. 步骤式检修向导。
4. 检修记录页面。
5. 专家审核与知识回流页面。

功能：

1. 左侧或顶部导航。
2. 当前用户信息，可先写死。
3. 当前主演示设备提示。
4. 当前主演示故障提示。
5. 一键进入主演示流程。

不做：

- 完整后台管理。
- 复杂权限。
- 多角色菜单控制。

---

### 3.3 异常检测入口

目标：

- 用户用一句话或点击预设异常进入分析阶段。
- 对话只作为入口，不作为主体。

功能：

1. 输入现场现象：

```text
站控柜内工控机温度告警，风扇声音异常，怀疑散热不良。
```

2. 提供“一键触发主演示异常”按钮。
3. 系统生成诊断摘要：
   - 设备：研华 ACP-4000 / IPC-610 工控机。
   - 故障：高温告警 / 风道堵塞 / 散热异常。
   - 风险：可能影响数据上传稳定性和监控告警可靠性。
4. 展示步骤选项：
   - 定位异常设备。
   - 查看故障分析。
   - 进入检修步骤。
   - 提交专家审核。

实现要求：

1. 第一版可以完全使用预设数据。
2. 可保留调用云 API 的接口，但必须有缓存兜底。
3. 大模型返回失败不能阻断演示。

---

### 3.4 步骤式检修向导

目标：

- 用步骤流替代大段文字回答。
- 用户按步骤执行，每一步都有图片区块占位、说明、安全提醒和完成按钮。
- MVP 阶段不实现图片插入、图片上传、图片管理或真实图片展示功能；只在页面结构和数据结构中预留图片位置，方便后续补充素材。

主演示步骤采用“5 个主步骤 + 步骤内检查项”的方式呈现：

| 步骤 | 名称 | 图片占位 | 检查项 | 完成条件 |
|---|---|---|---|---|
| 1 | 确认告警与定位设备 | 预留 `step-01-location` 占位区 | TEMP/FAN 灯状态、蜂鸣、风扇 rpm、系统/CPU 温度、站控柜位置 | 确认故障设备为目标工控机，并记录告警形态 |
| 2 | 执行安全准备 | 预留 `step-02-safety` 占位区 | 通知值班负责人、正常关机、拔除所有电源线、防静电、拍照记录线缆 | 安全项全部勾选后才能进入拆检步骤 |
| 3 | 检查机柜环境与风道 | 预留 `step-03-airflow` 占位区 | 环境温度是否 ≤40°C、进风口/出风口是否遮挡、机箱开孔是否被覆盖 | 确认风道是否存在外部堵塞或环境越限 |
| 4 | 检查滤网、风扇和接线 | 预留 `step-04-filter-fan` 占位区 | 门滤网、风扇滤网积尘，风扇异响/停转/低速，FAN1/FAN2 接线顺序 | 完成清理或更换判断，并确认接线顺序无误 |
| 5 | 恢复运行与验证 | 预留 `step-05-verify` 占位区 | 滤网装回、前门锁闭、开机、TEMP/FAN 状态、风扇 >500 rpm、系统 ≤55°C、CPU ≤70°C、数据上传稳定 | 连续观察 ≥10 分钟，无再告警、无重启、数据上传正常 |

功能：

1. 步骤列表。
2. 当前步骤详情。
3. 当前步骤图片占位区。
4. 操作说明。
5. 安全提醒。
6. 来源依据简要展示。
7. 上一步 / 下一步。
8. 标记完成。
9. 步骤完成状态。

步骤内检查项示例：

```text
步骤 1：确认告警与定位设备
- 记录 TEMP LED：常亮 / 闪烁 / 正常
- 记录 FAN LED：常亮 / 闪烁 / 正常
- 记录蜂鸣器：持续蜂鸣 / 已静音 / 无
- 读取风扇转速：是否 < 500 rpm
- 读取系统温度：是否 > 55°C
- 读取 CPU 温度：是否 > 70°C
- 确认设备位置：输气场站 / 站控柜 / 工控机
```

```text
步骤 2：执行安全准备
- 通知值班负责人或调控中心
- 正常关闭操作系统
- 拔除所有电源线
- 等待设备冷却
- 佩戴接地防静电手环
- 禁止带电拆装任何部件
- 拍照记录风扇线缆位置和 FAN1/FAN2 顺序
```

实现要求：

1. MVP 只做图片占位区，不做图片插入、上传、替换、裁剪、预览和管理。
2. 数据结构预留 `asset_key`、`asset_url` 或 `placeholder_key` 字段，第一版可以为空或使用占位文案。
3. 页面在对应位置显示固定尺寸图片区块，占位文案为“图片待补充”或“后续添加步骤示意图”。
4. 不做热点识别。
5. 不做真实三维。
6. 不做可旋转模型。

视觉风格：

1. 整体参考“站控慧眼 / 智能诊断台”示意图。
2. 使用浅色工作台背景、白色卡片、细边框、轻阴影和蓝紫色强调按钮。
3. 页面布局优先采用“左侧窄导航 + 中间诊断/向导主区域 + 右侧诊断流程与快捷操作”的结构。
4. 主色建议使用蓝紫渐变或高亮蓝，辅助色使用浅灰蓝、淡边框和深色正文。
5. 不做深色大屏风格，不做复杂工业三维看板。

### 3.5 检修记录生成

目标：

- 向导完成后形成一次检修记录。
- 作业卡不作为核心演示，只作为记录导出结果。

功能：

1. 汇总用户执行过的步骤。
2. 汇总每一步完成状态。
3. 汇总诊断摘要、风险、影响范围、安全检查结果。
4. 展示检修完成记录。
5. 支持打开打印版页面。
6. 支持浏览器另存为 PDF。

不做：

- 后端 PDF 生成。
- Word 导出。
- 复杂排版。

---

### 3.6 轻量 RAG 证据与知识来源

目标：

- 让关键诊断和步骤有依据。
- 不做复杂文档解析。

功能：

1. 预置 8-12 条知识条目。
2. 针对高温、风道、滤网、风扇、安全要求返回固定证据卡片。
3. 每条证据展示：
   - 来源名称。
   - 章节或条目名。
   - 简短引用。
   - 关联步骤。

实现要求：

1. 第一版可使用 SQLite 关键词匹配或直接按步骤绑定证据。
2. 不接 DashVector。
3. 不做 PDF 导入。
4. 不做扫描件 OCR。

第一版建议预置知识条目：

| 编号 | 标题 | 绑定步骤 |
|---|---|---|
| KB-001 | ACP-4000 / IPC-610 散热系统结构 | 步骤 1、3、4 |
| KB-002 | TEMP/FAN LED 与蜂鸣告警含义 | 步骤 1 |
| KB-003 | 风扇 `<500 rpm` 告警阈值 | 步骤 1、5 |
| KB-004 | 系统温度 `>55°C` 与 CPU `>70°C` 告警阈值 | 步骤 1、5 |
| KB-005 | 环境工作温度 `0~40°C` | 步骤 3 |
| KB-006 | 断电、防静电和禁止带电拆装要求 | 步骤 2 |
| KB-007 | 门滤网与风扇滤网检查清理方法 | 步骤 4 |
| KB-008 | 风扇更换与 FAN1/FAN2 接线顺序 | 步骤 4 |
| KB-009 | 恢复运行确认清单 | 步骤 5 |
| KB-010 | 异常升级条件 | 步骤 5 |

### 3.7 知识图谱子图

目标：

- 保留赛题要求中的知识图谱能力。
- 展示当前故障相关关系，不追求大规模图谱。

功能：

1. 展示当前故障相关子图。
2. 包含设备、部件、故障、原因、检查方法、处理措施、安全要求、影响范围。
3. 专家修正后可新增或更新 1 条关系。

主演示子图：

```text
某输气场站 -> 包含 -> 站控柜
站控柜 -> 包含 -> 工控机
工控机 -> 包含 -> 风道
工控机 -> 包含 -> 滤网
工控机 -> 包含 -> 风扇
工控机 -> 包含 -> TEMP/FAN 告警指示
工控机 -> 发生 -> 高温告警
高温告警 -> 可能原因 -> 风道堵塞
高温告警 -> 可能原因 -> 滤网积尘
高温告警 -> 可能原因 -> 风扇异常
风扇异常 -> 判断依据 -> 风扇转速 <500 rpm
高温告警 -> 判断依据 -> 系统温度 >55°C
高温告警 -> 判断依据 -> CPU 温度 >70°C
风扇异常 -> 检查方法 -> 核对 FAN1/FAN2 接线顺序
高温告警 -> 影响 -> 数据上传稳定性
高温告警 -> 影响 -> 监控告警可靠性
处理措施 -> 前置安全要求 -> 静电防护
处理措施 -> 前置安全要求 -> 禁止带电拆卸非热插拔部件
恢复验证 -> 要求 -> 连续观察 ≥10 分钟
```

技术：

1. SQLite 存 `kg_entities` 和 `kg_relations`。
2. 前端用 ECharts graph 或简单 SVG 展示。
3. 不使用 Neo4j。
4. 不做图算法。

### 3.8 专家审核与回流

目标：

- 展示系统“越用越准”。
- 只做一条专家修正闭环。

功能：

1. 待审核列表展示当前检修记录。
2. 专家打开记录，查看：
   - 用户输入。
   - 诊断摘要。
   - 步骤执行结果。
   - 证据卡片。
   - 知识图谱关系。
3. 专家填写修正文本：

```text
沙尘环境下，站控柜工控机滤网维护周期应从季度检查缩短为月度检查。
恢复运行后需观察数据上传稳定不少于 10 分钟。
```

专家回流也可以补充官方容易遗漏但现场很关键的注意事项：

```text
更换或拆装风扇后，必须核对风扇线接入告警板的 FAN1/FAN2 顺序；
接线顺序错误会导致报警功能异常，出现误报或漏报。
```

4. 专家选择回流目标：
   - 写入知识库。
   - 更新图谱关系。
   - 写入专家经验。
5. 审核通过后，再次触发同类异常时展示：

```text
专家修正 · 已审核
```

不做：

- 两步复杂审核。
- 多字段结构化编辑。
- 驳回/退回补充的复杂状态。

### 3.9 演示兜底

目标：

- 不让云 API、网络、图片未补充影响演示。

功能：

1. 预置诊断摘要。
2. 预置步骤选项。
3. 预置 Agent 会诊结果。
4. 预置证据卡片。
5. 预置知识图谱。
6. 预置专家修正演示数据。
7. 一键重置演示数据。
8. 预置维修指导中的阈值、步骤、证据和专家修正内容。

## 4. P1 建议完成功能

### 4.1 基础登录

建议实现：

1. 一线人员账号。
2. 专家账号。
3. 登录和退出。

可简化：

1. 第一版可以先写死当前用户。
2. 专家审核页面可通过路由直接进入。

### 4.2 多 Agent 会诊过程动画

建议实现：

1. 接诊/检索 Agent 卡片。
2. 分析诊断 Agent 卡片。
3. 操作合规 Agent 卡片。
4. 卡片依次亮起，模拟分析过程。

可简化：

1. 使用预置结果。
2. 不真实并发调用多个 Agent。

### 4.3 连线专家模拟入口

建议实现：

1. “连线专家”按钮。
2. 模拟专家在线状态。
3. 模拟弹窗。
4. 标注“预留 IM/RTC 接口”。

### 4.4 语音播报模拟入口

建议实现：

1. 每个步骤显示语音图标。
2. 可预留 `audio_path`。
3. 如果有时间，播放预生成音频。

### 4.5 视频输入模拟入口

建议实现：

1. 页面显示视频上传入口。
2. 不做真实识别。
3. 标注“视频识别能力预留”。

2. 本地视觉模型训练。

4. Docker。
5. 扫描件 OCR。
6. 后端 PDF 生成。
7. Word 导出。
8. 真实三维建模。
9. 可旋转模型。
10. 自动热点识别。
11. 真实视频识别。
12. 真实连线专家。
13. 完整管理员后台。

## 8. 演示主流程

主演示场景：

```text
站控柜内研华 ACP-4000 / IPC-610 工控机温度告警，
风扇声音异常，怀疑风道堵塞或滤网积尘导致散热异常。
```

演示步骤：

1. 打开系统工作台。
2. 点击“开始异常检测”。
3. 输入现场现象，或点击“一键触发主演示异常”。
4. 系统生成诊断摘要和步骤选项。
5. 展示多 Agent 会诊结果：
   - 分析诊断 Agent：判断优先检查风道堵塞、滤网积尘和风扇异常。
   - 操作合规 Agent：提示断电、防静电、禁止带电拆装。
   - 知识检索 Agent：展示 TEMP/FAN 告警阈值、风扇转速阈值和安全要求来源。
6. 点击“进入检修向导”。
7. 第 1 步：显示异常设备定位占位区，记录 TEMP/FAN 灯、蜂鸣、风扇转速和温度读数。
8. 第 2 步：显示安全确认占位区，勾选通知负责人、关机、拔除电源、防静电、禁止带电拆装。
9. 第 3 步：显示风道检查占位区，确认环境温度、进风口/出风口和机箱开孔情况。
10. 第 4 步：显示滤网和风扇检查占位区，确认滤网积尘、风扇异响/低速/停转、FAN1/FAN2 接线顺序。
11. 第 5 步：显示恢复验证占位区，确认风扇 >500 rpm、系统 ≤55°C、CPU ≤70°C、告警解除、数据上传稳定 ≥10 分钟。
12. 生成检修记录。
13. 切换到专家审核页面。
14. 专家补充“沙尘环境滤网维护周期缩短”和“恢复后观察数据上传稳定不少于 10 分钟”等经验。
15. 审核通过并写入知识库、图谱和专家经验。
16. 再次触发同类异常。
17. 系统展示“专家修正 · 已审核”内容。

## 9. 开发顺序建议

推荐按“能跑起来优先”开发：

1. 龙芯空壳验证：前端 Hello + Flask Hello。
2. 页面骨架：工作台、异常检测入口、向导页。
3. 演示种子数据：步骤流、图片占位字段、证据、图谱。
4. 步骤式检修向导：步骤切换、图片区块占位、完成状态。
5. 检修记录生成。
6. 专家审核与回流。
7. 再次触发命中专家修正。
8. 知识图谱子图展示。
9. 多 Agent 会诊卡片和过程动画。
10. 打印页导出。
11. 一键重置演示数据。

种子数据优先级：

1. 5 个向导步骤。
2. 5 个图片占位 key 或预留素材路径字段。
3. 10 条知识条目。
4. 1 组图谱实体和关系。
5. 1 条专家修正前后对比数据。
6. 1 组 Agent 会诊缓存。

## 10. 验收标准

MVP 完成标准：

1. WSL 环境能完整跑通主演示流程。
2. 龙芯环境能启动前端 dev server 和 Flask 后端。
3. 工作台能进入异常检测入口。
4. 系统能生成诊断摘要和步骤选项。
5. 系统能进入步骤式检修向导。
6. 每个步骤能显示固定尺寸图片占位区，并预留后续素材字段。
7. 用户能逐步完成 5 个步骤。
8. 步骤 1 能展示或记录风扇转速、系统温度、CPU 温度等关键判断项。
9. 步骤 2 能展示断电、防静电、禁止带电拆装等强制安全项。
10. 步骤 5 能展示恢复验证阈值：风扇 >500 rpm、系统 ≤55°C、CPU ≤70°C、观察 ≥10 分钟。
11. 系统能生成检修记录。
12. 系统能展示轻量证据卡片。
13. 系统能展示当前故障知识图谱子图。
14. 专家能填写一条修正并审核通过。
15. 修正内容能写入知识库 / 图谱 / 专家经验。
16. 再次触发同类异常时能展示“专家修正 · 已审核”。
17. 云 API 不可用时，主演示流程仍可通过缓存走通。
18. 全流程保持只读不控，不接入真实生产控制网络。
