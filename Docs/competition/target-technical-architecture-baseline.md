## 1. 编写目的

本文档用于确定系统最终交付版本的目标技术架构，为软件功能设计、产品实现、功能测试和安装部署提供统一技术依据。

本文档描述的是系统面向完整业务能力的目标架构，不以某一阶段的页面原型或临时实现方式作为技术选型依据。后续各项功能设计应遵循本基线；尚未最终确定的组件，须经过兼容性验证和技术评审后再固定版本。

## 2. 架构建设目标

目标架构需要满足以下要求：

- 支撑工控设备异常接入、辅助诊断、标准化检修、恢复验证、专家审核和知识回流的完整业务闭环；
- 支撑键盘与语音输入，以及文本、图片、音频、视频等多模态资料的统一接入与处理；
- 支撑设备手册、检修案例、检修记录、作业卡和知识图谱的统一知识管理；
- 支撑基于检修场景、设备信息和当前步骤的智能问答；
- 支撑知识的来源追溯、适用范围标识、审核发布和版本管理；
- 支撑龙芯处理器与银河麒麟操作系统环境下的部署和运行；
- 支撑后续设备类型、故障领域、知识类型和模型能力的持续扩展；
- 控制系统部署复杂度，满足离线或受限网络环境的安装交付要求。

## 3. 总体架构原则

### 3.1 B/S访问模式

系统采用浏览器/服务器模式。工程师、专家和用户管理员通过浏览器访问系统，无需在用户终端单独安装业务客户端。

### 3.2 前后端分离

前端负责业务页面、交互状态、图谱可视化和操作反馈；后端通过统一接口提供用户权限、检修业务、知识管理、智能检索和文件处理能力。

### 3.3 模块化单体为主

核心业务后端采用模块化单体架构。用户权限、异常事件、检修任务、检修向导、专家审核、检修记录、作业卡和知识管理等模块在统一应用内组织，并通过清晰的领域边界解耦。

该方案在保证模块可维护性的同时，减少大量微服务带来的注册发现、分布式事务、链路治理和离线部署成本。

### 3.4 智能能力独立组织

文档解析、OCR、语音转写、图像理解、向量化、混合检索、结果重排序和大模型调用等能力作为独立智能能力层组织。智能能力可以与核心业务部署在同一节点，也可以根据算力条件部署在独立节点。

### 3.5 业务数据与知识数据分工存储

- PostgreSQL负责用户、设备、异常事件、检修任务、检修步骤、审核记录、作业卡和知识元数据等结构化业务数据；
- Neo4j负责设备、型号、部件、故障、现象、原因、措施、案例和资料之间的图关系；
- pgvector负责知识片段、案例片段和检修记录片段的向量表示及相似度检索；
- 文件存储负责原始手册、图片、音频、视频、附件和导出的PDF作业卡。

### 3.6 可替换的模型接入方式

系统通过统一模型网关调用大语言模型、视觉模型、向量模型和重排序模型。业务模块不直接绑定某一模型厂商或某一模型版本，以便根据最终运行环境和算力条件更换模型实现。

## 6. Python核心业务后端基线

### 6.1 技术定位

核心业务后端确定采用Python实现。Python不仅用于模型调用，也负责系统的主要业务接口、业务规则和数据协调。

核心业务框架确定采用FastAPI，并以模块化单体方式组织。主要原因如下：

- 适合构建前后端分离的REST API；
- 基于类型标注和Pydantic完成输入输出校验；
- 支持异步请求和流式响应，适合智能问答和任务进度推送；
- 可以直接复用Python的文档解析、知识检索和模型调用生态；
- 能够通过模块划分、依赖注入、事务管理和测试体系承载完整业务系统。

Django、Django REST Framework和Flask不作为核心业务框架。后续确有独立工具需要时可以单独评估，但不得在核心业务服务中并行引入第二套Web框架。

### 6.2 业务模块划分

Python核心业务服务初步划分为以下模块：

- 身份认证与角色权限模块；
- 用户与组织管理模块；
- 工程师工作台模块；
- 异常事件接入模块；
- 设备与故障信息模块；
- 检修资料选择模块；
- 智能诊断与检索编排模块；
- 检修预方案模块；
- 标准化检修向导模块；
- 恢复运行验证模块；
- 检修记录与作业卡模块；
- 专家审核模块；
- 检修知识库模块；
- 知识图谱管理模块；
- 知识回流与发布模块；
- 系统设置与运行管理模块。

### 6.3 后端分层

核心业务后端采用清晰的内部层次：

```text
API接口层
    ↓
应用服务层
    ↓
领域规则层
    ↓
数据访问与外部服务适配层
```

- API接口层负责请求接收、权限校验和响应组织；
- 应用服务层负责业务流程编排和事务边界；
- 领域规则层负责检修任务、步骤状态、审核和知识发布规则；
- 数据访问与适配层负责PostgreSQL、Neo4j、Redis、文件存储和模型网关访问。

### 6.4 核心组件职责

| 组件 | 确定职责 |
| --- | --- |
| FastAPI | REST接口、依赖注入、权限入口、流式响应和接口文档 |
| Pydantic | 请求模型、响应模型、配置模型和数据校验 |
| SQLAlchemy 2.x | PostgreSQL数据访问、事务和持久化映射 |
| Alembic | PostgreSQL数据库结构迁移与版本管理 |
| Neo4j Python Driver | Bolt连接、Cypher执行和图数据库事务 |
| Celery | 文档解析、OCR、向量化、知识抽取和多媒体处理任务 |
| Redis | Celery消息代理、任务状态、短期缓存和必要的会话辅助数据 |

核心请求链路与耗时任务采用不同处理方式：普通业务请求由FastAPI同步完成并在单次数据库事务内结束；文档解析、向量化和多媒体处理等耗时操作提交到Celery执行，前端通过任务状态接口获取进度和结果。

### 6.5 用户管理与权限基线

系统采用工程师、专家和用户管理员三类固定角色。角色用于确定用户可进入的工作区和可调用的业务接口，不建设复杂的动态权限配置平台。

- 工程师账号关联所属场站和班组，进入工程师工作台并执行现场检修业务；
- 专家账号关联所属机构和专业方向，进入专家工作台并执行案例审核和知识维护；
- 用户管理员进入独立用户管理后台，维护工程师和专家账号，不获得检修、审核、知识发布和图谱编辑权限；
- 用户记录至少包含用户编号、登录账号、姓名、角色、组织归属、账号状态、创建时间和更新时间；
- 登录账号在系统内唯一，停用账号不得建立新的登录会话；
- 密码只保存安全散列值，密码重置生成临时凭据或进入重新设置流程，不在普通日志中记录明文；
- 新增、编辑、角色调整、启停和密码重置操作写入审计记录；
- 后端在接口层完成身份校验，在应用服务层再次校验关键业务权限，不能只依赖前端菜单隐藏实现隔离。

## 7. Neo4j知识图谱基线

### 7.1 技术定位

知识图谱确定采用Neo4j作为图数据库，使用Cypher完成图数据创建、更新、查询和路径分析，使用Neo4j官方Python驱动与核心业务后端连接。

Neo4j主要承载以下节点：

- 设备类别；
- 设备型号；
- 设备部件；
- 故障领域；
- 故障类型；
- 故障现象；
- 故障原因；
- 检查方法；
- 处置措施；
- 检修步骤；
- 工具与备件；
- 安全注意事项；
- 检修案例；
- 技术手册；
- 告警与监测指标。

主要关系包括：

- 设备包含部件；
- 型号属于设备类别；
- 部件可能发生故障；
- 故障表现为现象；
- 现象可能由原因导致；
- 原因通过方法确认；
- 故障采用措施处置；
- 检修步骤需要工具或备件；
- 案例涉及设备、故障和措施；
- 手册适用于设备或型号；
- 告警指向现象或故障；
- 知识条目来源于手册、案例或检修记录。

### 7.2 PostgreSQL与Neo4j的职责边界

PostgreSQL与Neo4j不重复承担同一类主数据职责：

- PostgreSQL保存用户、任务、状态、审核、文件索引和知识发布记录；
- Neo4j保存用于知识关联、路径检索和图谱展示的节点及关系；
- PostgreSQL中的知识条目使用稳定业务标识与Neo4j节点关联；
- 图谱新增、修改和删除必须形成变更记录并支持审核追溯；
- 跨数据库更新通过应用服务和幂等同步机制保证可恢复性。

### 7.3 龙芯环境兼容性风险

截至本基线编制时，Neo4j官方系统要求明确列出的处理器架构为x86_64和ARM，未明确列出LoongArch64。因此，“采用Neo4j”作为知识图谱产品选型已经确定，但其在龙芯、银河麒麟环境下的具体部署方式尚未最终确定。

后续必须完成以下专项验证：

1. 验证Neo4j指定版本在LoongArch64 OpenJDK上的启动和运行情况；
2. 验证Neo4j依赖的本地库、文件锁、内存映射和JNA能力；
3. 验证Bolt协议连接、Cypher查询、事务和备份恢复；
4. 验证典型图谱规模下的布局数据加载和查询性能；
5. 根据验证结果确定原生部署、独立兼容节点部署或其他适配方式。

在专项验证完成前，部署文档不得将Neo4j在LoongArch64上的原生运行表述为已经确认的技术事实。

## 8. 统一知识与智能检索基线

### 8.1 知识处理流程

```text
资料上传
  ↓
文件识别与安全校验
  ↓
文本、表格、图片和音频内容提取
  ↓
内容清洗与语义分段
  ↓
设备、型号、部件和故障标签提取
  ↓
向量生成与索引建立
  ↓
知识条目和图谱关系生成
  ↓
人工审核与正式发布
```

### 8.2 混合检索流程

检索服务采用以下能力组合：

- 设备、型号、部件和故障领域条件过滤；
- 关键词检索；
- 调用千问text-embedding-v4 API生成1024维文本向量；
- 使用pgvector进行语义向量检索；
- Neo4j图关系扩展检索；
- 多路召回结果融合；
- 调用千问qwen3-rerank API进行精排；
- 调用千问qwen3.7-plus API基于检索证据组织回答；
- 返回手册、案例、检修记录和图谱节点等引用来源。

### 8.3 检修智能问答上下文

检修智能体回答时应组合以下上下文：

- 当前用户与角色；
- 当前设备类别、型号和部件；
- 当前异常现象和监测数据；
- 当前检修任务及所处步骤；
- 已选择的检修资料；
- 检索到的手册、案例和图谱路径；
- 已完成的操作及现场补充信息；
- 安全约束和适用范围。

### 8.4 模型接入与RAG编排

模型能力由后端统一模型网关封装，上层业务模块不直接依赖具体厂商接口。模型网关以OpenAI Python SDK调用兼容接口，并以DashScope Python SDK补充阿里云百炼原生能力，统一处理模型标识、输入输出格式、调用凭据和异常信息。

RAG流程使用LangChain Core组织提示模板、检索链、模型调用和结构化输出解析。设备、故障、检修步骤、审核发布等核心业务规则仍由FastAPI应用服务实现，不写入框架链路，避免业务逻辑与模型编排组件耦合。

模型及外部服务的HTTP访问统一使用httpx，结合Tenacity实现限定次数的失败重试。超时、重试次数、等待间隔和可重试错误类型通过配置集中管理；鉴权失败、参数错误等不可恢复错误不进行盲目重试。

应用配置使用pydantic-settings从环境变量和部署环境配置中加载。数据库密码、JWT密钥和模型API密钥不得进入源代码或普通日志。Python标准logging作为日志基础设施，structlog输出结构化日志，并使用请求标识、任务标识和用户标识关联同一次业务处理过程。

### 8.5 语音输入处理流程

语音输入用于快速形成异常描述或检修智能体追问，与作为证据长期保存的音频材料分开处理。其基本流程如下：

```text
用户主动启动语音输入
  ↓
浏览器请求麦克风权限并采集短时音频
  ↓
用户停止录音或达到单次时长限制
  ↓
前端提交临时音频至语音识别接口
  ↓
千问qwen3-asr-flash生成识别文本
  ↓
识别文本回填到当前输入区
  ↓
用户修改、取消或确认提交
```

首页语音输入结果回填到现场异常描述，检修智能体语音输入结果回填到当前步骤追问框。系统应显示待机、录音、识别中、识别完成和识别失败状态，并防止重复启动录音或重复提交识别任务。麦克风拒绝授权、设备不可用、录音为空、网络超时或识别失败时保留原有文字内容，并允许用户继续键盘输入。

语音输入产生的音频属于临时处理数据，识别完成、取消或失败后按照临时文件策略清理。只有用户明确将录音作为现场音频材料接入时，系统才为其建立文件记录并纳入检修任务生命周期。

## 9. 目标部署形态

系统优先采用适合离线交付的服务化部署方式：

```text
银河麒麟 / LoongArch64节点
├── Nginx与前端静态资源
├── Python核心业务服务
├── 智能检索与多模态处理服务
├── PostgreSQL与pgvector
├── Redis
├── 文件资源库
├── Prometheus与Grafana
└── systemd服务配置

图数据库节点
└── Neo4j（部署位置由LoongArch64专项验证结果确定）

模型推理节点
└── 语言、视觉、向量、重排序及语音模型
```

模型推理节点可以与业务节点合并，也可以根据CPU、内存和加速设备条件独立部署。模型服务通过统一接口接入，不改变上层检修业务流程。

软件以LoongArch64离线安装包作为正式交付载体。安装包包含前端静态资源、Python运行环境及离线依赖、应用服务、数据库初始化与升级脚本、systemd服务配置、默认配置模板和必要字体资源。安装过程应支持环境检查、组件初始化、服务启停、版本升级和安全卸载，并避免依赖安装期间访问外部软件源。

Prometheus负责采集服务可用性、接口耗时、异步任务、数据库连接和主机资源等指标，Grafana提供运行状态看板。监控组件及其依赖仍须在目标龙芯、银河麒麟环境完成安装和运行验证。

## 10. 安全与可追溯基线

- 使用固定角色访问控制区分工程师、专家和用户管理员权限；
- 用户管理后台与工程师端、专家端相互隔离，管理员不继承专家的专业内容权限；
- 用户新增、编辑、角色调整、启停和密码重置形成操作审计；
- 对用户身份、接口访问和文件访问进行认证与授权；
- 对检修任务、关键步骤、恢复验证和专家审核进行操作留痕；
- 对知识条目和图谱节点的新增、修改、审核和发布进行版本记录；
- 对模型生成内容保留引用来源、生成时间和业务上下文；
- 对上传文件实施类型、大小、访问权限和恶意内容检查；
- 麦克风仅在用户主动操作并授权后启用，语音输入结束或取消后及时停止采集并释放媒体资源；
- 短时语音输入的临时音频不作为任务附件长期保存，识别过程及清理结果应可追踪；
- 密码、数据库凭据和模型访问凭据不得明文写入代码或安装包；
- PostgreSQL使用pg_dump形成可恢复备份，Neo4j使用与其版本和授权匹配的官方备份工具，文件资源库执行增量备份；
- 数据库、图数据库和文件资源库的备份应使用同一备份批次标识，并定期执行恢复验证。
