## 1. 会议内容概述

本次会议围绕当前 MVP 演示走查展开，覆盖八个话题：语音播报实现方式、专家电话会诊的演示形态、语音输入的设备限制与演示策略、首页信息密度、**接诊到检修向导之间的流程重构（本次会议核心，占篇幅最大）**、案例库的数据组织方式、工控机品牌型号的业务真实性、赛题"检修案例上传审核入图谱"功能的补齐方式。会议还透露了重要项目背景：本项目源自国家管网集团实际项目（山东德州齐鹅站现场调研），月底可能纳入中心汇报向管网领导展示。

## 2. 会议核心结论

1. **项目方向没有变化**，老师对整体骨架继续认可，本次全部是"在现有骨架上加深"的意见，无推倒性否定。
2. **最主要的问题被重新定位**：老师认为当前最大缺口不在检修向导本身（"后续步骤但凡有点行业知识都能想到"），而在**"前面怎么得到这个方案"的过程**——从简要接诊到详细信息补充、再到方案生成，整个过程要"一边思考一边动态生成"，并且方案要经过用户确认和编辑。这是对此前"体现 Agent 参与过程"意见的具体化和升级。
3. **老师希望的最终呈现**：接诊像真实的智能系统在工作（动态生成、逐块弹出、照片定位、可编辑预方案），业务信息像真实行业（罗克韦尔等主流品牌型号可选），闭环覆盖赛题全部基本功能（含案例上传审核入图谱）。
4. **下一阶段最重要目标**：重构"接诊 → 详细信息补充 → 预方案确认 → 检修向导"的中段流程 + 用 JSON 案例库支撑 3 个案例 + 补齐知识回流演示。
5. **新增外部动力**：月底中心汇报可能向国家管网领导展示，演示质量要求实质性提高，时间点比初赛更近。

## 3. 老师最关注的事项（按重要程度排序）

1. **"怎么得到方案"的过程可见性**（篇幅最长、反复解释、亲自打开夸克演示）——简要输入→思考→动态补充→思考→预方案→确认，每个环节都要有生成感和交互感。
2. **业务信息的行业真实性**——"它没有体现这些实际业务的信息……工控机品牌型号这就是实际业务信息"；油气行业实际用罗克韦尔多，浙大中控、霍尼韦尔次之，施耐德较少，研华在这个行业用得少。
3. **赛题基本功能的完整覆盖**——学生主动提出赛题第 4 点（一线人员上传检修案例→审核→纳入知识图谱）未做，老师当场给出演示级实现方案。
4. **演示的层次与可信度**——首页信息密度要收敛（四周可折叠）、图谱数据要加多"不会显得有问题"、龙芯与 pad 分工演示、步骤数要显得动态（3 步/5 步不同案例）。
5. **对外汇报机会**——月底管网中心汇报，作为"智能体探索"案例。

## 4. 老师明确提出的要求

**流程与交互**：
- 首页"开始接诊"与后续步骤之间必须插入一个**分析/思考过程**，不能直接跳页。
- 详细信息补充要**动态逐块生成**：思考过程中缺什么信息就在界面上弹出什么，用户逐个点击确认/补充；界面不必九宫格规整，可按需用下拉框、列表等。
- 详细信息确认页：**上一步拍的现场照片放中间**，系统根据照片匹配出位置/布局图，需补充的信息在四周逐步生成。
- 详细信息补齐后再有一个思考过程（生成检修方案），可做成弹窗或悬浮窗过渡。
- **新增"预方案确认"主界面**：像 AI 写作生成提纲那样输出检修步骤提纲，每个步骤及其检查项可**增加、修改、删除**，用户确认后才进入检修向导。
- 检修步骤数要体现动态性：不同案例生成不同步数（老师举例：龙芯演示 3 步的，pad 演示 4-5 步更复杂的）。
- 首页四周辅助区块改为**默认隐藏、点击展开**；中间的现象描述 + 故障采集入口（图片/录音/视频/资料勾选）保持平铺。
- 检修资料勾选可加**快捷按钮**按类别全选（举例："工控机异常""储气罐异常"）。

**数据与业务**：
- 设计**小案例库：约 3 个案例，每个案例 = 一个 JSON 文件**，列出每个界面/环节要展示的数据；先把现有案例整理成 JSON，再让大模型复制改写出其余案例，人工检查"别太离谱"。
- 补充详细信息时，**工控机品牌下拉必须有足够的品牌型号**（罗克韦尔、浙大中控、霍尼韦尔、施耐德等）；第二个案例可用罗克韦尔。
- 国家管网集团全国场站的公开资料可整理为背景知识；老师此前发的**现场实拍照片**回头可以用上。
- 研华案例保留（老师确认"这些故障基本上是通用的"），但要找其他品牌工控机资料**核对比对、整理更详细**。

**知识回流（赛题第 4 点）**：
- 演示级实现即可：历史案例加"已审核·已归档"状态，审核在 PC 端完成，归档后加入知识图谱。
- 知识图谱做**添加前/添加后两份 JSON**，先演示一遍、添加后再演示一遍；**新增节点要有标识**（标 new 或显示新增时间，时间显示为最近时间）。
- 图谱整体要**加多数据、弄复杂一些**；数据多了之后演示时专门定位到新增的那一块。

**演示形态**：
- 语音播报：预制生成好的完整音频内置（可用讯飞接口生成），不做实时合成。
- 专家电话会诊：做**模拟的腾讯会议式界面**——点击→勾选专家→确认→"正在接入"动效→接通后头像转正常，纯前端动画，演示到此可暂停讲解。
- 语音输入转文字：龙芯虚拟机麦克风权限不可用；**演示策略**为龙芯先跑主流程，讲到客户端时切 pad 展示语音输入等多模态，口径"现场使用通过移动终端"。

## 5. 老师否定或认为存在问题的内容

| 内容 | 老师的问题判断 | 处理 |
|---|---|---|
| 首页当前信息量 | "这个界面就内容有点太多" | 不是删除而是**收纳**：四周辅助区块改为点击展开 |
| 接诊后直接跳出固定四步 | "它为什么生成了四步呢""这两个界面之间得有一个分析过程""得根据不同情况生成四五六（步），那得是动态的" | 中段流程重构（本次最大改动） |
| 当前业务信息 | "没有体现这些实际业务的信息"——品牌型号只有研华，而油气行业实际以罗克韦尔为主 | 补品牌型号库 + 第二案例 |
| 当前知识图谱 | "这个太简单了"（只有节点和边的基本形态、数据少） | 加多数据、节点/边带属性、加新增标识 |
| 检修向导的相对价值 | 没有否定，但明确降权："但凡有点行业知识都能想得到……前面这个东西（怎么得到方案）是更关心的" | 开发重心前移 |

**注意一个表面矛盾**：上次意见"把功能平铺开" vs 本次"内容有点太多、可以隐藏"。两者不冲突——真实含义是**分层**：核心业务能力（采集入口、任务入口）要平铺可见，辅助信息（四周的状态、说明类区块）要收纳折叠。"平铺"反对的是藏在聊天框后面，"折叠"反对的是所有东西同权重堆满一屏。

## 6. 已确定、未确定和需要确认的事项

**已明确确定（可直接开发）**：中段流程重构（思考过渡 + 动态补充 + 预方案确认）；案例库 JSON 化（3 个）；品牌型号下拉扩充；知识回流演示闭环（审核归档 + 图谱前后对比 + new 标识）；图谱扩容；预制语音播报；专家会诊模拟界面；首页四周折叠；龙芯 + pad 分工演示策略。

**尚未确定**：图数据库/某工具在龙芯上的部署可行性（对话中提及"之前试过不行、后来说环境可以"，语焉不详，且老师明确说"用那个展示也行，不用也行"——演示不依赖它）；剪辑 3 音频中的内容。

## 8. 产品需求转化结果

### R1【P0】接诊思考过渡 + 详细信息动态补充
- **来源**：第 7 节前三行原话。**解决的问题**：接诊到步骤之间是黑箱、四步显得预制。
- **功能**：点"开始接诊"→ 进入思考过渡（流式思考文案 + 进度感，3-6 秒）→ 详细补充页逐块生成：现场照片居中，系统标注匹配位置（预制映射），四周按思考节奏依次弹出补充项卡（补充描述、选品牌型号、确认告警灯态、勾选检修资料等，每项带"确认"）；全部确认后进入 R2。
- **前端**：思考过渡组件（可全屏可悬浮）、逐块弹出动画、照片居中布局、补充项卡组件。**后端**：无新接口，数据来自案例 JSON。**素材**：每案例一张现场照片 + 位置映射标注。
- **验收**：演示中无任何"直接跳页"；每个补充项都由思考文案引出；照片定位标注措辞为"依据照片与台账匹配"（不写图像识别）。

### R2【P0】预方案确认页（可编辑步骤提纲）
- **来源**："预方案……增加、删除、修改步骤和检查项"。**解决的问题**：方案生成无确认环节、步骤数显得写死。
- **功能**：详细信息确认后弹"正在生成检修方案"思考窗 → 输出步骤提纲（步骤标题 + 检查项列表，带依据角标）→ 用户可增/删/改步骤和检查项 → 点"确认方案"→ 进入检修向导，向导按确认后的提纲渲染。
- **前端**：提纲编辑组件（增删改、拖拽可选）、确认态与向导数据打通。**后端**：无（编辑结果存前端状态/内存）。
- **验收**：现场可当场删一个检查项、加一个检查项，向导随之变化——这是"动态生成"最有力的现场证据。

### R3【P0】案例库 JSON 化（3 个案例）
- **来源**："设计个小案例库……一个 JSON 文件"。**功能**：定义案例 Schema：`case_id、设备(品牌/型号/位置)、现场照片、简要现象、思考文案(各阶段)、补充项清单、预方案提纲(步骤数各异)、向导步骤详情、验证阈值、证据条目、图谱增量`。案例一 = 现有研华散热案例整理迁移；案例二 = 罗克韦尔（品牌换、故障通用）；案例三自选（可为浙大中控或霍尼韦尔）。用大模型基于案例一生成二三，**人工逐条检查**。
- **验收**：切换案例 = 换一个 JSON，全流程（含步数）随之变化；三案例步数错开（如 3/4/5 步）。

### R4【P0】品牌型号业务字段库
- **来源**："工控机品牌型号就是实际业务信息……罗克韦尔用的多"。**功能**：详细补充页的品牌下拉含：罗克韦尔（Rockwell/AB）、浙大中控（SUPCON）、霍尼韦尔、施耐德、西门子、研华等，每品牌下挂 2-3 个代表型号（前端静态数据）。
- **验收**：现场点开下拉，评委/老师能看到完整的行业品牌谱系。

### R5【P0】知识回流演示闭环（赛题第 4 点）
- **来源**：第 7 节对应行。**功能**：检修记录页支持"上传检修案例总结"入口 → 记录状态流转：待审核→已审核·已归档 → 归档触发图谱更新：加载"添加后"JSON，新增节点/边带 `new` 标识与新增时间（显示近期时间），演示时可一键定位到新增区域。
- **验收**：可完整演示"一线上传→PC 审核→归档→图谱出现带 new 标识的新知识"，前后图谱肉眼可辨。

### R6【P1】知识图谱扩容

### R7【P1】专家会诊模拟界面

## 10. 推荐的 MVP 核心业务闭环（更新版）

登录 → 工作台（中部：简要接诊 = 一句现象 + 拍照/录音/视频样例 + 快捷类别勾选检修资料；四周辅助区折叠）→ 点"开始接诊" → **思考过渡 1**（解析现象、匹配台账）→ **详细信息补充页**（现场照片居中 + 位置匹配标注，四周随思考逐块弹出：品牌型号下拉（全品牌谱系）、灯态确认、阈值确认、资料勾选，逐项确认）→ **思考过渡 2**（"正在生成检修方案"，引用判据滚动）→ **预方案确认页**（步骤提纲 + 检查项，现场演示删一项/加一项）→ 确认 → **检修向导**（按确认后提纲渲染，每步图片 + 勾选 + 语音播报）→ 恢复验证 → 检修记录 + 作业卡 → **上传检修案例总结 → 专家审核 → 已归档 → 知识图谱新增节点（new 标识）**，可选演示专家连线模拟界面。全程右侧轨迹流持续输出思考与依据（上轮方案 B.3，与本次"思考过程"要求同源）。

体现"专用工具而非聊天机器人"的锚点：接诊表单化、照片定位、品牌台账、判据页码、可编辑作业方案、审核归档状态机——每个环节都是业务对象在流转，而不是消息在滚动。

## 12. 技术实现建议

- **全部前端模拟即可，本阶段不接真实大模型**：老师明确"以比赛演示为主"且给出案例库路线——思考文案、补充项、预方案、图谱增量全部预制在案例 JSON 中。学生问"哪个部分接真实 API 演示更好"，老师的回答实质是"先案例库"（未指定任何环节接真实 API）。
- **案例 Schema 是本次架构核心**：先用现有研华数据手工整理出 `case-01.json` 并跑通全流程，再让大模型仿写 `case-02`（罗克韦尔）`case-03`，人工逐条检查（老师原话"别太离谱"）。
- **不需要图数据库**：图谱继续 JSON + 前端布局；会议中关于龙芯部署某工具的讨论无结论且老师明确"不用也行"。
- **可复用**：现有异常接入四步的字段与校验逻辑（换壳进补充项卡）、Agent 定时器（驱动思考过渡）、向导渲染（改数据源）、审核模拟接口。
- **接口不新增**：案例 JSON 可由现有静态托管或 `/api/demo/scenario` 加 case 参数返回（后端 10 行内改动，也可纯前端 import）。
- **降复杂度关键**：补充项"逐块弹出"用固定时序数组驱动即可，不必真实依赖思考内容解析；预方案编辑状态存 React state，不落库。
