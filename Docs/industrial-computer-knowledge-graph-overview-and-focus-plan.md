# 工控机知识图谱全景与局部交互实施方案

> 版本：V1.0
> 日期：2026-07-13
> 当前状态：方案已确认，下一步进入实现
> 适用范围：专家知识图谱、工程师候选知识提交、案例回流与知识发布展示
> 开发规格：`industrial-computer-knowledge-graph-development-spec.md`

## 1. 本阶段决策

本阶段建设一张以“工控机”为中心的综合知识图谱。图谱允许包含环境、供电、硬件、安装运维和监控告警等多个故障领域，但当前只将“散热异常”建设为唯一完整、可操作、可回流的案例闭环。

当前确定：

- 图谱中心统一使用“工控机”，不绑定某个具体品牌或型号；
- 底层维护一张可持续扩展的大图谱，不为每个故障单独建设互不关联的小图；
- 全局视图展示知识覆盖范围，并支持通过案例、知识版本或节点定位相关局部关系；
- 散热异常保留现有完整业务闭环；
- 其他故障第一阶段只作为展示知识，允许查看节点与关系，不建设完整接诊和检修流程；
- 工程师可以提出候选节点与关系，但不能直接修改正式图谱；
- 候选变更经过专家审核和知识发布后，才进入正式图谱并参与后续诊断；
- 粉尘堵塞、温度凝露、供电异常等轻量案例及真实手册补充列为后续阶段，不阻塞本阶段实现。

## 2. 当前基础

项目已经具备：

- 工程师检修记录生成和案例提交；
- 工程师与专家独立账号；
- 专家结构化审核和知识发布；
- `KB-008 V1.0 → V1.1` 版本变化；
- SQLite 知识版本、正式关系和工程师同步状态持久化；
- 专家“当前图谱 / 本次更新”视图；
- 工程师本地知识图谱和版本同步；
- 发布后知识参与再次验证的完整演示链路。

当前不足：

- 现有图谱仍以散热异常关系为主，领域覆盖不够直观；
- 旧图谱 JSON、演示图谱种子、SQLite 正式关系和前端硬编码节点尚未完全统一；
- 专家图谱中的部分节点和坐标写在前端，数据扩展成本较高；
- 全局知识和当前案例局部知识缺少清晰的视觉层级；
- 工程师提出候选图谱变更的页面状态还不完整；
- 除散热异常外，其他故障暂时缺少来源案例和正式资料，不应表述为已验证知识。

## 3. 建设目标

### 3.1 产品目标

形成以下演示表达：

> 系统维护的是一张覆盖多类工控机故障的综合知识图谱；本次演示只聚焦散热异常，通过工程师案例、专家审核和知识发布，展示局部知识如何新增、修改并正式进入大图谱。

### 3.2 功能目标

本阶段完成：

1. 建立工控机全局知识图谱数据结构；
2. 展示五个故障领域及其代表性故障节点；
3. 支持“全局知识图谱”和“本次新增与修改”两种正式视图；
4. 支持点击节点查看详情、来源、状态和关联关系；
5. 支持在全局图谱或本次变更中通过当前案例、`KB-008` 或节点定位并放大相关关系；
6. 支持工程师提出候选节点和候选关系；
7. 支持专家审核、修改、合并或拒绝候选变更；
8. 发布后将正式关系写入 SQLite，并在图谱中显示知识版本与案例来源；
9. 保持现有工程师本地知识同步和再次验证链路不回归。

### 3.3 演示目标

演示时能够依次完成：

```text
打开工控机知识图谱全景
  -> 展示多个故障领域和知识规模
  -> 切换到本次新增与修改
  -> 自动放大工程师提出的候选节点和关系
  -> 专家审核并发布 KB-008 V1.1
  -> 候选关系转为正式关系
  -> 工程师同步新版本
  -> 新关系参与后续知识验证
```

## 4. 图谱范围

### 4.1 中心节点

中心节点固定为：

```text
工控机
```

具体品牌和型号作为“设备型号”节点或节点属性存在，不作为整个图谱的中心名称。

### 4.2 第一阶段故障领域

全局图谱展示五个一级领域：

1. 环境工况异常；
2. 供电系统故障；
3. 硬件部件失效；
4. 安装与运维不规范；
5. 监控告警失效。

代表性二级内容：

```text
环境工况异常
  ├─ 温度超标
  ├─ 湿度凝露
  ├─ 粉尘堵塞
  └─ 振动冲击

供电系统故障
  ├─ 外部市电异常
  ├─ 电源本体异常
  ├─ 配电附件异常
  └─ 电压监测异常

硬件部件失效
  ├─ 散热组件异常
  ├─ 存储硬盘故障
  ├─ 主板与底板故障
  ├─ 监控模块故障
  └─ RTC 电池异常

安装与运维不规范
  ├─ 安装错误
  ├─ 检修违规
  ├─ 参数配置错误
  ├─ 配件混用
  └─ 返修违规

监控告警失效
  ├─ LED 指示失效
  ├─ 蜂鸣器线路故障
  ├─ SMBus / PMBus 通讯中断
  └─ 告警复位接口故障
```

第一阶段这些内容用于展示图谱覆盖面，不代表全部完成真实案例验证。

## 5. 散热异常唯一完整闭环

散热异常是第一阶段唯一需要达到完整业务深度的子图，至少覆盖：

```text
工控机
  -> 散热组件异常
  -> TEMP/FAN 告警
  -> 风扇转速低于 500 rpm
  -> 滤网积尘 / 风道受阻 / 风扇老化
  -> 清理滤网与风道
  -> 清理后复测转速
  -> 更换老化风扇
  -> 断电、挂牌、防静电
  -> 告警解除、转速恢复、温度恢复
  -> 当前散热检修案例
  -> KB-008 V1.1
```

该子图需要与现有实际流程保持一致：

- 现场异常事实来自当前完整案例；
- 工程师确认最终原因和实际处理；
- 工程师提出候选知识及关系；
- 专家审核候选内容；
- 发布结果写入正式知识版本和图谱关系；
- 工程师同步后新知识参与现场问答或诊断验证。

## 6. 两种图谱视图

### 6.1 全局知识图谱

用途：展示知识覆盖范围。

要求：

- 默认显示“工控机”和五个一级故障领域；
- 二级节点按领域形成知识簇；
- 大量叶子节点允许聚合或降低视觉权重；
- 显示节点数、关系数、领域数、正式知识数和来源案例数；
- 不在初始画面同时展开所有节点详情。
- 支持搜索或定位案例、知识版本和节点；
- 定位后可以临时放大相关一至两跳关系，但不形成独立的“案例聚焦”视图。

### 6.2 本次新增与修改

用途：展示知识回流造成的图谱变化。

本视图自动定位并放大本次候选或正式发布关系的一至两跳邻居。其他无关节点保留在背景并降低透明度，不再额外设置“散热案例聚焦”视图。

状态表达：

- 既有节点：中性样式；
- 工程师候选节点：黄色高亮；
- 工程师候选关系：黄色虚线；
- 专家修改节点：橙色高亮；
- 专家拒绝或删除关系：红色删除标记；
- 专家发布节点：绿色高亮；
- 正式发布关系：绿色实线；
- 其他无关节点：降低透明度。

## 7. 工程师与专家权限边界

### 7.1 工程师

工程师可以：

- 从当前检修案例生成候选节点和关系；
- 修改候选节点名称与说明；
- 补充判断条件、实际处理和恢复结果；
- 选择关联已有节点或建议新增节点；
- 为候选节点绑定现场参数和案例证据；
- 提交专家审核。

工程师不能：

- 直接修改已发布正式节点；
- 直接删除正式关系；
- 直接让候选知识参与正式诊断。

### 7.2 专家

专家可以：

- 查看工程师候选变更及来源证据；
- 修改节点名称、类型和关系语义；
- 将候选节点合并到已有节点；
- 接受、拒绝或补充候选关系；
- 补充安全条件、适用范围和恢复标准；
- 发布正式知识版本和图谱关系。

## 8. 数据模型要求

### 8.1 节点

建议统一为：

```json
{
  "id": "CAUSE-FAN-AGING",
  "name": "风扇老化",
  "type": "fault_cause",
  "domain": "hardware",
  "status": "candidate",
  "changeType": "added",
  "verificationLevel": "verified_case",
  "knowledgeId": "KB-008",
  "knowledgeVersion": "1.1",
  "sourceCaseIds": ["CASE-ACP4000-001"],
  "evidenceRefs": ["清理后风扇转速仍偏低", "更换后转速恢复"],
  "proposedBy": "lishifu",
  "reviewedBy": null
}
```

### 8.2 关系

建议统一为：

```json
{
  "id": "REL-FAN-AGING-LOW-RPM",
  "source": "CAUSE-FAN-AGING",
  "relation": "causes",
  "target": "STATE-FAN-RPM-LOW",
  "status": "candidate",
  "changeType": "added",
  "knowledgeId": "KB-008",
  "knowledgeVersion": "1.1",
  "sourceCaseId": "CASE-ACP4000-001",
  "evidenceType": "verified_case",
  "evidenceRef": "恢复前后风扇转速记录"
}
```

### 8.3 可信等级

统一使用：

| 等级 | 含义 |
| --- | --- |
| `verified_case` | 已由完整案例验证 |
| `official_document` | 来源于正式手册或规范 |
| `expert_confirmed` | 已由专家确认 |
| `presentation_seed` | 用于展示覆盖面的预置知识 |
| `collecting` | 正在收集资料，尚未形成正式知识 |

其他故障领域第一阶段主要使用 `presentation_seed` 或 `collecting`，不得描述为已经通过真实案例验证。

## 9. 数据源统一要求

下一轮实现需要逐步解决以下数据分散问题：

- 旧版 `backend/data/graph_relations.json`；
- `backend/data/presentation/graph_seed.json`；
- SQLite `graph_relations` 正式关系；
- `AdminShell.jsx` 中硬编码的图谱节点、位置和关系；
- 工程师本地知识快照中的关系。

建议采用：

```text
JSON 图谱种子
  -> 初始化 SQLite 正式节点和关系
  -> 工程师提交候选变更
  -> 专家审核形成正式版本
  -> 图谱 API 按视图返回全局、局部或变更数据
  -> 专家端和工程师端读取同一套节点、关系与版本模型
```

前端仅负责布局、聚焦和样式，不再把业务节点硬编码为唯一数据源。

## 10. 本阶段不做

- 不建设真实图数据库；
- 不要求图谱自动从 PDF 抽取；
- 不为全部故障建设完整诊断和检修流程；
- 不要求所有展示节点都有真实案例；
- 不把展示知识包装成已验证知识；
- 不建设复杂的自由拖拽图谱编辑器；
- 不建设多人实时协同编辑；
- 不扩展真实大模型或自动知识抽取能力。

## 11. 下一阶段事项

以下内容记录为下一步，不阻塞当前大图谱和局部交互：

1. 补充粉尘堵塞轻量案例；
2. 补充温度凝露轻量案例；
3. 补充冗余电源或供电异常案例；
4. 补充 RTC 电池、硬盘或主板异常案例；
5. 补充蜂鸣器或监控总线异常案例；
6. 收集相应官方手册、告警说明、安全规程和恢复标准；
7. 将轻量案例与图谱节点建立可追溯关系；
8. 根据真实资料把 `presentation_seed` 节点升级为 `official_document`、`expert_confirmed` 或 `verified_case`。

轻量案例只需包含案例编号、环境、设备类型、异常现象、关键证据、最终原因、处理方法、恢复结果和关联知识，不复制散热案例的完整操作流程。

## 13. 验收标准

### 全局图谱

- 中心节点显示为“工控机”；
- 能看到五个一级故障领域；
- 图谱具有足够的展示节点和关系；
- 默认页面不因节点过多而重叠或不可读；
- 能按领域查看代表性二级节点。

### 局部交互

- 可以定位当前散热检修案例；
- 定位或进入本次变更后只突出相关一至两跳关系；
- 其他领域仍存在但降低视觉权重；
- 点击节点能查看类型、状态、来源、版本、案例和关联关系；
- 可以在全局知识图谱和本次新增与修改之间稳定切换；
- 页面不存在独立的“散热案例聚焦”视图。

### 变更闭环

- 工程师可以提交候选节点或关系；
- 候选内容不会直接进入正式诊断；
- 专家可以修改、合并、拒绝或接受候选变更；
- 发布后候选关系转为正式关系；
- 正式节点可以追溯到案例和知识版本；
- 工程师同步后新关系参与现有知识验证。

### 工程质量

- 图谱业务数据不再只依赖前端硬编码；
- SQLite 发布事务和现有测试不回归；
- Node `20.19.4` 构建通过；
- 后端自动化测试通过；
- 1366×768 和 1920×1080 主视口无关键节点遮挡；
- WSL 验证后再进入龙芯开发态验证。

## 14. 当前结论

本阶段先完成“全局知识图谱 + 本次新增与修改 + 候选变更 + 专家发布”的能力，不同时扩展多个完整案例。

散热异常继续作为唯一完整闭环，其他故障用于证明图谱的知识覆盖和扩展方向。待当前交互和数据模型稳定后，再逐步补充粉尘堵塞、温度凝露、供电异常等轻量案例及其真实资料。
