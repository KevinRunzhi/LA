# 检修作业卡 PDF 导出 Spec

> 版本：V1.0
> 日期：2026-07-14
> 范围：仅改造工程师端“打印作业卡”功能
> 非目标：不改造检修记录页面、专家审核页面、知识库页面和知识图谱页面

## 1. 背景

当前“打印作业卡”直接调用 `window.print()` 打印检修记录交互页面。页面中的导航、按钮、输入框、诊断过程、执行轨迹和知识回流信息会一起进入 PDF，且没有 A4 分页规则，因此导出的文档内容层级混乱、分页不可控，也不能突出本次检修结论。

本次改造不重新设计现有业务页面，而是为打印动作增加一套独立的正式文档视图。屏幕继续显示原检修记录页面；用户点击“打印作业卡”后，浏览器只打印专用的两页 A4 作业卡。

## 2. 文档定位

本功能导出的正式文档名称为：

**工控机故障检修作业卡**

该文档生成于检修执行和恢复验证完成之后，核心任务是讲清楚：

1. 什么设备在什么位置发生了什么故障；
2. 现场确认的最终原因是什么；
3. 实际采取了哪些处理措施；
4. 处理前后参数是否证明设备已经恢复；
5. 是否存在遗留风险、后续建议和审核要求。

详细操作指导仍由检修向导承担。PDF 只保留已执行步骤的摘要和完成状态，不复制 Agent 思考过程、知识检索过程或大段操作说明。

## 3. 用户流程

1. 工程师完成检修向导；
2. 工程师在检修记录页确认最终原因、实际处理、恢复结果和恢复参数；
3. 工程师点击“打印作业卡”；
4. 系统组装当前记录对应的正式打印数据；
5. 浏览器打开打印对话框，只呈现两页 A4 作业卡；
6. 用户选择“另存为 PDF”或直接打印；
7. 推荐文件名为 `工控机故障检修作业卡_<记录编号>_<日期>.pdf`。

## 4. 两页内容结构

### 4.1 第一页：故障闭环与恢复验证

第一屏必须在短时间内回答“发生了什么、怎么修好的、是否恢复”。

1. 文档抬头
   - 系统名称
   - 文档名称
   - 记录编号
   - 当前状态
   - 作业日期
2. 检修结论摘要
   - 最终故障原因
   - 实际处理措施
   - 恢复结果
3. 基本信息
   - 设备
   - 场站和位置
   - 故障与告警
   - 发生时间
   - 检修人员和班组
   - 检修完成情况
4. 故障闭环
   - 现场现象
   - 最终故障原因
   - 实际处理
   - 恢复结果
5. 处理前后参数对比
   - 风扇转速
   - 系统温度
   - CPU 温度
   - TEMP/FAN 告警
   - 连续观察时间

### 4.2 第二页：执行摘要、风险与追溯

第二页用于证明作业过程完整、结果可追溯。

1. 已完成检修步骤
   - 只显示步骤名称、完成状态和安全提示
   - 不展开每一步的长篇操作说明
2. 现场证据摘要
   - 材料数量
   - 图片、视频、音频或附件名称
   - 无材料时明确显示“本次未附加现场材料”
3. 遗留风险与后续建议
   - 遗留风险
   - 建议复查时间或维护建议
   - 专家复核状态
4. 记录追溯
   - 来源记录编号
   - 目标案例编号
   - 关联知识编号
   - 生成时间
5. 签字区
   - 检修人员
   - 复核人员
   - 审核日期

## 5. 打印数据模型

首版继续复用现有检修记录数据，但打印组件必须通过明确字段接收数据，不能从页面 DOM 读取文本。

```js
{
  document: {
    recordId,
    caseId,
    status,
    generatedAt
  },
  equipment: {
    name,
    site,
    location
  },
  incident: {
    occurredAt,
    duration,
    recurrence,
    fault,
    description
  },
  result: {
    finalCause,
    actualResolution,
    recoveryResult,
    residualRisk
  },
  recovery: {
    before,
    after,
    observationMinutes
  },
  execution: {
    completedSteps,
    safetyConfirmed,
    materials
  },
  traceability: {
    engineer,
    team,
    targetKnowledgeIds
  }
}
```

后续后端正式保存检修记录时，应由记录接口直接返回同构数据，打印组件无需再次拼装。

## 6. 字段规则

1. 最终结论优先使用工程师确认字段，不能继续使用早期“疑似”诊断替代；
2. Agent 诊断只作为过程信息，不进入正式打印结论；
3. 数值必须携带单位；
4. 空值统一显示“未记录”，不得使用虚构内容填充；
5. 未完成的步骤必须明确显示“未完成”，不能计入完成数量；
6. 历史记录和本次记录最终应使用同一打印结构；
7. 第一版只面向当前完整散热异常案例，但字段名称保持通用，便于后续扩展其他工控机故障。

## 7. 视觉与打印规范

- 纸张：A4 纵向；
- 页数：固定两页；
- 页边距：页面内容区约 14 mm；
- 颜色：白底、深蓝灰正文、青绿色状态强调；
- 字体：优先使用思源黑体/思源宋体，回退到系统中文字体；
- 标题、表格、步骤和签字区不得跨页切断；
- 每页显示记录编号和页码；
- 不打印导航栏、顶栏、返回按钮、输入框、上传按钮和滚动条；
- 不使用深色屏幕背景、阴影动画和透明模糊效果；
- 不依赖截图拼接，文字必须保持可选择和可检索。

## 8. 技术方案

### 8.1 首版

1. 新增独立 React 打印组件 `MaintenanceJobCardPrint`；
2. 使用 React Portal 将打印文档挂到 `body`，与现有应用 DOM 分离；
3. 屏幕状态下隐藏打印组件；
4. 点击“打印作业卡”时给 `body` 增加打印状态类并调用 `window.print()`；
5. `@media print` 中隐藏 `#root`，只显示打印组件；
6. 使用 `@page`、毫米尺寸和分页规则固定为两页 A4；
7. 打印结束后恢复页面标题和打印状态类。

该方案不增加前后端依赖，适合当前 React + Vite + Flask 演示环境，也能通过浏览器“另存为 PDF”得到正式文件。

### 8.2 后续增强

需要服务端直接下载 PDF 时，再增加：

```text
GET /api/records/{recordId}/pdf
```

服务端必须复用同一报告数据模型和同一打印模板，避免屏幕预览与下载文件内容不一致。

## 9. 异常处理

- 打印所需关键字段为空时，仍允许预览，但显示“未记录”；
- 没有现场材料时不产生空白图片区，只显示文字说明；
- 步骤数量变化时第二页自动排列，但不得溢出 A4；
- 中文长文本允许自然换行，表格单元格不得横向溢出；
- 浏览器取消打印后必须恢复原页面标题和正常界面；
- 其他页面原有打印行为不得受到本组件的打印样式影响。

## 10. 验收标准

1. 点击现有“打印作业卡”按钮后，打印预览只出现正式作业卡；
2. PDF 固定两页 A4，第一页为故障闭环，第二页为执行与追溯；
3. 导出内容不包含应用导航、按钮、输入框、滚动条和交互状态；
4. 最终原因、实际处理和恢复结果来自工程师当前确认值；
5. 处理前后参数和单位完整、数值一致；
6. 步骤只保留清晰摘要，不重复长篇操作过程；
7. 空字段、无材料和未审核状态有明确表达；
8. 中文无乱码、黑块、截断、重叠或表格断裂；
9. 两页均显示记录编号和页码；
10. 前端生产构建通过，并将实际 PDF 逐页渲染为图片完成视觉检查。
