# R0【P0】单页动态接诊工作区 Spec

> 版本：V1.4
> 日期：2026-07-10
> 适用范围：首页提交现场现象，到正式进入“分析诊断”之前
> 优先级：P0
> 实现性质：前端高保真模拟，不新增依赖，不接真实大模型和设备接口

---

## 0. 2026-07-10 已确认决策与实施批次

实施状态：Windows/WSL 开发侧的 R0 主交互链路已经贯通；现场材料、位置确认、设备信息、告警状态和独立接入摘要页均已完成浏览器验证。尚未完成预制条件分支、逐字段来源展示、全量异常状态和龙芯端逐模块验证，因此 R0 仍为“进行中”。

本轮在原 Spec 基础上确认以下调整：

1. 首页除异常描述外，必须提供现场材料入口，至少包含故障图片和故障视频；语音、检修资料保留扩展入口。
2. 用户选中的图片或视频必须带入动态接诊工作区并在中央展示，不能继续只显示固定示意图。
3. 第一版只做浏览器本地选择、预览、替换、删除和状态传递；不上传服务器、不持久化、不进行真实图片或视频识别。
4. 每张动态卡片出现前必须说明“已有信息、信息缺口、参考依据、为什么生成该任务”。
5. 支持 2–3 个预制条件分支，让不同确认结果生成不同的补充任务，而不是所有输入播放同一套卡片。
6. 自动推荐按字段逐项出现，并显示来源类型；用户仍需确认或修改。
7. 接入摘要分为原始描述、系统整理结果和本次诊断任务三层。
8. “点击依据联动高亮图片区域”暂缓，不进入 R0。

2026-07-10 第二轮界面确认：

- 图片、视频、音频统一作为现场材料，默认以紧凑四宫格/九宫格展示，不让单个材料铺满页面。
- 1–4 项采用自适应四宫格，5–9 项采用九宫格；超过可显示 `+N`，点击单项后再进入大图、视频或音频预览。
- 接诊工作区中央展示完整材料集合，不再只展示“当前选中的一张图片”。
- 场站地图从现场证据中拆出，形成独立“设备位置识别”小模块，展示场站、区域、机柜、设备路径及待确认状态。
- 设备位置识别不与材料区并排常驻：开始接诊后，接诊 Agent 首先生成位置识别任务卡；工程师确认后，位置识别 Agent 才生成“补充设备信息”任务。
- 位置任务卡的标题、状态、确认按钮和完成摘要与设备信息、告警状态任务保持一致；中央材料宫格在位置任务生成时保持居中且不缩窄。

R0 按以下批次依次实施：材料采集与展示 → 卡片生成原因 → 条件分支 → 逐字段推荐 → 分层摘要 → 全链路与龙芯验证。

### 0.2 条件分支定义

| 分支 | 触发条件 | 后续任务变化 | 摘要诊断方向 |
| --- | --- | --- | --- |
| 标准散热 | 工控机 + TEMP/FAN 告警 + 风扇低速 | 保留 TEMP/FAN、风扇转速、系统温度、CPU 温度确认 | 工控机散热异常方向 |
| 无灯态高温 | 选择“仅温度升高（无 TEMP/FAN）”或将灯态改为正常/未点亮 | 保留温度类字段，提示强告警证据减少 | 环境散热 / 传感器复核优先 |
| 设备/告警不一致 | 选择 PLC 控制柜或通信中断/数据不上送 | 后续任务切换为控制器 RUN、通信 LINK、数据上送、电源状态；摘要阻止误入散热诊断 | 控制器 / 通信链路待补充 |

重新评估结果必须同时出现在当前任务卡、右侧辅助对话和独立接入摘要页，不能只更换一条提示文案。
R0 只闭环主演示的工控机散热诊断；设备/告警不一致分支用于证明系统能够停止错误推断并生成不同补充任务，不伪造尚未建设的 PLC/通信诊断结论。

## 1. 文档目标

本 Spec 用于把当前“异常接入四个整页依次跳转”的交互，重构为“Agent 在同一个现场工作区内逐步生成补充内容”。

改造后的体验必须体现：

> 工程师不是在填写一套预先暴露的固定表单，而是在查看故障位置的同时，根据 Agent 的分析结果，逐项补充系统临时提出的现场信息。

本次只重构首页到诊断页之间的异常接入流程，不修改后续多 Agent 会诊、诊断结论和检修向导的业务内容。

---

## 2. 当前问题

当前实现使用 `activeStep` 在以下四个整页之间切换：

1. 描述现场现象；
2. 补充设备型号；
3. 确认灯值和阈值；
4. 接入信息确认。

存在的问题：

- 页面跳转让流程看起来像固定表单；
- 所有步骤在产品设计上已经提前确定，Agent 只像装饰；
- 故障位置图只在第一步出现，后续失去现场上下文；
- 设备信息和阈值信息分别占满整页，页面空间利用不自然；
- “Agent 生成下一步”的叙事与实际页面结构不一致；
- 评委难以感受到系统在根据现场信息动态组织任务。

---

## 3. 目标体验

### 3.1 核心原则

1. 异常接入的逐项确认阶段使用一个动态工作页面；全部确认后进入独立“接入摘要确认页”。
2. 首页提供故障图片、故障视频等现场材料入口；材料不做真实识别，但必须能被选择和预览。
3. 中央现场证据模块始终保留，以紧凑宫格同时展示用户选择的图片、视频和音频；单项只有点击后才放大预览。
4. 设备位置识别是独立小模块，不覆盖在现场材料上。
5. Agent 工作过程持续显示，不单独跳转到“思考页面”。
6. Agent 必须先展示信息缺口和生成原因，再出现对应补充卡片。
7. 未生成的卡片完全隐藏，不显示空占位和“待生成”。
8. 当前卡片高亮，已完成卡片收缩成摘要，避免页面越堆越乱。
9. 至少一个用户修改动作能够触发预制的重新评估或额外补充任务。
10. 全部信息确认后，进入独立接入摘要确认页，并在核对后提供“启动智能诊断”操作。
11. 流程节点、Agent 文案和浮现卡片必须同步。

### 3.2 目标流程

```text
首页输入现场现象
  -> 选择故障图片/视频等现场材料（可使用预置样例）
  -> 点击“开始接诊”
  -> 进入单页动态接诊工作区
  -> 中央以宫格展示全部现场材料
  -> 接诊 Agent 自动开始流式分析
  -> Agent 生成设备位置识别任务
  -> 工程师确认场站 / 控制中心 / 站控柜位置
  -> 位置识别 Agent 生成设备信息任务
  -> 设备信息卡浮现
  -> 用户自动识别或手动确认
  -> Agent 继续核对设备告警依据
  -> 灯态与运行状态卡浮现
  -> 用户确认或修改
  -> Agent 汇总接入信息
  -> 进入独立“接入摘要确认页”
  -> 汇总现场事件、现场材料、故障位置、设备告警与运行参数
  -> 用户点击“启动智能诊断”
  -> 进入现有分析诊断流程
```

## 4. 页面总体布局

### 4.1 桌面端布局

```text
┌──────────┬──────────────────────────────────────┬──────────────┐
│ 诊断流程 │              动态接诊工作区            │ Agent 工作区  │
│          │                                      │              │
│ 渐进节点 │   ┌────────设备信息卡（动态）──────┐   │ 流式分析     │
│          │   │                              │   │ 检索依据     │
│          │   └──────────────────────────────┘   │ 当前建议     │
│          │                                      │              │
│          │       ┌────────────────────┐         │              │
│          │       │  现场证据画布        │         │              │
│          │       │  上传图片/视频展示   │         │              │
│          │       └────────────────────┘         │              │
│          │                                      │              │
│          │   ┌────告警状态卡（动态）────────┐   │              │
│          │   └──────────────────────────────┘   │              │
│          │   全部确认后进入独立接入摘要确认页      │              │
└──────────┴──────────────────────────────────────┴──────────────┘
```

### 4.2 布局实现约束

- 动态卡片不能使用完全自由的随机坐标。
- 主区域采用 CSS Grid 或预设锚点槽位：`top-left`、`top-right`、`bottom-left`、`bottom-right`、`summary`。
- 卡片视觉上像“浮现”，结构上仍处于稳定布局中。
- 卡片出现和收缩不得改变中央现场证据画布的尺寸。
- 主工作区允许纵向滚动，但当前操作和主要图片应尽量在首屏可见。
- 左侧诊断流程与右侧 Agent 工作区各自可滚动，不挤压中央内容。
- 不实现拖动、自由缩放、连线或复杂空间布局。

### 4.3 现场材料宫格与设备位置模块

现场材料区域按以下方式展示：

1. 图片显示缩略图，视频显示封面/播放标识，音频显示音频图标与文件信息。
2. 1–4 项为自适应两列宫格，5–9 项为三列宫格；材料卡保持紧凑，不因单个文件扩大到整个区域。
3. 点击材料卡后打开独立预览层；默认工作区始终展示材料集合。
4. 没有用户材料时显示“尚未接入现场材料”和预置样例入口。
5. 场站位置图迁移至独立“设备位置识别”模块；识别说明放在右侧辅助对话，卡片只保留场站图、故障点、简短位置和工程师确认操作。

要求：

- 首页提供清晰的“故障图片”“故障视频”选择入口，语音和检修资料保留扩展入口；
- 支持选择、预览、替换和删除，重新开始演示时清理对象 URL；
- 上传材料在整个异常接入阶段不消失；
- 保留站控柜 A01 位置上下文和故障点动效；
- 卡片不能遮挡主要故障点；
- 后续可由案例数据替换预置样例、位置图和故障点坐标；
- 位置识别必须由工程师确认；确认后位置图作为独立小模块保留在后续步骤中。

边界文案统一使用“已选择现场材料”“依据现场材料与设备台账进行匹配”，不得写“图片识别成功”或“视频分析完成”。

## 5. 交互状态机

当前四页 `activeStep` 不再直接决定整页内容，改为单页工作区状态。

建议状态定义：

```js
const intakeWorkspaceStates = {
  ANALYZING_SYMPTOM: "analyzing_symptom",
  LOCATION_CONFIRMATION_REQUIRED: "location_confirmation_required",
  LOCATION_CONFIRMED: "location_confirmed",
  DEVICE_INFO_REQUIRED: "device_info_required",
  RECOGNIZING_DEVICE: "recognizing_device",
  DEVICE_INFO_CONFIRMED: "device_info_confirmed",
  ALARM_INFO_REQUIRED: "alarm_info_required",
  ALARM_INFO_CONFIRMED: "alarm_info_confirmed",
  BUILDING_SUMMARY: "building_summary",
  READY_FOR_DIAGNOSIS: "ready_for_diagnosis",
  STARTING_DIAGNOSIS: "starting_diagnosis",
};
```

### 5.1 状态转换

| 当前状态 | 触发条件 | 下一状态 | 页面变化 |
| --- | --- | --- | --- |
| `ANALYZING_SYMPTOM` | 首页输入提交后自动运行接诊 Agent | `LOCATION_CONFIRMATION_REQUIRED` | 设备位置识别卡浮现，材料区保持独立 |
| `LOCATION_CONFIRMATION_REQUIRED` | 工程师确认场站和故障点 | `LOCATION_CONFIRMED` | 位置图保留为独立小模块，右侧输出确认依据 |
| `LOCATION_CONFIRMED` | 位置识别 Agent 完成设备台账匹配 | `DEVICE_INFO_REQUIRED` | 设备信息卡浮现 |
| `DEVICE_INFO_REQUIRED` | 点击自动识别 | `RECOGNIZING_DEVICE` | 卡内显示识别过程 |
| `RECOGNIZING_DEVICE` | 模拟识别完成 | `DEVICE_INFO_REQUIRED` | 建议值逐项填入，等待人工确认 |
| `DEVICE_INFO_REQUIRED` | 用户确认设备信息 | `DEVICE_INFO_CONFIRMED` | 设备卡收缩为摘要 |
| `DEVICE_INFO_CONFIRMED` | Agent 完成告警依据检索 | `ALARM_INFO_REQUIRED` | 告警状态卡浮现 |
| `ALARM_INFO_REQUIRED` | 用户确认灯态和运行状态 | `ALARM_INFO_CONFIRMED` | 告警卡收缩为摘要 |
| `ALARM_INFO_CONFIRMED` | 自动开始汇总 | `BUILDING_SUMMARY` | Agent 输出接入汇总过程 |
| `BUILDING_SUMMARY` | 汇总完成 | `READY_FOR_DIAGNOSIS` | 切换到独立“接入摘要确认页” |
| `READY_FOR_DIAGNOSIS` | 用户点击启动智能诊断 | `STARTING_DIAGNOSIS` | 按钮 loading，进入现有诊断链路 |

### 5.2 状态保护

- Agent 运行期间禁止重复触发当前动作。
- 当前必填信息未确认时，不生成下一张卡片。
- 已生成的卡片允许重新展开修改。
- 修改已完成卡片后，其后续摘要应同步更新。
- `STARTING_DIAGNOSIS` 状态不得再次点击触发诊断。
- 离开页面时清理全部计时器，避免返回后重复生成。

## 6. 动态内容生成顺序

### 6.1 阶段 A：接诊 Agent 分析现场现象

进入工作区后自动运行，不要求用户再次点击“整理现场描述”。

建议流式内容：

```text
正在解析现场异常描述……
识别到温度告警与风扇声音异常。
正在定位相关设备区域……
已匹配站控柜 A01 工控机区域。
正在核对设备台账和告警关系……
当前信息不足，需要补充设备型号、位置和角色。
```

建议依据逐步出现：

- 场站设备台账：站控柜 A01；
- 现场异常描述：温度告警、风扇声音异常；
- 设备位置关系：控制中心工控机区域；
- ACP-4000 / IPC-610 设备类型索引。

完成结果：

```text
已生成“补充设备信息”任务。
```

### 6.2 阶段 B：设备信息卡

出现位置：主工作区左上或中央图左上方预设槽位。

字段：

| 字段 | 初始值 | 系统建议值 |
| --- | --- | --- |
| 设备型号 | 空 | 研华 ACP-4000 / IPC-610 |
| 设备角色 | 空 | 站控画面与数据采集终端 |
| 关联告警 | 空 | TEMP/FAN 告警 |

交互：

- 用户可以逐项使用下拉菜单选择；
- 可以点击“自动识别”；
- 自动识别时，在卡片内部逐行显示“读取描述、匹配台账、核对型号、关联告警”；
- 建议值按字段逐项填入，不得瞬间全部出现；
- 自动识别完成后仍需用户点击“确认设备信息”；
- 确认后卡片收缩成一条设备摘要，并显示“已完成”；
- 摘要提供“修改”按钮，可重新展开。

### 6.3 阶段 C：告警状态与运行值卡

出现条件：设备信息已确认，Agent 完成对应手册和告警依据检索。

建议 Agent 内容：

```text
设备型号已确认。
正在检索 ACP-4000 / IPC-610 告警定义……
已匹配 TEMP/FAN 指示灯和散热异常相关条目。
需要核对现场灯态与运行状态。
```

出现位置：主工作区右下或中央图右侧预设槽位。

第一版字段：

| 字段 | 建议显示方式 | 系统建议值 |
| --- | --- | --- |
| TEMP/FAN 灯态 | 下拉选择 | 红灯，异常 |
| 风扇运行状态 | 下拉选择 | 转速偏低 |
| 系统温度状态 | 下拉选择或可编辑文本 | 高温异常 |
| CPU 温度状态 | 下拉选择或可编辑文本 | 高温异常 |

约束：

- 页面文案以“状态确认”为主，不强制展示具体温度数值；
- 若保留阈值，阈值放在“查看判据”内，不占用主卡片；
- 支持“采用系统建议”；
- 用户确认后收缩为告警摘要，并显示“已完成”；
- 摘要中至少展示 TEMP/FAN 异常和风扇运行异常。

### 6.4 阶段 D：独立接入摘要确认页

出现条件：设备信息和告警状态均已确认，Agent 汇总完成。

出现方式：完成最后一项确认后，从动态接诊工作区切换到独立摘要确认页；左侧流程栏和右侧辅助对话继续保留。

必须包含：

- 现场异常摘要；
- 故障位置；
- 设备型号和角色；
- 关键告警状态；
- 已接入的现场材料类型；
- 下一阶段将执行的诊断任务；
- “启动智能诊断”主按钮。

摘要分三层展示：

1. 现场原始描述与已选材料。
2. 系统整理结果：位置、设备、现象、告警和用户修正项。
3. 本次诊断任务：接下来要核对的依据和要生成的结果。

诊断任务示例：

1. 解析 TEMP/FAN 告警与风扇运行异常；
2. 检索设备散热结构和维修依据；
3. 匹配历史散热故障案例；
4. 生成诊断结论与检修方案。

### 6.5 卡片生成原因、条件分支和逐字段推荐

每张卡片出现前，Agent 区按固定结构输出：

```text
已有信息：已确认站控柜 A01，现场描述包含温度与风扇异常
信息缺口：缺少设备型号和关联告警
参考依据：设备台账、设备类型索引、当前现场材料
生成任务：补充设备信息
```

卡片标题下方同时显示一句“生成原因”，保证 Agent 文案和业务控件之间有可见因果关系。

R0 第一版只实现少量、可控的条件分支：

- 风扇状态改为“完全停转”时，追加“蜂鸣器状态”确认项，并把故障方向调整为风扇停转优先。
- 系统温度被确认正常但 TEMP/FAN 仍告警时，追加“告警是否持续/是否复位”确认项。
- 其他情况继续默认设备卡 → 告警卡 → 摘要路径。

自动推荐按字段逐项填入，不得一次性全部完成。字段旁显示来源类型，例如“台账匹配”“知识库推荐”“手册依据”，但不虚构精确概率。

## 7. 卡片视觉与动画规范

### 7.1 卡片状态

| 状态 | 视觉要求 |
| --- | --- |
| 未生成 | 不渲染、不占位 |
| 正在生成 | 对应槽位出现轻量骨架或细线扫描，不显示空白大卡 |
| 当前操作 | 清晰边框、高对比标题、轻微强调色，不使用发光渐变 |
| 已完成 | 收缩为摘要条，显示完成图标和“已完成” |
| 重新编辑 | 恢复展开，保持原值 |
| 校验失败 | 字段旁显示具体缺失项，不使用全局错误弹窗 |

### 7.2 动画节奏

- 卡片浮现：`280ms - 420ms`，淡入并向上移动 `8px - 12px`；
- 卡片收缩：`220ms - 320ms`；
- Agent 字符流：延用当前接诊 Agent 的逐字符输出方案；
- 每条检索依据间隔：`300ms - 600ms`；
- Agent 完成到卡片出现：延迟约 `400ms - 600ms`；
- 自动识别字段填充：每项间隔 `350ms - 600ms`；
- 不使用弹跳、旋转卡片、蓝紫渐变和大面积光效。

### 7.3 工业风要求

- 复用当前深色工业主题 token；
- 当前操作使用青蓝色，完成状态使用克制的绿色，风险状态使用琥珀色或红色；
- 卡片边界必须清楚，正文和辅助文字满足深色背景可读性；
- 位置图自身较亮时，应使用边框或暗色底座隔离，不能通过整体压暗图片影响识别；
- Agent 区域保持单一连续内容面，不拆成大量小卡片。

## 8. 诊断流程同步规则

左侧诊断流程仍采用渐进生成，不提前显示全部步骤。

| 工作区状态 | 左侧可见节点 |
| --- | --- |
| `ANALYZING_SYMPTOM` | 描述现场现象：处理中 |
| `DEVICE_INFO_REQUIRED` | 描述现场现象：已完成；补充设备信息：当前 |
| `DEVICE_INFO_CONFIRMED` | 补充设备信息：已完成 |
| `ALARM_INFO_REQUIRED` | 新增确认告警状态：当前 |
| `ALARM_INFO_CONFIRMED` | 确认告警状态：已完成 |
| `BUILDING_SUMMARY` | 新增接入信息确认：生成中 |
| `READY_FOR_DIAGNOSIS` | 接入信息确认：当前 |
| `STARTING_DIAGNOSIS` | 接入信息确认：已完成；分析诊断：生成中 |

要求：

- 节点出现时间必须与 Agent 的“已生成下一项任务”一致；
- 生成中的节点不可点击；
- 已完成节点可点击，并将对应摘要卡展开；
- 不允许流程节点跳转到已经删除的旧整页页面。

## 9. 组件拆分建议

在不大改架构的前提下，建议从当前 `InputStage` 中拆出以下组件：

```text
IntakeWorkspace
  ├─ SiteContextVisual
  ├─ DynamicTaskLayer
  │    ├─ DeviceInfoTaskCard
```

职责：

## 11. Sub-Spec 与验收标准

### S0：现场材料选择与展示

**需求**

- 首页提供故障图片和故障视频入口；
- 用户选择的材料随异常描述一起进入动态接诊工作区；
- 中央证据画布显示当前材料，场站位置图作为辅助上下文。

**验收标准**

- [ ] 图片可通过文件选择器选入并立即看到预览；
- [ ] 视频可选入并显示文件名及可播放预览或明确缩略状态；
- [ ] 支持替换、删除和切换当前材料；
- [ ] 没有选择材料时可使用预置样例继续演示；
- [ ] 重新开始或组件卸载时释放对象 URL，无明显内存泄漏；
- [ ] 页面不声称已经完成真实图片/视频识别；
- [ ] 1366×768 下上传入口、中央预览和开始接诊按钮均可操作。

### S1：首页到工作区交接

**需求**

- 首页输入现场现象后，点击“开始接诊”；
- 进入单页接诊工作区；
- 原输入内容完整带入；
- 接诊 Agent 自动运行，不需要第二次点击整理按钮。

**验收标准**

- [ ] 空输入不能开始接诊，并显示明确提示；
- [ ] 有效输入只触发一次 Agent；
- [ ] 进入后中央位置图立即可见；
- [ ] 不出现旧的“第 1/4 步”整页标题；
- [ ] 刷新或重新开始演示时状态可正确重置。

### S2：单页稳定布局

**需求**

- 中央位置图持续存在；
- 动态卡片在预设槽位出现；
- 左右栏和主区域保持独立。

**验收标准**

- [ ] 卡片出现和收缩时位置图不缩放、不跳动；
- [ ] 卡片不遮挡故障点和主要设备区域；
- [ ] 1366×768 下主要操作可见；
- [ ] 1920×1080 下页面不出现过大空白；
- [ ] 窄屏下改为纵向排列，无组件重叠和横向溢出。

### S3：接诊 Agent 动态分析

**需求**

- 自动播放流式分析、检索依据和完成结论；
- 完成后才生成设备信息卡。

**验收标准**

- [ ] 文本采用字符级或短语级流式输出；
- [ ] 至少 4 条专业检修依据逐步出现；
- [ ] 运行中有“正在分析/正在检索”状态；
- [ ] 未完成前设备信息卡不渲染；
- [ ] 文案明确指向下一项“补充设备信息”。

### S4：设备信息动态补充

**需求**

- 设备卡由 Agent 生成；
- 字段初始为空；
- 支持自动识别和手动选择；
- 自动识别逐项填充。

**验收标准**

- [ ] 四个字段均为真实可操作控件；
- [ ] 自动识别期间按钮不可重复点击；
- [ ] 建议值不是瞬间全部出现；
- [ ] 缺少必填项时不能确认；
- [ ] 确认后卡片收缩并显示“已完成”；
- [ ] 点击“修改”能够恢复展开和原值。

### S5：告警状态动态补充

**需求**

- 设备确认后，Agent 继续检索再生成告警卡；
- 用户可以确认或修改灯态和运行状态。

**验收标准**

- [ ] 设备未确认时告警卡完全隐藏；
- [ ] Agent 文案与 ACP-4000 / IPC-610 告警定义相关；
- [ ] TEMP/FAN、风扇、系统温度、CPU 温度均可修改；
- [ ] 主卡片不强制显示复杂数值阈值；
- [ ] 确认后显示紧凑告警摘要；
- [ ] 修改设备信息后，告警建议可以重新计算或重置。

### S6：接入摘要确认页与启动诊断

**需求**

- 所有必要信息完成后才进入独立摘要确认页；
- 用户确认摘要后进入现有诊断流程。

**验收标准**

- [ ] 摘要正确反映用户最终选择，不使用过期默认值；
- [ ] 摘要包含位置、设备、异常、告警和诊断任务；
- [ ] “启动智能诊断”只在 `READY_FOR_DIAGNOSIS` 可用；
- [ ] 点击后显示 loading 且不能重复触发；
- [ ] 能正常进入当前多 Agent 分析诊断链路；
- [x] 摘要页点击“启动智能诊断”能够进入多 Agent 会诊（WSL 浏览器已验证）。

### S7：流程节点同步

**需求**

- 左侧流程节点与 Agent 和卡片状态一致。

**验收标准**

- [ ] 初始只显示“描述现场现象”；
- [ ] Agent 完成后才出现“补充设备信息”；
- [ ] 设备确认并完成检索后才出现“确认告警状态”；
- [ ] 全部确认后才出现“接入信息确认”；
- [ ] 已完成、当前、生成中三种状态清晰；
- [ ] 生成中的节点不能打断当前流程。

### S8：动画、可访问性与稳定性

**需求**

- 动画用于说明状态变化，不影响操作稳定性。

**验收标准**

- [ ] 动画结束后没有残留遮罩或不可点击区域；
- [ ] 所有表单具有可识别标签；
- [ ] Agent 输出区域使用 `aria-live="polite"`；
- [ ] 支持 `prefers-reduced-motion`，关闭非必要位移动画；
- [ ] 所有 timeout/interval 在卸载和重置时清理；
- [ ] 不新增 npm 依赖。

## 14. 完成定义

R0 只有同时满足以下条件才算完成：

1. 从首页提交后进入一个动态接诊工作页面，全部确认后进入独立接入摘要确认页；
2. 首页可以选择故障图片/视频，中央证据画布在整个接入阶段持续展示已选材料；
3. 接诊 Agent 自动流式分析，并在每张卡片出现前说明信息缺口与生成原因；
4. 设备卡和告警卡由 Agent 分阶段生成，而非初始全部展示；
5. 自动推荐逐字段出现并显示来源类型；
6. 用户可以真实修改和确认生成内容；
7. 至少一个修改结果能够触发预制的额外补充任务或重新评估；
8. 已完成卡片收缩保留，当前卡片突出显示；
9. 独立接入摘要页按原始描述、整理结果和诊断任务分层生成；
10. 左侧流程同步渐进出现；
11. 点击“启动智能诊断”可以稳定进入现有诊断流程；
12. 无组件重叠、明显布局跳动、重复计时器、对象 URL 泄漏或新增依赖。
