# 检修知识库与工控机知识图谱内容扩充 Spec

> 版本：V1.2
> 日期：2026-07-14
> 开发分支：`feature/knowledge-content-expansion`
> 技术栈：React 18 + Vite 6 + Flask + SQLite + JSON 种子
> 资料范围：`Info/数据` 与 `Info/acp4000_cooling_kb` 下 15 份 Rockwell Automation、Schneider Electric 与 Advantech 官方手册

## 1. 背景

当前系统已经具备检修知识库、全局知识图谱、专家发布、工程师同步和知识验证能力，但内容仍以散热异常为主：

- 检修知识库共有 8 个条目，只有 `KB-008` 具备完整内容和动态版本链路；
- 全局知识图谱共有 38 个节点、40 条关系，环境、供电、硬件、运维和监控五个领域已经成形，但非散热故障深度不足；
- SQLite 负责唯一完整案例的发布事务与工程师同步，官方手册知识仍以 JSON 种子形式维护；
- 两批 15 份官方手册合计约 939 页：第一批补充启动、固件、显示、存储、电池、通信、系统恢复和监控告警，第二批补充 ACP-4000 / IPC-610 散热、告警、滤网、风扇与智能系统模块的型号级依据。

本项目不把整本 PDF 直接作为知识条目，也不把官方维护说明伪装为真实检修案例。手册先登记为证据来源，再加工为结构化检修知识，最后抽取稳定关系进入全局知识图谱。

## 2. 目标

### 2.1 本阶段目标

1. 建立官方资料来源登记，明确厂家、产品范围、版本、页数和适用边界；
2. 将首批 9 条高价值手册知识加入检修知识库；
3. 将首批知识对应的故障、现象、原因、检查和处置节点加入全局知识图谱；
4. 所有新增知识都能够追溯到具体文档和 PDF 页码；
5. 保持“散热异常”为唯一完整动态案例闭环；
6. 保持“全局知识图谱 / 本次新增与修改”两个正式视图；
7. 不破坏专家发布、工程师本地同步和知识验证流程；
8. 为后续知识详情页、筛选和批量内容扩充提供稳定的数据模型。

### 2.2 非目标

本阶段不实现：

- PDF 全文检索或向量数据库；
- 自动从 PDF 无审核生成正式知识；
- 新增真实案例或改变历史案例状态；
- 将批量手册节点计入当前散热案例的“本次新增与修改”；
- 引入图数据库、D3、ECharts、React Flow 或其他新依赖；
- 把跨品牌资料中的型号参数套用到目标设备。

## 3. 为什么分阶段实施

内容扩充涉及三类不同风险：

1. **来源风险**：同为工控机，不同厂家和型号的电压、端子、报警阈值与拆装步骤并不通用；
2. **数据风险**：知识库适合保存完整检修步骤，图谱只适合保存稳定关系，不能一份内容复制两遍；
3. **展示风险**：现有固定 SVG 布局适合约 60 个节点，一次加入数百个手册概念会导致页面不可读。

因此按以下阶段实施：

| 阶段 | 内容 | 完成标志 |
| --- | --- | --- |
| 第一阶段 | 来源登记、首批知识、全局图谱基线扩充 | 数据可追溯，现有页面可读取，构建与测试通过 |
| 第二阶段 | 知识详情页、筛选、来源页码展示、图谱联动 | 用户可以从知识卡片查看完整内容并跳转图谱节点 |
| 第三阶段 | 更多手册主题、轻量演示案例、专家内容维护 | 内容经过人工复核，可由专家新增或修订 |
| 第四阶段 | 半自动提取、全文检索或向量检索 | 仅在内容规模和演示需求明确增长后实施 |

## 4. 数据源与事实等级

### 4.1 来源登记

新增 `backend/data/presentation/manual_sources.json` 作为官方资料目录。每份资料至少包含：

```json
{
  "id": "DOC-ROCKWELL-6177R-UM002F",
  "manufacturer": "Rockwell Automation",
  "publication": "6177R-UM002F-EN-P",
  "documentDate": "2018-05",
  "language": "en",
  "pageCount": 100,
  "file": "Info/数据/Rockwell-6177R 机架式工控机用户手册.pdf",
  "productScope": ["6177R 750R", "6177R 1450R"],
  "referencePriority": "exact_model"
}
```

### 4.2 适用范围

| 值 | 含义 | 允许使用的内容 |
| --- | --- | --- |
| `exact_model` | 目标型号完全一致 | 参数、端子、灯态、步骤和安全要求 |
| `same_series` | 同厂家相近系列 | 通用原理和相近配置，必须复核型号 |
| `cross_brand_general` | 其他厂家工控机 | 通用故障机理、检查方法和维护策略 |
| `engineering_inference` | 项目工程推导 | 必须显式标为推导，不得写成厂家规定 |

### 4.3 验证等级

- `official_document`：已经在厂家官方手册中核实；
- `expert_confirmed`：由专家确认的项目规则或操作要求；
- `verified_case`：由真实完整案例证据支持；
- `collecting`：资料已整理但仍缺少现场证据；
- `presentation_seed`：仅用于演示结构，不承担型号参数判断。

首批新增知识统一使用 `official_document`。它们可以进入知识库和全局图谱，但不会成为动态回流案例。

## 6. 知识图谱扩充模型

### 6.1 保持五个一级领域

本阶段不增加一级领域，避免破坏已经验收的全局布局：

- 环境工况异常；
- 供电系统故障；
- 硬件部件失效；
- 安装与运维不规范；
- 监控告警失效。

启动、固件、显示、通信和系统恢复作为二级或三级节点进入现有领域。后续只有在节点数量超过固定布局容量时，才评估增加一级领域或语义缩放。

### 6.2 新增节点类型

继续使用现有类型，并补充实际已经被渲染层兼容的语义：

- `fault_category`：启动、显示、通信、存储等故障类别；
- `fault_mode`：黑屏、POST 致命错误、日期时间丢失等表现；
- `fault_cause`：驱动冲突、配置损坏、外设故障等原因；
- `action`：断开外设、进入诊断、恢复默认值等动作；
- `alarm`：硬盘、网络、风扇、电压和系统告警；
- `recovery`：POST 完成、显示稳定、系统恢复等结果；
- `knowledge`：指向检修知识库条目。

### 6.3 关系词汇

本阶段关系限定为：

```text
包含、表现为、可能原因、可能导致、检查、监测、触发、指示、
优先检查、建议处理、要求、恢复至、来源于、适用于、关联知识
```

新增关系必须满足：

- source 和 target 节点真实存在；
- 关系 ID 唯一；
- 不使用同义但拼写不同的关系词；
- 节点详情中的结论能够通过 `sourceRefs` 追溯；
- 官方手册基线节点不带 `changeType`，不进入本次动态变更视图。

### 6.4 规模控制

第一阶段目标：

```text
知识条目：8 → 17
图谱节点：38 → 55～60
图谱关系：40 → 70～90
动态完整案例：仍为 1
```

## 7. 前后端兼容策略

### 7.1 后端

- `GET /api/admin/knowledge` 继续返回 `knowledge_base.json` 中的条目；
- `GET /api/admin/knowledge-graph` 继续读取 `industrial_computer_graph.json`；
- 新字段作为 JSON payload 返回，不修改现有接口字段；
- SQLite 中 `KB-008` 的 V1.0/V1.1 发布链路保持不变；
- 一键重置继续只重建动态案例与图谱种子。

### 7.2 前端

第一阶段复用现有页面：

- 知识库自动显示新增知识卡片；
- 全局图谱自动显示新增节点和关系；
- 节点详情继续显示类型、领域、状态、来源和关联关系；
- “本次新增与修改”只显示当前散热案例的动态节点和关系。

第二阶段再实现：

- 静态知识详情页；
- 按故障域、设备和验证等级筛选；
- 手册页码与适用范围可视化；
- “在图谱中查看”和“查看关联知识”双向跳转。

## 8. 内容加工流程

```text
登记文档
  → 确认型号与版本
  → 定位故障章节和页码
  → 提取症状、原因、检查、动作、安全和恢复标准
  → 标记适用范围
  → 人工复核
  → 写入知识条目
  → 抽取稳定图谱关系
  → 数据校验
  → 页面验收
```

禁止：

- 根据关键词命中直接生成正式结论；
- 将维修建议扩写成手册未给出的量化阈值；
- 用演示构造数据替代真实现场证据；
- 为了让图谱更大而加入无详情、无来源的装饰节点。

## 9. 验收标准

### 9.1 内容验收

- 15 份资料均有唯一文档 ID；
- 每条新增知识至少关联 1 份文档和具体页码；
- 所有页码在对应 PDF 页数范围内；
- 型号参数、灯态和操作步骤不存在跨型号误用；
- 官方知识、跨品牌参考、工程推导和真实案例能够区分。

### 9.2 数据验收

- 知识 ID、文档 ID、节点 ID 和关系 ID 均唯一；
- 所有关系端点存在；
- 新增知识必填字段完整；
- 图谱不存在新增孤立节点；
- 动态 `changeTemplate` 的 5 个节点和 6 条关系保持不变。

### 9.3 功能验收

- 检修知识库显示新增卡片；
- 全局知识图谱显示新增故障簇；
- 点击新增节点可以查看详情和一跳关系；
- 专家端与工程师端读取同一全局图谱；
- “本次新增与修改”仍只聚焦散热动态案例；
- 专家发布、工程师同步和知识验证流程不回归。

### 9.4 技术验收

- JSON 可解析；
- 内容数据自动化测试通过；
- 后端现有测试通过；
- Node 20.19.4 前端构建通过；
- 浏览器无控制台错误；
- 1366px 与 1920px 下全局图谱仍可操作和阅读。

## 10. 后续阶段入口

第一阶段完成后，第二阶段优先开发静态知识详情页和知识—图谱双向跳转。是否继续批量增加内容，应以以下条件为准：

1. 第一批知识的字段和页面展示已经稳定；
2. 全局图谱在演示分辨率下仍然清晰；
3. 已明确下一批优先故障域；
4. 对应手册章节完成复核；
5. 若要加入案例，已经取得真实维修记录或明确标记为“演示构造 / 资料收集中”。

## 11. 第二阶段实施：资料型知识进入检修知识库

### 11.1 产品结构

检修知识库明确分为两类内容，不把原始手册伪装成案例：

| 视图 | 内容 | 用户主要操作 |
| --- | --- | --- |
| 知识条目 | 真实案例经验、结构化检修方法、告警规则 | 搜索故障、查看完整步骤、核对来源、定位图谱 |
| 检修手册 | 厂商原始 PDF、型号范围、版本和覆盖领域 | 按厂商/型号检索、查看关联知识、打开原始 PDF |

两类内容使用以下证据链关联：

```text
厂商检修手册 + PDF 页码
  → 人工复核后的结构化知识条目
  → 全局知识图谱中的故障/原因/检查/动作节点
```

### 11.2 本阶段接口

- `GET /api/admin/manuals`：返回手册目录、关联知识数量和受控文件地址；
- `GET /api/admin/manuals/<manual_id>`：返回单份手册详情；
- `GET /api/admin/manuals/<manual_id>/file`：只允许打开目录中登记且位于仓库内的 PDF；
- `GET /api/admin/knowledge`：继续返回结构化知识，前端展示症状、原因、检查、处置、安全和恢复标准；
- `GET /api/admin/knowledge-graph`：根据 `knowledgeId` 定位并放大关联节点。

### 11.3 展示与交互

1. 页面顶部显示知识条目、厂商手册和已引用资料数量；
2. “知识条目 / 检修手册”双视图分别保留自己的筛选维度；
3. 点击知识条目打开右侧详情，展示适用范围、结构化步骤和手册页码；
4. 点击来源可在新窗口直接打开原始 PDF；
5. 点击“在知识图谱中定位”进入全局图谱并放大该知识关联节点；
6. 点击手册可查看型号范围、检修领域和已经加工出的知识；
7. `KB-008` 仍提供案例版本变化入口，其他手册知识保持只读。

### 11.4 分阶段继续扩充

本阶段完成的是“容器和第一批内容”。后续资料应按小批次加入：

1. 优先补充已有手册中尚未结构化的供电、内存、LED/蜂鸣、维护拆装等高价值章节；
2. 每批先补手册目录和页码，再补知识条目，最后抽取图谱关系；
3. 只有拿到真实工单、现场照片、测量记录和恢复结果后，才新增“真实案例”；
4. 没有真实案例但适合演示的故障，只标为资料知识或演示种子；
5. 固定图谱接近可读上限后，优先按领域聚焦和语义缩放，不继续无边界堆节点。

### 11.5 第二批 Advantech 散热专项资料

`Info/acp4000_cooling_kb` 作为完整的散热专项知识包保留，包含项目整理稿、抽取稿、资料清单、下载脚本和 8 份官方 PDF。资料分层如下：

| 资料层 | 数量 | 用途 |
| --- | ---: | --- |
| 当前官方资料 | 5 | Ed.6 合并用户手册、三份当前规格书、SAB-2000 Ed.2；作为当前散热知识的正式来源 |
| 历史官方资料 | 3 | Ed.5 合并手册、ACP-4000 Ed.1、IPC-610-H Ed.1；只用于旧设备和版本差异核对 |
| 项目整理资料 | 3 | 故障检修指南与两份章节抽取稿；用于内容加工，不伪装成厂家原文 |
| 资料管理文件 | 2 | 文档清单和官方 PDF 下载脚本 |

第二批资料不重复创建新的“散热案例”。它们用于补强现有 `KB-001`～`KB-009` 的来源，并为散热图谱节点补充可追溯页码：

- Ed.6 PDF 物理页 39 对应手册印刷页 23，包含风扇 `<500 rpm`、系统温度 `>55°C`、CPU 温度 `>70°C` 与 Alarm Reset 三分钟静音；
- Ed.6 PDF 物理页 32～33 对应风扇和滤网更换；
- Ed.6 PDF 物理页 40～41 对应 FAN 接线顺序与 Smart Fan/传感器配置；
- 当前规格书用于确认风扇数量、风量、滤网尺寸与备件号；
- 历史版本在界面中必须显示“历史版本”提示，不得作为当前设备操作参数的默认依据。
