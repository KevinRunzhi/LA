# 软件杯 A1 赛题最终方向分析报告
## 基于官方赛题原文的重新校准（截止 7 月 20 日）

> 赛题：A1 - 基于多模态大模型技术的设备检修知识检索与作业系统（出题企业：龙芯中科）
> 硬性约束：B/S 架构，部署于 LoongArch + 银河麒麟高级服务器版（不满足 = 0 分）；目标机 4 核 / 8GB / 256GB
> 评分：功能完整性 30% / 用户体验 20% / 创新与实用 20% / 文档与演示 20% / 商业可行性 10%

## 一、基于官方赛题原文后的重新判断

官方原文里有五个改变判断的关键信号：

**信号 1：官方业务场景明确锚定"钢铁、汽车制造等行业的产线设备"**，并强调"非计划停机单次损失超百万元"。答辩叙事必须站在**制造企业**立场讲，"电网公司配电运维"的叙事框架有偏题风险——设备可以不变，但**场景的主语必须换**。

**信号 2：官方同时写明"检修设备：不限，可自行发挥"**。这是免死金牌——选什么设备不扣分，扣分的是**叙事没接上制造业痛点**。问题不是"配电设备行不行"，而是"配电设备的故事怎么讲"。

**信号 3：官方提供的参考手册是《摩托车发动机维修手册.pdf》**。这是最容易被忽视、但价值极高的信号：官方给了一份现成的、评委人手一份的标准知识库素材。**系统如果能直接导入这份官方 PDF 并跑通问答，等于当着评委的面用他们自己的数据验证了系统**——最低成本、最高可信度的"通用性证明"，必须利用。

**信号 4：评分结构改变游戏规则**。功能完整性 30% + 文档与演示 20% + 用户体验 20% = 70% 的分数在"官方要求的功能全不全、文档规不规范、界面好不好用"，只有 20% 在创新。**"官方功能清单逐条对齐"比"某一项技术做得炫"更重要**。官方功能清单里藏着两个大多数队伍会漏掉的必做点：**"合规校验提醒"和"根据设备类型与检修等级个性化推送流程"**。

**信号 5：LoongArch + 麒麟部署不满足 = 0 分，且目标机只有 4 核 / 8GB**。这直接判了"本地部署大模型/向量库/图数据库"死刑。同时官方规则第 2 条明确写了"支持本地部署大模型服务**或云端大模型服务**"——**云端大模型是官方明文允许的**，架构合规性没有争议。

## 二、上一轮结论中哪些仍然成立，哪些需要修正

### 仍然成立

1. 赛题本质是"可信（溯源）+ 可执行（作业指引）+ 可进化（审核回流）"，不是聊天机器人——官方功能三大块（多模态检索 / 标准化作业指引 / 知识沉淀更新）与此完全对应。
2. 收窄设备与故障范围、不做大而全平台。
3. 本地轻、云端重的部署分层。
4. "专家修正回流"是差异化王牌——官方原文"经审核后纳入知识图谱""支持手动标注与修正大模型输出结果"证实它是**必做项**。
5. 配电设备在"图片直观、素材易得、图谱丰满"上的优势依然真实存在。

### 需要修正

1. **场景主语必须从"电网公司/供电局"改为"制造企业（钢铁/汽车厂）设备动力部门"**。设备不用大改，叙事必须重构。
2. **必须新增"官方摩托车手册"作为第二知识库**，用于演示系统通用性和文档导入能力。
3. **知识图谱从"锦上添花"升级为"官方点名要求"**（审核后内容要"纳入知识图谱"）——图谱必须真实存在（作为数据结构），但依然不需要图数据库。
4. **新增两个必做功能：合规校验提醒、按设备类型与检修等级的个性化流程推送**。
5. **文档与演示占 20%**：7 份文档 + 7 分钟视频必须进排期，至少预留 4~5 天。
6. **部署演练升级为第零优先级**——0 分条款面前，一切功能都排在"先在龙芯云虚拟机上跑通 Hello World 全链路"之后。

### 对疑问 7.1 的逐条回答

1. 选"纯电网配电设备"确实有偏离制造业背景的风险——不是设备偏，是立场偏。
2. "制造企业厂区配电设备"明显优于"纯电网配电设备"，改造成本几乎为零（设备、故障、知识库全部复用，只换企业背景和用户角色）。
3. 完全可以理直气壮地解释：钢铁厂是典型高耗能连续生产企业，**厂区供配电系统一旦故障，全产线非计划停机**——这正是官方"非计划停机单次损失超百万元"的最直接成因之一。轧钢线、连铸机、汽车焊装线全部依赖厂区 10kV/0.4kV 供配电。
4. "不够生产设备"的质疑存在可能，但可彻底化解。
5. **化解叙事三板斧**：
   - 开场用"某钢铁厂电缆接头过热引发母线停电、连铸中断损失数百万"式事故案例定调，把配电定义为"产线动力命脉"而非"辅助设施"。
   - 设备范围加入 1 类真正的产线设备——**低压异步电机**（产线传送/风机/泵用电机），只做 1~2 个故障（轴承过热、绕组烧毁，红外图极直观），从设备清单上堵死质疑。
   - **接入官方摩托车发动机手册作为第二知识库**，证明系统是通用平台——"我们选配电做深度示范，但系统天生服务全部产线设备"。

## 四、最终推荐的行业与业务切入点

**结论：保留配电方向，完成"三个重新包装"。**

**项目定位（命名方向）：**

> **面向制造企业的多模态设备检修智能助手**——以钢铁/汽车制造企业**厂区供配电与动力设备**为核心示范场景，支持任意行业设备手册即插即用。

主标题不出现"配电"二字（配电只是示范场景，不是产品边界），副标题体现制造业。可另起短名（如"慧检"式产品名）+ 上述副标题。

**场景设定**：制造企业（建议"某大型钢铁联合企业"或"某汽车整车厂"）**设备动力部 / 机电维保班组**。
用户角色：一线维保电工（新员工）+ 班组技术专家（审核者）。
不建议设定为电网公司（偏题），也不建议第三方运维服务商（商业故事复杂化，弱化"企业知识沉淀"主线）。

**双知识库策略（本轮新增核心策略）：**

- 知识库 A（主演示）：厂区配电与动力设备检修知识库（自建：规程 + 案例 + 图谱）。
- 知识库 B（通用性证明）：**官方《摩托车发动机维修手册.pdf》原样导入**，现场演示"换一个行业的手册，系统照样能检索问答"。同时覆盖：文档解析功能演示、系统通用性证明、"评委用自己的数据验证你的系统"的信任红利。

## 五、最小设备范围和故障范围

**设备 4 类（3 类配电 + 1 类产线动力，不再多）：**

1. 高压开关柜（10kV）
2. 电缆及电缆接头
3. 干式配电变压器（厂房内常见，比油变更贴合制造车间）
4. 产线低压异步电机（衔接"生产设备"的关键一类）

**核心故障 7 个（全链路做深）：**

| 设备 | 故障 | 多模态素材 |
|---|---|---|
| 开关柜 | 触头/母排过热 | 红外热像图（素材丰富） |
| 开关柜 | 局部放电 | 柜体放电痕迹照片 + 描述 |
| 电缆接头 | 接头发热 | 红外图（最经典演示） |
| 电缆接头 | 绝缘老化/表面放电 | 外观照片 |
| 干式变压器 | 温控告警/绕组超温 | 温控器面板读数照片 |
| 电机 | 轴承过热/异响 | 红外图 + 文本描述 |
| 电机 | 绕组烧毁 | 拆解外观照片 |

**初赛不适合做的故障**：局放定量分析、油色谱诊断、继电保护定值校验、谐波分析——专业计算多、讲错风险高、实现成本高。保护装置降级为知识库条目（可问答，不做识别、不做重点演示）。

## 六、最小可交付闭环

```text
① 知识库管理：导入配电知识库（预构建）+ 现场导入官方摩托车手册 PDF
② 多模态输入：拍照/上传图片 + 文本描述 + 设备型号选择
③ 多模态识别：云端多模态大模型 API 识别设备/现象/读数（识别失败回退人工选择）
④ RAG 检索问答：向量检索 → 证据生成 → 每条结论挂"来源规程原文"溯源卡片
⑤ 知识图谱：设备—部件—故障—原因—检测—处置 关系图，前端可视化联动
⑥ 标准化作业指引：按"设备类型 + 检修等级(日常/一级/二级)"个性化生成作业卡
   ├ 步骤化操作指引（可逐步勾选）
   └ 合规校验提醒（停电/验电/挂牌/接地等安全强制项 checklist）★官方点名
⑦ 专家审核：技术专家修改 AI 输出、标注对错、审核案例
⑧ 知识回流：审核通过内容 → 重新向量化入库 + 纳入图谱 → 下次同类问题命中修正版
```

**7 分钟视频叙事骨架**：30 秒痛点（钢厂停电事故）→ 1 分钟系统总览 → 3 分钟主线（拍红外图 → 识别 → 溯源问答 → 图谱 → 作业卡带合规提醒）→ 1 分钟专家修正回流前后对比 → 1 分钟导入官方摩托车手册证明通用 + 龙芯部署画面收尾。

## 七、必须真做 vs 可预置 vs 可模拟

### 必须真实实现（评分 30% 的骨架 + 题眼）

1. **多模态识别**：必须真实调用云端多模态大模型 API（赛题标题即"多模态大模型技术"）。做"确定性兜底"：预置演示图片走稳定 prompt，置信度低时回退人工选择。
2. **RAG 检索 + 溯源**：真实向量检索、真实生成、真实引用原文。
3. **专家修正真实影响后续检索**：官方"优化系统适配性"的字面要求，也是最强演示桥段（修正前答错 → 回流 → 再问答对）。技术上 = 审核后重嵌入 + 检索优先级。
4. **按设备类型 + 检修等级的个性化作业指引、合规校验提醒**：官方点名必做，实现可轻（规则表驱动）。
5. **官方 PDF 手册导入**：评委可能拿它验证。

### 可以预置数据实现

- 配电知识库全部内容（离线整理好，结构化入库）。
- 知识图谱：**不必用图数据库**。SQLite 三元组表 + 前端 ECharts 关系图渲染——官方只要求"纳入知识图谱"这个逻辑结构存在。
- 演示用例的图片与预期输出（答辩走确定性脚本）。

### 可以轻量模拟

- 文档解析范围：PDF 用纯 Python 的 pypdf 抽文本（龙芯无编译风险）；Word 支持可砍或转 PDF；扫描件 OCR 不做本地引擎（如需调云端 OCR API）。
- 作业卡生成：**模板 + RAG 内容填充**，不让模型自由生成整卡。模板保证格式永远专业稳定（步骤/工器具/安全措施/验收标准四段式），模型只填充内容槽位。

### 初赛不建议做

用户权限体系（写死两个角色即可）、移动端 App（Web 响应式即可）、实时传感器接入、报表大屏、多租户、本地部署任何模型。

## 八、创新点优先级

### 第一梯队（性价比最高，全做）

1. **专家修正驱动的"知识进化"现场演示**——回流前后对比，7 分钟视频最强记忆点，同时得"功能完整性"和"创新"两处分。
2. **可信溯源卡片**——直击大模型幻觉痛点，评委必问必看。
3. **行业知识库即插即用（双库演示）**——用官方摩托车手册证明通用性；商业可行性 10% 的分也在这里（"一套平台服务多行业"即商业故事）。
4. **合规校验提醒做成"安全强制项不可跳过"的交互**——把官方要求做出体验亮点（用户体验 20%）。

### 第二梯队（有余力再做）

5. 图谱驱动根因追溯（点故障高亮因果链）。
6. 检修等级个性化推送做成可配置规则。

### 听起来好但不要碰

- 自训练/微调视觉模型提升识别准确率——投入巨大、无法现场验证、与 4 核 8GB 冲突。
- 图数据库 + 图算法——部署风险换不来对应分数。
- 本地部署大模型"体现国产化"——8GB 跑不出能看的效果；国产化叙事用"龙芯跑应用 + 国产云大模型（Qwen/DeepSeek）"更稳，且官方明文允许云端。

### 对关键疑问的直接回答

- "知识图谱 + 专家修正回流" **远比** "多模态识别准确率"值得投入：前者是官方功能清单字面要求（30% 分区），后者官方从未要求准确率指标。
- 本赛题评委更看重**功能全 + 闭环通**，单点技术深度只有在功能齐全之后才加分。
- 避免"普通 RAG demo 感"三件套 = 溯源卡片 + 作业卡 + 回流演示。

## 九、LoongArch 部署约束下的技术取舍

**总原则：目标机上只允许出现"发行版仓库自带的 + 纯 Python 的"东西。任何需要现场编译 C/C++/Rust 的依赖都视为事故。**

### 推荐架构（最稳版）

```text
[LoongArch + 银河麒麟 V10/V11]（4核/8GB）
 ├─ Nginx：托管前端静态文件 + 反向代理（系统仓库安装）
 ├─ Python3（系统自带）+ venv
 │   └─ Flask + Jinja2 + requests + pypdf + PyMySQL(如用MariaDB) —— 全纯Python
 ├─ SQLite（Python 内置，零部署）：业务数据 + 知识条目 + 图谱三元组 + 向量(JSON/BLOB)
 └─ systemd 托管服务 + 日志
        │  HTTPS（仅出方向 requests 调用）
        ▼
[云端]
 ├─ 多模态大模型 + 文本大模型：阿里云百炼(Qwen-VL/Qwen) 或 DeepSeek —— 官方点名允许
 ├─ Embedding API：云端向量化
 └─ （可选）云数据库/云向量库 —— 小知识库其实不需要
```

### 三个激进但正确的取舍

1. **数据库用 SQLite 而不是 MySQL**。知识库几百条、演示并发个位数，SQLite 绰绰有余且零部署风险。银河麒麟 loongarch64 软件源没有自带 MySQL 安装包，需要源码编译（且要先卸载系统自带 MariaDB）——比赛期间在 4 核机上源码编译 MySQL 是自找灾难。如文档层面希望体现"数据库"，写 SQLite 或用系统仓库自带的 MariaDB。
2. **向量检索不用向量库**。几百个知识块，云端 API 算 embedding，本地存 SQLite/JSON，纯 Python 算余弦相似度，几十毫秒出结果。FAISS/Milvus 在 LoongArch 上属高风险编译件，收益为零。
3. **调大模型不用官方 SDK，用 requests 直连 HTTP API**。很多大模型 SDK 依赖 pydantic v2（核心 pydantic-core 是 Rust 编译件）或 grpcio 等二进制依赖，loongarch64 上无预编译 wheel 就会触发现场编译。requests 是纯 Python，永远装得上。同理**后端框架首选 Flask 而非 FastAPI**（FastAPI 强依赖 pydantic v2）。

## 十、龙芯 LoongArch 平台生态调研结论

### 背景事实：新旧世界之分

LoongArch 存在互不兼容的"旧世界/新世界"两套生态。系统为 Kylin（银河麒麟）、Loongnix 或 UOS、内核 4.19 开头的属旧世界（ABI 1.0）；新世界（ABI 2.0）是与上游开源社区共建的生态，两者二进制不兼容（旧世界软件需 libLoL 兼容层才能在新世界运行）。此外还存在 loongarch64 与 loong64 架构标记之别。

**实际含义**：网上找到的"loongarch64 预编译二进制"未必能在比赛的麒麟系统上跑。**一切以"官方龙芯云虚拟机上、系统自带仓库里有什么"为准**。银河麒麟 V10 LoongArch 版属旧世界；V11 较新，拿到环境后第一时间 `uname -r` 确认内核与 ABI 状态。

### 逐项结论

1. **生态总体**：服务器基础组件（gcc、Python3、Nginx、MariaDB、OpenJDK）在麒麟/龙芯仓库基本齐全可用；第三方生态显著弱于 x86/ARM，越靠近 AI/二进制依赖越危险。
2. **常见 Web 部署**：Nginx + Python 后端 + 静态前端在麒麟龙芯上是被大量国产化交付验证过的稳定组合，风险低。
3. **Python**：系统自带 Python3 可用，纯 Python 包安装无碍；含 C 扩展的科学计算包是重灾区（scikit-learn、matplotlib 等装上后无法正常使用、需源码编译且编译常因 gcc 问题中断）。策略：不碰需要编译的包；如确需 numpy 优先用系统仓库 `python3-numpy`。
4. **Node.js**：官方只支持 x64/ARM，LoongArch 需源码自行编译且可能要改源文件、成功率不稳。结论：目标机上不跑 Node，Node 只存在于 Windows 开发机用于构建前端。
5. **最易出问题的依赖类型**：C/C++ 扩展（numpy/scipy/opencv/lxml/psycopg2）、Rust（pydantic-core、cryptography 新版、tokenizers）、grpc（grpcio、部分云 SDK）、Electron/Chromium 系。
6. **框架**：Flask 最稳（Flask/Jinja2/Werkzeug 全纯 Python）；Django 纯 Python 但偏重；FastAPI 因 pydantic v2 的 Rust 依赖列为"先验证再用"。
7. **前端**：Vue/React 一律 Windows 上 `npm run build`，只把 `dist` 拷到龙芯由 Nginx 托管。目标机零 Node 依赖。
8. **Docker**：避免。龙芯上 Docker 可装，但 Docker Hub 绝大多数镜像无 loong64 版本，会陷入"自己在龙芯上 build 镜像"的深坑。裸机 systemd 部署。
9. **数据库驱动**：首选 SQLite（Python 内置）；若用 MariaDB 选 PyMySQL（纯 Python），不用 mysqlclient（C 扩展）；psycopg2 需编译不推荐。
10. **向量库/图数据库**：完全避免本地部署。云化或 SQLite + 纯 Python 计算替代。
11. **本地部署大模型**：4 核 / 8GB 不现实，即使 llama.cpp 有 LoongArch 优化也只能跑玩具级小模型。官方明文允许云端，用云。
12. **OCR/OpenCV/PyTorch/ONNX/FAISS/Milvus/Neo4j**：全部高风险或不可用（无官方 loongarch wheel/二进制，源码编译周期长、成功率低）。全部列入"不建议使用"清单。
13. **"云端重能力 + 本地只跑 Web/API"**：是唯一建议架构，且合规无忧——官方规则第 2 条明确允许云端大模型服务；第 1 条只要求软件部署在 LoongArch + 麒麟上运行，B/S 应用本体（前端 + 后端 + 数据 + 业务闭环）全在龙芯上即满足。
14. **最适合"Windows 开发 + LoongArch 最小部署"的技术栈**：Vue3（Windows 构建）+ Nginx + Flask（纯 Python）+ SQLite + requests 直连云 API。
15. **降低迁移风险的项目结构**：requirements.txt 只允许纯 Python 包（进检查清单）；云端点/密钥/路径集中在单一 config；前后端彻底分离（后端只出 JSON API）；systemd unit 而非裸启动脚本；一键部署脚本 deploy.sh（建目录、装 venv、离线装 wheels、拷 dist、起服务）。
16. **提前验证方法**：拿到龙芯云虚拟机的第一天完成"空壳部署演练"——`uname -r` 确认系统；系统源装 Nginx；python3 建 venv；离线装 Flask；跑通一个 hello API；Nginx 托管一个静态页；requests 调通一次云端大模型。此链一通，项目不会 0 分。此后每周重新部署一次最新代码。
17. **兜底方案**：所有 Python 依赖在 Windows 上 `pip download` 打离线包随代码提交（纯 Python wheel 全平台通用），防比赛环境网络受限；疑难 PDF 离线解析后以结构化数据导入；识别失败回退人工选择；云 API 断网启用预置演示应答缓存（仅演示兜底，文档如实说明）。

### 技术清单汇总

**可用清单**：Nginx（仓库）、Python3 + venv（系统自带）、Flask/Jinja2/Werkzeug、requests、pypdf、PyMySQL、SQLite（内置）、MariaDB（仓库）、systemd、ECharts（前端静态资源）、Vue3（仅 Windows 构建）。

**高风险清单（先验证再用）**：FastAPI/pydantic v2、numpy（仅限系统仓库版）、任何大模型官方 SDK、Django（重）、psycopg2。

**不建议使用清单**：本地大模型、FAISS、Milvus、Neo4j、PyTorch、ONNX Runtime、OpenCV、OCR 本地引擎、Docker、Node.js（目标机）、MySQL 源码编译、Electron。

## 十一、Windows 开发到 LoongArch 部署的风险清单

| 风险 | 等级 | 症状 | 规避 |
|---|---|---|---|
| pip 现场编译失败 | 高 | Failed building wheel | 只用纯 Python 包；离线 wheel 包随行 |
| SDK 隐藏二进制依赖 | 高 | 装 openai/dashscope SDK 连带 pydantic-core/grpcio 编译 | 用 requests 直连 HTTP API |
| Node/前端工具链上机 | 高 | 龙芯上装不出 Node | 前端 Windows 构建，只传 dist |
| MySQL 源码编译 | 高 | 数小时编译 + 依赖地狱 | SQLite 或仓库版 MariaDB |
| Docker 镜像无 loong64 | 高 | pull 不到镜像 | 不用 Docker，裸机 systemd |
| 新旧世界二进制不兼容 | 中 | 网上下载的 loongarch 二进制报架构/ABI 错误 | 只用系统仓库 + 源码内纯 Python |
| 比赛环境外网受限 | 中 | 装不了包/调不通云 API | 离线依赖包 + 第一时间实测出网连通性 |
| 4 核 8GB 性能瓶颈 | 低 | 并发慢 | 单机演示无压力，避免本地重计算 |
| 最后一周才首次部署 | 高 | 一切问题集中爆发 | 第一天演练 + 每周重部署 |
