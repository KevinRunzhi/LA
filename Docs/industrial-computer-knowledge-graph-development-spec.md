# 工控机知识图谱全局与变更视图开发 Spec

> 版本：V1.0
> 日期：2026-07-13
> 产品方案来源：`industrial-computer-knowledge-graph-overview-and-focus-plan.md`
> 现有展示基线：`expert-knowledge-graph-display-spec.md`
> 实现环境：React 18 + Vite 6 + Flask + SQLite
> 当前状态：开发 Spec 已完成，等待按批次实施

## 1. 目标与完成定义

本阶段把当前散热子图升级为一张以“工控机”为中心、覆盖多个故障领域的综合知识图谱，同时保持散热异常为唯一完整案例闭环。

页面只提供两个正式视图：

1. 全局知识图谱；
2. 本次新增与修改。

本阶段完成必须同时满足：

1. 中心节点统一显示“工控机”，不使用具体型号作为全局中心；
2. 全局图谱包含环境、供电、硬件、安装运维和监控告警五个一级领域；
3. 图谱数据来自后端统一结构，不再以 `AdminShell.jsx` 硬编码节点作为唯一来源；
4. 全局知识图谱和本次新增与修改能够稳定切换；
5. 点击节点后，镜头平滑移动并放大该节点及一跳关系；
6. 支持鼠标滚轮缩放、拖拽平移和一键返回全局；
7. 节点详情显示类型、领域、知识状态、来源、案例、版本和关联关系；
8. 工程师候选节点与关系不会直接进入正式图谱；
9. 专家发布后，候选关系在同一 SQLite 事务中转为正式关系；
10. 发布动画能够清晰表现“待审核 → 正式并入图谱”；
11. 现有 `KB-008 V1.0 → V1.1`、工程师同步和知识验证流程不回归；
12. Node `20.19.4` 构建、后端自动化测试和浏览器录制路径通过；
13. WSL 验证完成后，再进入龙芯开发态验证。

## 2. 当前技术约束

### 2.1 已有技术栈

```text
前端：React 18.3.1 + Vite 6.4.3
图标：lucide-react
后端：Flask 3.0.3
数据：JSON 种子 + SQLite
固定 Node：20.19.4
固定 npm：10.8.2
```

### 2.2 本阶段依赖决定

第一版不新增图谱或动画依赖：

- 不引入 ECharts；
- 不引入 D3；
- 不引入 React Flow；
- 不引入 Canvas/WebGL 图谱库；
- 不引入 Framer Motion；
- 不引入真实图数据库。

采用：

```text
React 状态
  + 原生 SVG 节点与连线
  + 预计算固定坐标
  + SVG <g> 场景 transform
  + requestAnimationFrame 镜头插值
  + CSS 状态动画
```

选择理由：

- 当前预计 40–80 个节点、60–120 条关系，SVG 足够承载；
- 演示需要每次打开时保持相同布局，不使用随机力导向布局；
- 固定坐标便于保证本次变更节点出现在镜头中心；
- React 和 SVG 可以直接复用现有状态、样式和权限逻辑；
- 不新增依赖更有利于龙芯和离线环境；
- SVG 便于实现节点点击、键盘选择、路径高亮和发布动画；
- 以后规模明显增长时，可以保留 API 和数据模型，只替换渲染层。

## 3. 演示优先的展示策略

### 3.1 视觉方向

延续现有深色工业控制台风格，图谱画布采用：

- 深蓝黑背景；
- 低对比工程网格；
- 细线连接；
- 少量颜色表达节点类型；
- 黄色表达候选变更；
- 橙色表达专家修改；
- 绿色表达正式发布；
- 右侧详情保持稳定，不让画布因详情变化改变宽度。

图谱不是持续运动的“科技背景”。除用户操作和发布动作外，所有节点必须保持稳定。

### 3.2 页面结构

```text
┌─────────────────────────────────────────────────────────────┐
│ 页面标题 / 图谱统计       [全局知识图谱] [本次新增与修改] │
├─────────────────────────────────────────────────────────────┤
│ 发布状态 / 版本 / 来源案例 / 返回全局                      │
├────────────────────────────────────────┬────────────────────┤
│                                        │                    │
│               图谱画布                 │     节点详情       │
│                                        │                    │
│                                        │                    │
├────────────────────────────────────────┴────────────────────┤
│ 图例：领域 / 既有知识 / 候选 / 专家修改 / 已发布          │
└─────────────────────────────────────────────────────────────┘
```

桌面端建议：

```css
grid-template-columns: minmax(0, 1fr) 300px;
```

- 1366px 视口中详情区约 `280–300px`；
- 1920px 视口中详情区不超过 `340px`；
- 画布优先获得剩余空间；
- 右侧详情更新不得触发画布重新布局；
- 小于 `1100px` 时详情移到画布下方，不覆盖图谱。

### 3.3 全局知识图谱

初始镜头显示：

- 中心“工控机”；
- 五个一级领域；
- 代表性二级节点；
- 重要叶子节点；
- 全局统计和图例。

默认不显示每条边的永久文字标签，避免关系文字相互遮挡。关系名称只在以下情况出现：

- 选中节点后的相关边；
- 鼠标悬停某条边；
- 本次新增与修改视图；
- 右侧关联关系列表。

### 3.4 本次新增与修改

进入该视图后：

1. 计算本次变更节点及其一跳邻居；
2. 计算这些节点的包围盒；
3. 镜头自动平移、缩放到包围盒；
4. 其他节点保留在画布中但降低至 `0.12–0.2` 透明度；
5. 候选节点和关系按状态高亮；
6. 右侧默认选中本次最重要的候选节点；
7. 发布后不跳转页面，在原位置播放并入正式图谱动画。

该视图承担当前散热案例的局部展示，但页面名称始终是“本次新增与修改”，不新增“散热案例聚焦”入口。

## 4. 图谱数据规模与布局

### 4.1 第一版规模

建议首版控制在：

```text
节点：45–65 个
关系：65–100 条
一级领域：5 个
完整闭环案例：1 条
正式知识：现有 8 条
本次候选/变更节点：3–6 个
本次候选/变更关系：3–8 条
```

### 4.2 布局原则

使用确定性分区布局，不使用运行时力导向：

```text
                  环境工况异常
                       │
安装与运维不规范 ── 工控机 ── 硬件部件失效
                       │
                  供电系统故障

             监控告警失效位于右下关联区
```

具体坐标保存在图谱种子数据中：

```json
{
  "layout": {
    "x": 520,
    "y": 340,
    "cluster": "hardware",
    "rank": 2
  }
}
```

要求：

- 全局布局首次加载不计算随机位置；
- 相同种子每次打开位置完全一致；
- 新候选节点优先放置在其主要关联节点附近；
- 候选节点缺少坐标时，后端给出建议簇，前端使用簇内预留槽位；
- 不允许候选节点因数组顺序变化在每次刷新时换位置；
- 节点数量超过当前布局容量时，优先折叠低层节点，不缩小到不可读。

## 6. 前端状态模型

```js
const graphViewStates = {
  OVERVIEW: "overview",
  CHANGES: "changes",
};

const [view, setView] = useState("overview");
const [selectedNodeId, setSelectedNodeId] = useState("DEVICE-INDUSTRIAL-COMPUTER");
const [camera, setCamera] = useState({ x: 0, y: 0, scale: 1 });
const [interaction, setInteraction] = useState("idle");
const [publishMotion, setPublishMotion] = useState("idle");
```

`interaction`：

```text
idle
dragging
zooming
animating
```

`publishMotion`：

```text
idle
preparing
merging
completed
```

状态约束：

- 镜头动画期间允许用户按 Escape 取消并返回稳定状态；
- 用户拖拽时立即中止未完成的自动镜头动画；
- 页面切换后重新进入时，根据当前视图恢复全局或变更包围盒；
- 发布动画只播放一次，不因 React 重渲染重复播放；
- 浏览器刷新后根据正式状态直接展示发布后结果，不重放发布动画。

## 7. 点击放大与镜头系统

### 7.1 场景变换

所有节点和边放在同一个 SVG 场景组：

```jsx
<g transform={`translate(${camera.x} ${camera.y}) scale(${camera.scale})`}>
  <GraphEdges />
  <GraphNodes />
</g>
```

节点和边必须共享同一个坐标系，禁止继续分别用 HTML 百分比定位和独立 SVG 连线。

### 7.2 单击节点

单击节点执行：

1. 设置 `selectedNodeId`；
2. 计算节点及一跳邻居；
3. 高亮相关节点与边；
4. 无关内容降低透明度；
5. 镜头平滑移动到目标区域；
6. 右侧详情切换为当前节点；
7. 显示“返回全局”操作。

不是把单个节点无限放大，而是让节点和一跳关系共同进入适合阅读的镜头范围。

建议镜头参数：

```text
全局最小缩放：0.65
常规初始缩放：1.00
节点聚焦缩放：1.35–1.65
最大手动缩放：2.20
聚焦边距：80–120 图谱坐标单位
```

### 7.3 自动 fit bounds

实现：

```js
fitGraphBounds(nodes, viewport, padding, scaleLimits)
```

输入目标节点集合，返回：

```js
{
  x,
  y,
  scale,
}
```

用于：

- 页面首次进入全局视图；
- 点击节点；
- 进入本次新增与修改；
- 发布后重新容纳新增节点；
- 点击返回全局。

### 7.4 手动缩放和平移

支持：

- 鼠标滚轮以指针位置为中心缩放；
- 按住空白画布拖拽平移；
- 工具栏 `+`、`−` 和“返回全局”；
- Escape 返回全局镜头；
- 点击空白区域取消节点选择，但不强制立即复位镜头。

第一版不做：

- 节点自由拖拽；
- 框选；
- 多选编辑；
- 小地图；
- 无限画布编辑器。

这些能力会增加演示复杂度，但不增强当前知识回流主线。

## 8. 动画规范

### 8.1 动画原则

- 动画只服务“定位、理解和发布结果”；
- 节点平时保持静止；
- 不使用循环漂浮、持续呼吸或随机摆动；
- 不使用会改变节点最终坐标的弹簧动画；
- 一次主要操作只播放一个主动画；
- 自动镜头移动期间不同时弹出多个提示。

### 8.2 时间与缓动

```css
--graph-motion-fast: 180ms;
--graph-motion-base: 360ms;
--graph-motion-camera: 560ms;
--graph-motion-publish: 760ms;
--graph-ease-out: cubic-bezier(0.22, 1, 0.36, 1);
--graph-ease-standard: cubic-bezier(0.4, 0, 0.2, 1);
```

### 8.3 视图切换动画

全局 → 本次新增与修改：

```text
0–120ms   无关节点降低透明度
80–560ms  镜头移动并放大变更包围盒
260–520ms 候选关系由中性线转为黄色虚线
360–560ms 右侧详情切换到首个候选节点
```

本次新增与修改 → 全局：

```text
0–480ms   镜头返回全局包围盒
180–480ms 其他领域恢复透明度
320–520ms 关系标签收起
```

### 8.4 节点点击动画

- 镜头移动：`420–560ms`；
- 被选节点只放大到 `1.06–1.10`，避免遮挡邻居；
- 相关边在 `180ms` 内增强；
- 无关节点淡化到 `0.16`；
- 详情内容使用 `120ms` 淡出 + `180ms` 淡入；
- 连续点击不同节点时，从当前镜头位置直接插值，不先返回全局。

### 8.5 发布并入动画

专家点击发布成功后：

```text
1. 发布状态条显示“正在写入知识版本”        0–260ms
2. 候选节点黄色边框转为绿色                180–520ms
3. 候选虚线沿路径方向绘制为绿色实线        260–760ms
4. 来源案例和知识版本关系依次出现           420–760ms
5. 节点轻微高亮一次后保持正式状态           620–980ms
6. 状态条更新为“已正式并入知识图谱”         760–980ms
```

关系绘制使用 SVG：

```css
stroke-dasharray: var(--edge-length);
stroke-dashoffset: var(--edge-length);
```

动画结束后设置 `stroke-dashoffset: 0`，不得循环播放。

### 8.6 减弱动画

`prefers-reduced-motion: reduce` 下：

- 镜头在 `1ms` 内到达目标位置；
- 保留节点、边和状态颜色变化；
- 不播放路径绘制和节点缩放；
- 详情内容直接替换。

## 9. 节点和关系视觉规范

### 9.1 节点层级

| 层级 | 表现 |
| --- | --- |
| 中心设备 | 最大节点，显示“工控机” |
| 一级领域 | 中型节点，领域名称和数量 |
| 二级故障 | 常规节点，显示简短名称 |
| 叶子知识 | 较小节点，聚焦时显示完整标签 |
| 案例/知识版本 | 使用独立形状或边角标识 |

### 9.2 变更状态

| 状态 | 节点 | 关系 |
| --- | --- | --- |
| 既有正式知识 | 中性深蓝 | 细灰蓝实线 |
| 工程师候选 | 黄色描边 | 黄色虚线 |
| 专家修改 | 橙色描边 | 橙色虚线 |
| 专家拒绝 | 降低透明度并显示删除标记 | 红色短虚线 |
| 已发布 | 绿色描边 | 绿色实线 |

颜色必须同时配合文字状态或线型，不能只依赖颜色表达含义。

### 9.3 标签策略

- 中心和一级领域始终显示标签；
- 常规二级节点显示一行或两行短标签；
- 叶子节点在全局缩放低于阈值时只显示关键字；
- 镜头放大后显示完整名称；
- 关系名称默认隐藏，聚焦后显示；
- 标签不得随镜头缩放小于可读下限，可按缩放级别切换简写和全称。

右侧详情固定展示：

```text

节点类型

知识状态
可信等级

来源案例

```

状态差异：

- `presentation_seed`：显示“预置展示知识”；
- `collecting`：显示“资料待完善”；
- `verified_case`：显示“完整案例验证”；
- `official_document`：显示“正式资料依据”；
- `expert_confirmed`：显示“专家已确认”；
- `candidate`：显示“工程师候选，尚未参与正式诊断”；
- `published`：显示“已正式参与知识检索”。

本阶段详情面板只读。候选内容编辑继续发生在工程师案例确认和专家审核页面，不在图谱总览中建设通用编辑器。

## 11. 后端数据结构

### 11.1 图谱节点表

新增：

```sql
CREATE TABLE IF NOT EXISTS graph_nodes (
  id TEXT NOT NULL,
  version TEXT NOT NULL,
  name TEXT NOT NULL,
  node_type TEXT NOT NULL,
  domain TEXT NOT NULL,
  status TEXT NOT NULL,
  change_type TEXT,
  verification_level TEXT NOT NULL,
  knowledge_id TEXT,
  source_case_id TEXT,
  payload TEXT NOT NULL,
  published_by TEXT,
  published_at TEXT,
  PRIMARY KEY (id, version)
);
```

`payload` 保存详情、证据引用、布局坐标和适用范围，第一版避免为所有展示属性增加独立列。

### 11.2 图谱关系表兼容扩展

保留现有 `graph_relations`，增加可选字段：

```text
relation_id
source_node_id
target_node_id
status
change_type
verification_level
payload
```

现有 `source / relation / target` 文本字段继续保留，保证旧接口和工程师知识快照兼容。

### 11.3 候选变更

第一版继续将候选节点和候选关系保存在 `expert_reviews.draft` 中：

```json
{
  "graphProposal": {
    "nodes": [],
    "relations": []
  }
}
```

专家发布事务同时：

1. 保存审核结果；
2. 创建知识版本；
3. 写入正式图谱节点；
4. 写入正式图谱关系；
5. 更新案例状态；
6. 更新工程师同步状态；
7. 更新演示状态。

任一步失败必须整体回滚。

## 12. 图谱种子

```text

```

结构：

```json

  "nodes": [],
  "relations": [],
  "views": {
    "overview": {
      "defaultSelectedNodeId": "DEVICE-INDUSTRIAL-COMPUTER"

}
```

旧 `graph_seed.json` 和 `graph_relations.json` 在新接口稳定前保留；新页面不再直接读取它们。迁移完成并验证无独有数据后，再决定是否删除或只保留兼容接口。

## 13. API 设计

### 13.1 获取图谱

```http
GET /api/admin/knowledge-graph?view=overview
GET /api/admin/knowledge-graph?view=changes
```

返回：

```json
{
  "data": {
    "view": "overview",
    "centerNodeId": "DEVICE-INDUSTRIAL-COMPUTER",
    "nodes": [],
    "relations": [],
    "changeNodeIds": [],
    "changeRelationIds": [],
    "statistics": {
      "nodeCount": 0,
      "relationCount": 0,
      "domainCount": 5,
      "knowledgeCount": 0,
      "caseCount": 0
    }
  }
}
```

### 13.2 获取节点详情

第一版节点详情可以直接随图谱返回，避免每次点击请求接口。数据明显增大后再拆分为：

```http
GET /api/admin/knowledge-graph/nodes/<node_id>
```

### 13.3 保存工程师候选变更

```http
POST /api/admin/cases/<case_id>/graph-proposal
```

请求：

```json
{
  "nodes": [],
  "relations": []
}
```

返回候选变更和校验结果，不写入正式 `graph_nodes` 或正式关系版本。

### 13.4 发布

继续复用：

```http
POST /api/admin/cases/<case_id>/publish
```

发布结果增加：

```json
{
  "graphChangeSummary": {
    "addedNodes": 2,
    "modifiedNodes": 1,
    "addedRelations": 3,
    "rejectedRelations": 0
  }
}
```

## 14. 数据校验

后端必须校验：

- 节点 ID 唯一；
- 节点名称非空；
- 节点类型属于允许枚举；
- 领域属于五个一级领域或通用类型；
- 关系起点和终点存在；
- 禁止节点指向自身；
- 同一版本内不重复写入同语义关系；
- 候选知识至少关联一个来源案例；
- 正式发布必须包含专家身份；
- `presentation_seed` 不得在接口中伪装成 `verified_case`；
- 被拒绝关系不得进入正式版本；
- 发布后工程师本地版本仍保持旧版本，直到主动同步。

## 15. 性能与稳定性

### 15.1 SVG 容量边界

首版性能目标：

```text
节点 ≤ 100
关系 ≤ 180
初次渲染 ≤ 500ms（本地开发环境）
镜头动画保持接近 60fps
```

超过容量时：

- 全局视图按领域折叠叶子节点；
- 本次变更仍只渲染相关一至两跳；
- 不通过继续缩小标签来容纳全部节点。

### 15.2 React 优化

- 节点和关系映射使用 `useMemo`；
- `GraphNode` 和 `GraphEdge` 使用 `React.memo`；
- 相机动画直接更新 SVG 场景组或使用单一相机 state，不逐节点更新位置；
- 拖拽过程中不重新计算一跳邻居；
- 指针移动用 `requestAnimationFrame` 合并更新；
- 不在每帧调用全量 DOM 测量；
- 详情面板与画布状态分离，详情更新不重新生成布局。

## 16. 可访问性与输入方式

- SVG 具有明确 `aria-label`；
- 节点支持 Tab 聚焦和 Enter 选择；
- 工具栏按钮有可见文字或 `aria-label`；
- 颜色同时配合形状、虚实线和文字状态；
- Escape 返回全局；
- `+` / `−` 按钮提供鼠标滚轮之外的缩放入口；
- 详情面板更新使用非打断式 `aria-live="polite"`；
- 减弱动画模式必须可用。

## 17. 演示脚本

### 发布前

1. 专家进入知识图谱；
2. 默认展示全局知识图谱；
3. 说明系统覆盖五个故障领域；
4. 点击一个领域节点，展示平滑放大和一跳关系；
5. 点击“返回全局”；
6. 切换到“本次新增与修改”；
7. 镜头自动定位候选节点和关系；
8. 点击候选节点，右侧显示工程师、案例、证据和待审核状态；
9. 返回专家审核并发布知识。

### 发布后

1. 返回“本次新增与修改”；

3. 黄色候选关系转为绿色正式关系；
4. 详情显示专家、知识版本和来源案例；
5. 返回全局，新增知识保留在大图谱中；
6. 工程师同步 V1.1；
7. 验证新关系参与后续问答。

## 18. 验收用例

### 18.1 全局视图

- 打开页面后完整看到中心节点和五个一级领域；
- 节点位置每次刷新一致；
- 图谱统计与返回数据一致；
- 点击任一一级领域后镜头平滑放大；
- 关系和详情与所选节点一致；
- 点击返回全局后恢复初始镜头；
- 多次点击不同节点不会闪回或跳动。

### 18.2 本次新增与修改

- 进入视图后自动容纳全部变更节点；
- 无关节点明显弱化但没有消失；
- 候选、专家修改和已发布状态可区分；
- 发布关系统一读取专家实际编辑结果；
- 页面不存在独立的“散热案例聚焦”视图；
- 没有候选变更时显示清晰空状态，不缩放到空白区域。

### 18.3 缩放和平移

- 滚轮缩放以指针位置为中心；
- 缩放不会超过 `0.65–2.20`；
- 拖拽画布不会触发节点点击；
- 用户操作能中止自动镜头动画；
- 快速连续点击节点不会产生相机漂移；
- 1366×768 下右侧详情不会遮挡画布；
- 1920×1080 下画布没有大面积无效空白。

### 18.4 发布动画

- 动画只在发布成功后播放一次；
- 发布 API 失败时不播放正式并入动画；
- 动画完成后视觉状态与 SQLite 正式状态一致；
- 刷新页面不会重新播放；
- 减弱动画模式下直接显示最终结果；
- 发布过程中没有控制台错误。

### 18.5 回归

- 工程师案例提交正常；
- 专家草稿保存正常；
- 知识版本仍从 V1.0 更新至 V1.1；
- 正式关系仍在 SQLite 事务中发布；
- 工程师同步前不命中新知识；
- 工程师同步后命中新知识；
- 一键重置恢复图谱种子和 V1.0 状态；
- 原知识库和历史案例页面不回归。

## 21. 本阶段不做

- 不增加第三种正式图谱视图；
- 不建设全部故障的完整案例；
- 不在图谱页自由编辑正式节点；
- 不做节点自由拖拽布局；
- 不做随机力导向动画；
- 不做小地图和框选；
- 不做大规模图数据库迁移；
- 不做自动 PDF 抽取和大模型知识生成；
- 不做多人协同编辑；
- 不为了视觉效果持续播放无业务意义的动画。

## 22. 最终实现结论

基于当前技术栈，最佳第一版不是引入复杂图谱框架，而是使用“固定工业知识布局 + React 原生 SVG + 可控镜头系统”。

全局知识图谱负责证明知识覆盖面；本次新增与修改负责证明工程师经验如何经过专家审核进入正式知识。点击节点时通过镜头平移和缩放展示局部关系，发布时通过一次明确的并入动画证明数据状态发生了真实变化。

该方案实现成本可控、画面稳定、适合录制，并且保留后续增加轻量案例、真实手册和更多图谱节点的扩展能力。
