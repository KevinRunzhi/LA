# R2【P0】可编辑检修预方案开发 Spec

> 版本：V1.1
> 日期：2026-07-10
> 产品设计来源：`r2-editable-maintenance-plan-design.md`
> 实现边界：React 前端内存状态，不新增依赖，不新增后端接口
> 当前状态：WSL 第一版已实现并通过 Node 20.19.4 构建与浏览器主路径验证；龙芯开发态待验证

## 1. 完成定义

R2 只有同时满足以下条件才算完成：

1. 诊断结论不能直接进入检修向导；
2. 点击后运行检修编排 Agent，并生成两级预方案；
3. 大步骤和小维修步骤支持增删改与上下移动；
4. 强制安全项不能被删除；
5. 确认后的步骤数量、标题、顺序和检查项真实进入向导；
6. 左侧流程读取确认后的方案；
7. 向导推进时不能重新请求原始步骤覆盖编辑结果；
8. 检修记录至少能反映确认后的步骤数量和标题；
9. Node 20.19.4 构建通过，浏览器验收路径通过；
10. 龙芯开发态验证后才能标记“龙芯已完成”。

## 2. 状态机

```js
const planStates = {
  IDLE: "idle",
  GENERATING: "generating",
  EDITING: "editing",
  CONFIRMING: "confirming",
  CONFIRMED: "confirmed",
};
```

| 当前状态 | 触发 | 下一状态 | 页面变化 |
| --- | --- | --- | --- |
| `IDLE` | 诊断结论点击生成方案 | `GENERATING` | 右侧 Agent 输出，大纲逐项出现 |
| `GENERATING` | 生成时序完成 | `EDITING` | 全量大纲可编辑 |
| `EDITING` | 增删改/排序 | `EDITING` | 更新工作版本与修改统计 |
| `EDITING` | 点击恢复初稿 | `EDITING` | 用系统初稿覆盖工作版本 |
| `EDITING` | 点击确认 | `CONFIRMING` | 校验标题、检查项与安全约束 |
| `CONFIRMING` | 校验通过 | `CONFIRMED` | 写入确认版本并进入向导 |

## 3. 数据结构

```js
{
  id: "step-04-filter-fan",
  order: 4,
  title: "检查滤网、风扇和接线",
  description: "检查积尘、风扇运行状态和接线顺序",
  source: "风扇与滤网检查流程",
  origin: "system",
  placeholder: "step-04-filter-fan",
  safety: "拆装前必须断电",
  thresholds: ["风扇 < 500 rpm 判定异常"],
  checks: [
    {
      id: "check-filter-dust",
      text: "检查门滤网积尘",
      origin: "system",
      source: "设备手册",
      locked: false
    }
  ]
}
```

为兼容现有向导，在确认时把对象检查项转换为向导当前使用的字符串数组，同时保留编辑元数据：

```js
confirmedGuideSteps = confirmedPlan.map(step => ({
  ...step,
  checks: step.checks.map(check => check.text),
  planChecks: step.checks,
}));
```

## 4. 组件职责

| 组件 | 职责 |
| --- | --- |
| `PlanGenerationTransition` | 复用 Agent 流式过渡，按时序生成方案 |
| `MaintenancePlanStage` | 管理系统初稿、工作版本、校验和确认 |
| `PlanOutline` | 渲染连续两级大纲 |
| `PlanStepEditor` | 大步骤编辑、移动、删除及子步骤管理 |
| `PlanCheckEditor` | 小维修步骤原位编辑、移动和删除 |
| `PlanRevisionSummary` | 显示阶段数、子步骤数和修改数 |

第一版允许组件暂时保留在 `App.jsx`，但状态和函数必须按上述职责分区，不能散落到向导组件内部。

## 5. 插入点

当前：

```text
DiagnosisStage -> enterGuide() -> GuideStage
```

修改为：

```text
DiagnosisStage
  -> generateMaintenancePlan()
  -> MaintenancePlanStage
  -> confirmMaintenancePlan()
  -> GuideStage
```

新增 `stage === "plan"`。该阶段在流程栏归属“检修向导”，但显示“预方案确认”节点。

## 6. 编辑规则

### 大步骤

- 标题和描述可修改；
- 可新增、删除、上移、下移；
- 至少保留一个；
- 包含锁定安全项时不可删除。

### 小维修步骤

- 原位修改文字；
- 可新增、删除、上移、下移；
- 每个大步骤至少保留一项；
- `locked: true` 的删除按钮保持禁用，并显示“强制安全项不可删除”，让工程师能明确理解限制原因。

### ID

新增节点使用浏览器会话唯一 ID：

```js
plan-step-${Date.now()}-${counter}
plan-check-${Date.now()}-${counter}
```

不得使用数组索引作为 React key。

## 7. 安全约束

第一版锁定以下现有项：正常关机、拔除所有电源、等待冷却、防静电。校验时确认它们仍存在于方案中。

失败文案示例：

```text
当前方案缺少强制安全约束“拔除所有电源”，无法进入检修向导。
```

## 8. 向导数据约束

- `confirmMaintenancePlan` 后清空旧检查勾选状态；
- `activeStep` 重置为 `0`；
- 向导只读取确认版本；
- `completeCurrentStep` 只在本地确认版本上标记完成；
- 不得调用 `api.steps()` 覆盖确认版本；
- 新增步骤没有专用图片时显示通用“工程师补充步骤”视觉占位。

## 9. 右侧辅助对话

生成阶段展示方案依据；编辑阶段记录最近修改：

```text
工程师删除了“检查蜂鸣器状态”。
工程师新增了“拍摄滤网积尘照片”。
当前方案由 5 个阶段、23 条维修步骤调整为 5 个阶段、23 条维修步骤。
```

第一版只保留最近 8 条修改记录，不做完整审计数据库。

## 10. 验收用例

### 主路径

1. 从诊断结论点击生成预方案；
2. Agent 过渡后进入大纲页；
3. 修改一个大步骤标题；
4. 删除一个普通小维修步骤；
5. 新增“拍摄滤网积尘照片”；
6. 新增一个大步骤；
7. 上移或下移步骤；
8. 确认方案；
9. 验证向导步数、标题和检查项与确认方案一致。

### 安全保护

- 尝试删除“拔除所有电源”，操作被阻止；
- 尝试删除整个安全准备阶段，操作被阻止；
- 清空标题或全部检查项时，确认按钮不可用或显示明确错误。

### 回退

- 返回诊断结论后可再次进入工作版本；
- 恢复系统初稿后所有工程师修改被清除；
- 不出现空白页、假加载或重复生成。

## 11. 实施批次

1. 阶段接线与生成过渡；
2. 两级大纲只读渲染；
3. 增删改与上下移动；
4. 安全锁定和校验；
5. 确认结果接入向导；
6. 流程栏、右侧轨迹和记录同步；
7. 构建、浏览器和龙芯验证。
