# SQLite 知识回流与同步 Spec

> 版本：V1.0
> 日期：2026-07-12
> 范围：仅案例回流、知识版本、知识图谱与工程师同步

## 1. 迁移边界

SQLite 负责唯一完整案例的工程师确认、专家审核、知识版本、图谱关系、发布记录和工程师同步状态。动态接诊、R2 预方案、多 Agent 会诊、媒体展示与历史摘要案例继续使用现有前端状态和 JSON 展示数据。

JSON 保留为种子、一键重置和离线演示素材；运行时正式状态以 `backend/data/presentation/presentation.db` 为准。

## 2. 数据表

- `presentation_state`：当前演示状态和活动角色；
- `cases`：唯一完整案例及工程师确认结果；
- `expert_reviews`：专家草稿、审核决定、修订记录；
- `knowledge_items`：知识主体及当前正式版本；
- `knowledge_versions`：每次发布的完整结构化知识快照；
- `graph_relations`：按知识版本保存的正式图谱关系；
- `engineer_sync_records`：工程师本地版本、同步状态和同步时间。

## 3. 事务边界

专家发布必须在一个 SQLite 事务中完成：保存审核结果、创建知识版本、写入图谱关系、更新知识当前版本、更新案例状态和演示状态。任一步失败必须全部回滚。

## 4. 工程师同步模型

专家发布后正式版本变为 V1.1，工程师本地版本仍可保持 V1.0，状态为 `update_available`。工程师同步后记录变为 V1.1 / `current`。工程师本地图谱和现场问答只读取该工程师已同步版本。

## 5. API

- `GET /api/engineer/knowledge-sync`：查询本地版本与最新版本；
- `POST /api/engineer/knowledge-sync`：同步最新知识版本；
- `GET /api/engineer/knowledge-snapshot`：读取工程师当前本地知识和图谱快照；
- 现有案例提交、专家草稿、知识发布、知识差异和验证接口保持兼容。

## 6. 重置

`POST /api/presentation/reset` 清空上述七张表并从 JSON 种子重建 V1.0 初始状态；历史摘要 JSON 不进入可编辑知识链路。

## 7. 验收

- Flask 重启后专家草稿和发布状态不丢失；
- 专家发布的知识与图谱关系在同一事务提交；
- 发布后工程师状态显示有新版本；
- 同步后工程师本地快照与专家正式版本一致；
- 一键重置可以恢复 V1.0；
- 现有专家三步页面和 10 条摘要展示不回归。

## 8. 工程师端可见闭环（已实施）

- 工程师首页显示本地版本、最新版本和同步状态；
- 发现专家 V1.1 后提供“同步本地知识”唯一主操作；
- 工程师左侧增加“本地知识图谱”，读取 `engineer/knowledge-snapshot`；
- 同步前只展示 V1.0 基础关系，新关系不参与诊断；
- 同步后展示 V1.1 结构化知识、专家实际发布关系、来源案例和同步时间；
- 知识回流验证只读取工程师本地快照：未同步返回 `syncRequired`，同步后才命中 V1.1。
