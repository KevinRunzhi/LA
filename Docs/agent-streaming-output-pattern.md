# Agent 流式输出呈现规范

> 适用范围：MVP 中所有需要体现 Agent 参与过程的页面，包括接诊 Agent、设备识别 Agent、分析诊断 Agent、多 Agent 会诊、检修向导辅助说明等。

## 1. 设计目标

当前已确定的输出形式不是普通聊天机器人式回答，而是面向工控设备检修场景的“专业诊断过程展示”。

核心目标：

- 让用户感知系统经历了“分析问题 -> 检索依据 -> 匹配知识 -> 生成下一步”的过程。
- 不堆叠大量卡片，避免界面变得碎片化。
- 保持类似 Claude / ChatGPT / 千问的思考输出节奏，但文案必须贴合油气场站、工控机、维修手册、检修依据等业务场景。
- 当前阶段先做前端模拟，不接真实 API，但交互结构要方便后续替换为真实 SSE / WebSocket / fetch stream。

## 2. 当前已采用的呈现形式

以“接诊 Agent”为第一版样式基准。

整体由三部分组成：

1. Agent 状态标题
2. 单一区域内的流式思考文本
3. 小字号半透明检修依据框
4. 完成后的整理结果

界面不采用多个大卡片并列展示，而是在右侧辅助对话区中用一个轻量面板承载全过程。

## 3. Agent 状态标题

运行中标题：

```text
接诊 Agent 正在思考中 >
正在分析现场现象与下一步接入方向
```

完成后标题：

```text
接诊 Agent 已完成 >
已完成现场异常整理
```

显示要求：

- 左侧保留小型 loading 图标或完成图标。
- 标题字号不宜过大，保持辅助信息感。
- 颜色使用灰蓝、低饱和工业色，不使用强蓝紫渐变。
- 标题末尾可使用 `>` 表示可展开或可追溯过程。

## 4. 流式思考文本

### 4.1 输出方式

当前采用“字符级流式输出”：

- 不是整段一次性出现。
- 不是一行一行直接跳出来。
- 而是文本从左到右逐步生成，类似大模型回答。
- 当前正在输出的最后一行后面显示闪烁光标。

### 4.2 输出节奏

接诊 Agent 当前节奏：

- 每约 48ms 输出 1 个字符。
- 总过程约 7-8 秒。
- 过程比普通 loading 慢一些，适合演示时让观众看到“系统正在分析”。

后续可按场景调整：

- 接诊 Agent：6-8 秒。
- 设备识别 Agent：5-7 秒。
- 多 Agent 会诊：每个 Agent 3-5 秒。
- 生成诊断结论：8-12 秒。

### 4.3 文案结构

接诊 Agent 当前示例：

```text
这是一次站控柜工控机散热相关异常接入。系统先整理现场事实，再判断需要补充的设备信息。
识别到关键词：温度告警、风扇声音异常、风扇转速偏低。
检索站控柜工控机异常接入规则。
匹配控制中心 / 站控柜 A01 / 工控机区域。
生成下一步补充项：设备型号、设备位置、设备角色、关联告警。
```

文案原则：

- 第一行说明当前 Agent 在做什么。
- 第二行提取用户输入或当前页面中的关键信息。
- 第三行开始体现检索、匹配、规则判断。
- 最后一行输出下一步动作，不直接给完整诊断结论。

## 5. 检修依据框

### 5.1 设计形式

检修依据不要做成大卡片列表，而是放在一个小字号、半透明、低干扰的框内。

运行中显示：

```text
正在搜索检修依据 >
```

完成后显示：

```text
已参考 6 条检修依据 >
```

### 5.2 动态效果

- 在思考文本输出到中段后，依据框出现。
- 运行中显示小 loading 图标。
- 依据条目逐条出现，不一次性全部展开。
- Agent 完成后显示全部依据。

### 5.3 当前接诊 Agent 依据内容

```text
- 现场异常描述：温度告警、风扇声音异常、风扇转速偏低
- 设备接入规则：站控柜工控机异常接入流程
- ACP-4000 / IPC-610 检修指导：散热与风扇模块检查项
- 知识图谱节点：TEMP/FAN 告警、蜂鸣器、风扇模块、滤网积尘
- 相似案例：工控机高温告警与风道堵塞
- 风险提示：正常关机、断开电源、防静电操作
```

后续其他 Agent 可以替换为对应依据，例如：

- 设备识别 Agent：设备图片、型号库、站控柜设备档案、历史接入记录。
- 诊断 Agent：故障树、知识图谱、维修手册章节、相似故障案例。
- 检修向导 Agent：作业步骤、安全要求、工具清单、验收阈值。

## 6. 完成后的结果输出

完成后在同一个轻量面板底部输出整理结果。

接诊 Agent 当前示例：

```text
接诊整理结果：当前异常优先归入站控柜工控机散热相关异常。
下一步：补充设备型号、确认设备位置与角色，并关联 TEMP/FAN、蜂鸣、温度升高等告警。
```

要求：

- 结果要短，不要在接诊阶段直接输出完整维修方案。
- 只给出当前阶段该推进到哪里。
- 下一步必须和右侧 / 左侧诊断流程中的步骤一致。

## 7. 视觉风格规范

### 7.1 面板

- 使用轻边框或半透明浅底。
- 不使用明显厚重阴影。
- 不使用强色块。
- 不使用多个大卡片堆叠。

### 7.2 字体层级

- Agent 标题：13px 左右，半粗。
- 状态副标题：11px 左右，低饱和灰色。
- 思考文本：12px 左右，灰蓝色。
- 依据条目：10.5px 左右，小字号。
- 结果文本：12px 左右，颜色略深。

### 7.3 颜色方向

保持工业工具感：

- 主文本：深灰蓝。
- 思考文本：中灰蓝。
- 辅助文本：浅灰蓝。
- 边框：低透明度灰蓝。
- 避免大面积蓝紫渐变和“AI 味”强的发光效果。

## 8. 后续复用方式

后续新增 Agent 展示时，建议统一复用以下结构：

```text
[Agent 名称] 正在思考中 >
[当前阶段说明]

[流式思考文本]

[正在搜索 / 已参考 N 条依据 >]
- 依据 1
- 依据 2
- 依据 3

[Agent 结果]
[下一步动作]
```

不同 Agent 只替换：

- Agent 名称
- 思考文本
- 依据列表
- 完成结果
- 下一步动作

不要每个 Agent 重新设计一种完全不同的视觉形态。

## 9. 当前实现位置

当前第一版已在接诊 Agent 中实现。

涉及文件：

- `frontend/src/App.jsx`
  - 接诊 Agent 状态
  - 字符级流式输出计时
  - 检修依据动态显示
  - 完成结果输出
- `frontend/src/styles/app.css`
  - Agent 轻量面板样式
  - 思考文本样式
  - 检修依据小框样式
  - 光标闪烁样式

当前实现仍为前端模拟，后续可将：

- `triageStreamLines`
- `triageEvidenceItems`
- `triageAgentStatus`
- `triageCharCount`

替换为真实后端返回的流式数据。

## 10. 后续接真实 API 的预留方向

后续接入真实大模型时，可按以下方式迁移：

1. 前端发送当前阶段、用户输入、设备上下文、已选步骤信息到后端。
2. 后端通过 SSE / WebSocket / fetch stream 返回事件。
3. 前端按事件类型更新界面：

```text
agent_start
thinking_delta
evidence_start
evidence_item
result_delta
agent_done
```

4. 当前前端模拟的字符计时逻辑可替换为后端真实 token / 文本增量。

## 11. 验收标准

后续复用这套样式时，应满足：

- Agent 输出不是瞬间完成。
- 思考文本有从左到右生成的感觉。
- 检索依据有“正在搜索”状态。
- 依据不是大卡片堆叠，而是小字号、低干扰展示。
- 完成结果简洁，能推动用户进入下一步。
- 整体看起来像工控设备检修诊断台，而不是通用聊天机器人。
