# LA MVP 当前状态记录

更新时间：2026-07-07

本文档用于记录当前 MVP 骨架已经实现的内容、演示边界和后续待补能力。当前版本仍是演示型 MVP，重点是把老师建议的“对话辅助 + 步骤式诊断/检修引导”流程先跑通，不是完整生产系统。

## 1. 当前 MVP 定位

当前 MVP 是“站控慧眼：工控设备检修助手”的第一版可交互骨架。

核心方向已经从“纯对话式助手”调整为：

- 对话只作为辅助入口和解释能力。
- 主流程采用步骤式页面交互。
- 用户先描述异常，再补充设备和阈值信息。
- 系统模拟多 Agent 分析过程。
- 分析完成后生成诊断结论。
- 诊断结论继续转入步骤式检修向导。
- 检修完成后生成记录，并预留专家审核回流。

当前演示场景固定为一个设备、一个故障闭环：

| 项目 | 当前内容 |
| --- | --- |
| 场景 | 油气管道输气场站 |
| 位置 | 某输气场站 · 站控柜 A01 |
| 设备 | 研华 ACP-4000 / IPC-610 工控机 |
| 故障 | 高温告警 / 风道堵塞 / 散热异常 |
| 主要现象 | 温度告警、风扇声音异常、前面板风扇转速很低 |
| 诊断方向 | 工控机散热异常，重点排查风道、滤网、风扇和接线 |

## 4. 工作台主流程状态

当前工作台已经实现一条完整的预设演示流程：

```text
异常接入
  -> 分析诊断
  -> 诊断结论
  -> 检修向导
  -> 检修记录
  -> 专家审核回流
```

右侧“诊断流程”已经按阶段动态生成：

- 初始只显示“异常接入”。
- 触发诊断后显示“分析诊断”。
- 诊断完成后显示“检修向导”。
- 生成记录后进入“记录与回流”阶段。

这样可以营造“大模型根据异常信息逐步生成后续流程”的效果，虽然当前逻辑仍是预设演示流程。

## 5. 异常接入模块

异常接入已经从单纯输入框扩展为四步页面式流程。

### 5.1 第一步：描述现场现象

已实现内容：

- 中间主区域显示文本输入框。
- 默认示例内容为：站控柜内工控机温度告警，风扇声音异常，前面板风扇转速很低。
- 右侧显示本步采集内容：现场现象、告警类型、图片入口预留。
- 图片上传按钮目前是预留入口，不执行真实上传。

### 5.2 第二步：补充设备型号

已实现内容：

- 设备型号、设备位置、设备角色、关联告警均改为下拉选择。
- 默认状态为空，需要用户选择。
- 每组下拉目前只放一个真实推荐选项和若干预留选项。
- 支持“自动识别并补充”按钮。

当前推荐值：

| 字段 | 推荐值 |
| --- | --- |
| 设备型号 | 研华 ACP-4000 / IPC-610 |
| 设备位置 | 站控柜 A01 内部工控机 |
| 设备角色 | 站控画面与数据采集终端 |
| 关联告警 | TEMP/FAN、蜂鸣、温度升高 |

“自动识别并补充”目前是前端模拟能力：

- 点击后显示短暂识别中。
- 约 850ms 后自动填入推荐值。
- 不调用真实模型。

### 5.3 第三步：确认灯值和阈值

已实现内容：

- TEMP/FAN LED、风扇转速、系统温度、CPU 温度均为可编辑输入。

当前阈值：

| 风扇转速 | < 500 rpm |
| 系统温度 | > 55°C |

### 5.4 第四步：接入信息确认

已实现内容：

- 汇总异常描述。
- 汇总设备型号。
- 汇总关键阈值。
- 展示系统将生成的诊断任务。

当前生成任务：

- 解析现场异常描述与告警信号。
- 匹配 ACP-4000 / IPC-610 散热结构。
- 检索 TEMP/FAN、风扇转速和温度阈值。
- 生成散热异常诊断结论与检修向导。

## 6. 分析诊断模块

分析诊断已经调整为“触发诊断 + 多 Agent 会诊 + 生成结论”的步骤式结构。

### 6.1 触发诊断

已实现内容：

- 用户在异常接入最后一步点击“触发诊断”。
- 前端调用 `/api/diagnosis/start`。
- 后端返回预设诊断结果。
- 页面切换到 Agent 动态运行阶段。

### 6.2 多 Agent 动态会诊

已实现类 Claude/GPT 的动态分析感：

- 显示“多 Agent 会诊进行中”。
- 显示诊断执行进度条。
- 显示 Agent 执行轨道。
- 每个 Agent 有等待中、运行中、已完成状态。
- 当前 Agent 展示逐条出现的分析文本。
- 每个 Agent 约停留 3.2 秒后进入下一个。
- 所有 Agent 完成后自动进入诊断结论步骤。

当前 Agent：

| Agent | 当前作用 |
| --- | --- |
| 分析诊断 Agent | 优先检查风道堵塞、滤网积尘和风扇异常 |
| 操作合规 Agent | 校验断电、挂牌、防静电等安全要求 |
| 知识检索 Agent | 命中 TEMP/FAN、转速、温度阈值等依据 |

注意：当前 Agent 运行是前端定时模拟，不是真实多 Agent 调度。

### 6.3 诊断结论步骤

诊断结论当前拆成三个页面式子步骤：

| 子步骤 | 已实现内容 |
| --- | --- |
| 触发诊断 | 展示已接收异常信息、设备、故障方向、诊断边界 |
| 多 Agent 会诊 | 展示 Agent 结果和知识条目证据 |
| 弹出并生成诊断结论 | 展示完整诊断结论、风险和原因排序 |

当前诊断结论：

- 判断为一次工控机散热相关异常。
- 优先怀疑风道堵塞 / 滤网积尘。
- 同时考虑风扇低速或停转。
- 需要确认机柜环境温度和通风条件。
- 建议进入步骤式检修向导。

## 7. 检修向导模块

检修向导已经从“作业卡记录”调整为“步骤式检修向导”。

已实现内容：

- 主区域每次只显示一个检修步骤。
- 每个步骤都有说明、图片占位区、检查项、安全提醒、阈值要求和来源依据。
- 检查项做成可点击勾选框。
- 当前步骤顶部显示本步确认进度。
- 未勾选完所有检查项时，“完成并继续”按钮不可用。
- 完成当前步骤后进入下一步。
- 右侧流程列表会标记当前步骤和已完成状态。
- 用户可以点击右侧步骤回跳。

当前检修步骤共 5 步：

| 步骤 | 标题 | 核心检查项 |
| --- | --- | --- |
| 1 | 确认告警与定位设备 | TEMP/FAN、蜂鸣器、风扇 rpm、系统/CPU 温度、站控柜位置 |
| 2 | 执行安全准备 | 通知负责人、正常关机、拔除电源、等待冷却、防静电 |
| 3 | 检查机柜环境与风道 | 环境温度、进风口、出风口、机箱开孔 |
| 4 | 检查滤网、风扇和接线 | 门滤网、风扇滤网、风扇状态、FAN1/FAN2 顺序 |
| 5 | 恢复运行与验证 | TEMP/FAN 解除、风扇转速、系统温度、CPU 温度、数据上传 |

图片相关状态：

- 当前每步都有图片占位区。
- 不做真实图片上传、插入或展示。
- 后续可以把手动标注好的图片放入对应步骤。

限制：

- 没有真实用户、专家账号和权限。
- 没有持久化知识沉淀。

## 14. 当前演示可讲的闭环

当前 MVP 已经可以演示以下完整路径：

1. 一线人员进入智能诊断台。
2. 在“异常接入”描述现场现象。
3. 选择或自动识别设备型号、位置、角色和关联告警。
4. 修改或确认 TEMP/FAN、风扇转速、系统温度、CPU 温度阈值。
5. 确认接入信息并触发诊断。
6. 系统展示多 Agent 会诊动态过程。
7. Agent 完成后生成诊断结论。
8. 用户查看可能原因排序。
9. 进入步骤式检修向导。
10. 工程师按步骤逐项勾选检查项。
11. 完成检修后生成检修记录。
12. 提交专家审核。
13. 专家修正回流到知识经验中。
14. 在知识图谱页查看与工控机散热异常相关的节点、关系和知识条目。

   当前系统还没有入口身份选择，演示时可以先做一个模拟登录页，角色包括一线检修人员、专家审核人员、管理员。

2. 改造设置页。
   设置页不要做成演示配置，而是做成实际员工日常使用的设置页面，例如个人信息、所属场站、班组、通知、默认设备范围、作业卡导出偏好。

3. 强化检修记录页。
   把当前简单记录页升级为正式检修台账/作业卡页面，展示设备信息、故障现象、诊断结论、检修步骤、安全确认、专家状态和导出入口。

4. 补充图片占位映射。
   当前每个检修步骤已有图片占位，后续可以把用户手动标注好的图片接入对应步骤。

5. 接入真实或半真实模型能力。
   在当前预设流程稳定后，再考虑接入云 API，让异常总结、步骤生成和辅助对话逐步从模拟变成真实能力。

## 16. 当前结论

当前 MVP 已经完成了“从异常接入到诊断、检修、记录、回流”的单场景闭环骨架。

它目前最适合承担：

- 比赛演示第一版。
- 产品流程验证。
- 和老师/队友讨论界面结构与交互逻辑。
- 后续接入模型、数据库、登录和更多设备场景的基础框架。

它还不是完整生产系统，关键限制是：流程预设、数据静态、Agent 模拟、无持久化、无真实登录、无真实模型能力。
