# LA MVP 当前状态记录

更新时间：2026-07-09

本文档用于记录当前 MVP 骨架已经实现的内容、演示边界、已放入的图片资源和后续要补的能力。当前版本仍是“高保真演示型 MVP 骨架”，重点是把老师建议的“对话辅助 + 步骤式诊断/检修引导”流程跑通，而不是完整生产系统。

当前所有工控机相关图片均按“研华 ACP-4000 / IPC-610 工控机”演示场景使用。

## 1. 当前 MVP 定位

当前 MVP 是“站控慧眼：工控设备检修助手”的可交互演示骨架。

核心产品方向已经从“纯对话式助手”调整为：

- 对话只作为入口和辅助解释。
- 主流程采用步骤式页面交互。
- 用户先描述异常，再补充设备、告警和阈值信息。
- 系统模拟多 Agent 分析过程。
- 分析完成后生成结构化诊断结论。
- 诊断结论继续转入步骤式检修向导。
- 检修完成后生成检修记录，并预留作业卡、专家审核和知识回流。

当前演示场景固定为一个设备、一个故障闭环：

| 项目 | 当前内容 |
| --- | --- |
| 场景 | 油气管道输气场站 |
| 位置 | 某输气场站 · 站控柜 A01 |
| 设备 | 研华 ACP-4000 / IPC-610 工控机 |
| 故障 | 高温告警 / TEMP-FAN 告警 / 风道堵塞 / 滤网积尘 / 散热异常 |
| 主要现象 | 温度告警、风扇声音异常、前面板风扇转速很低 |
| 诊断方向 | 工控机散热异常，重点排查风道、滤网、风扇和接线 |

## 2. 当前整体流程

当前工作台已经实现一条预设演示流程：

```text
首页智能诊断入口
  -> 异常接入
  -> 分析诊断
  -> 诊断结论
  -> 检修向导
  -> 恢复验证
  -> 检修记录
  -> 知识图谱 / 设置 / 后续回流
```

右侧“诊断流程”采用动态生成效果：

- 初始主要显示“异常接入”。
- 异常接入完成后生成“分析诊断”。
- 分析诊断完成后生成“检修向导”。
- 检修完成后进入“记录与回流”。

该逻辑用于营造“系统根据现场异常逐步生成诊断流程”的感觉。当前仍是预设流程，不是完全由大模型实时生成。

已实现内容：

- 页面中间放置较大的对话输入框。
- 输入现场现象后进入异常接入流程。
- 输入框内部有添加材料、图片、视频、语音等入口样式。
- 页面不堆过多内容，重点放在诊断入口。

当前能力边界：

- 文字输入可用。
- 图片、视频、语音入口为演示入口，不做真实上传和识别。
- 首页只是诊断起点，后续主要靠步骤式流程推进。

大概要实现的功能：

- 让一线检修人员先用自然语言描述问题。
- 系统接收问题后，不停留在对话，而是转入异常接入和诊断步骤。
- 后续正式 MVP 可把输入内容传给后端和大模型。

## 6. 异常接入模块

异常接入已经从单纯输入框扩展为四步页面式流程：

1. 描述现场现象。
2. 补充设备型号。
3. 确认灯值和阈值。
4. 接入信息确认。

### 6.1 第一步：描述现场现象

已实现内容：

- 左侧为现场现象输入框。
- 默认演示内容为：站控柜内工控机温度告警，风扇声音异常，前面板风扇转速很低。
- 右侧为场站位置提示图。
- 场站图上叠加蓝色闪烁故障点。
- 蓝点位于控制中心黑色机柜区域，用于提示站控柜 A01 工控机位置。

已添加图片：

| 图片 | 用途 |
| --- | --- |
| `site-station-overview.png` | 油气管道场站总览图，用于描述现场现象步骤的位置提示 |

## 7. 分析诊断模块

分析诊断已经调整为“触发诊断 + 多 Agent 会诊 + 生成结论”的步骤式结构。

### 7.1 触发诊断

已实现内容：

- 用户在异常接入最后一步点击“触发诊断”。
- 页面展示已接收异常接入信息。
- 展示设备、故障方向、诊断边界。
- 进入预设多 Agent 会诊流程。

大概要实现的功能：

- 明确诊断不是直接输出文字，而是基于结构化接入信息开始分析。
- 后续正式 MVP 可在此处调用后端诊断接口。

### 7.2 多 Agent 动态会诊

已实现类 Claude/GPT 的动态分析感：

- 显示“多 Agent 会诊进行中”。
- 显示诊断执行进度条。
- 显示 Agent 执行轨道。
- 每个 Agent 有等待中、运行中、已完成状态。
- 当前 Agent 展示逐条出现的分析文本。
- 每个 Agent 停留后进入下一个。
- 所有 Agent 完成后进入诊断结论步骤。

当前 Agent：

| Agent | 当前作用 |
| --- | --- |
| 分析诊断 Agent | 优先检查风道堵塞、滤网积尘和风扇异常 |
| 操作合规 Agent | 校验断电、挂牌、防静电等安全要求 |
| 知识检索 Agent | 命中 TEMP/FAN、转速、温度阈值等依据 |

当前能力边界：

- Agent 运行是前端定时模拟。
- 不是真实多 Agent 调度。
- 后续可以把 Agent 输出替换为大模型 API 或后端编排结果。

### 7.3 诊断结论

诊断结论当前拆成三个页面式子步骤：

| 子步骤 | 已实现内容 |
| --- | --- |
| 触发诊断 | 展示已接收异常信息、设备、故障方向、诊断边界 |
| 多 Agent 会诊 | 展示 Agent 结果和知识条目证据 |
| 弹出并生成诊断结论 | 展示完整诊断结论、风险和原因排序 |

当前诊断结论：

- 判断为一次工控机散热相关异常。
- 优先怀疑风道堵塞 / 滤网积尘。
- 同时考虑风扇低速或停转。
- 需要确认机柜环境温度和通风条件。
- 建议进入步骤式检修向导。

大概要实现的功能：

- 诊断结论不只停留在文本，而是转入可执行检修向导。
- 后续可将诊断结论按结构化 JSON 保存到数据库。

## 8. 检修向导模块

检修向导已经从“检修完成后的记录和导出”调整为“步骤式检修向导”。

已实现通用能力：

- 主区域每次显示一个检修步骤。
- 每个步骤都有说明、图片区域、检查项、安全提醒、阈值要求和来源依据。
- 检查项做成可点击勾选框。
- 当前步骤顶部显示本步确认进度。
- 未勾选完所有检查项时，不能完成并继续。
- 完成当前步骤后进入下一步。
- 右侧流程列表标记当前步骤和已完成状态。
- 用户可以点击右侧步骤回跳。
- 点击检查项时，中间图片会切换，并带淡入淡出效果。

当前检修步骤共 5 步。

- 站控柜位置。

## 11. 知识图谱页面

知识图谱页已实现节点图样式。

已实现内容：

- 右侧为当前节点详情、关联关系、知识条目。
- 节点可点击切换。
- 当前节点高亮。
- 图谱关系来自后端 JSON。
- 知识条目根据当前节点做简单关联展示。

当前节点：

- 输气场站。
- 站控柜。
- 工控机。
- 高温告警。
- 风道堵塞。
- 滤网积尘。
- 风扇异常。
- 恢复验证。

当前图谱关系：

- 输气场站包含站控柜。
- 站控柜包含工控机。
- 工控机包含风道、滤网、风扇。

- 高温告警与系统温度 >55°C、CPU 温度 >70°C 相关。
- 恢复验证要求连续观察 ≥10 分钟。

当前能力边界：

- 当前图谱是前端静态节点布局 + 后端 JSON 关系数据。
- 未接入真实图数据库。
- 未接入真实知识抽取。

## 16. 当前演示可讲闭环

当前 MVP 骨架已经可以演示以下路径：

1. 一线人员登录系统。
2. 进入智能诊断入口。
3. 输入工控机高温、风扇异常等现场现象。
4. 在“描述现场现象”页查看场站图和故障点。
5. 选择或自动识别设备型号、位置、角色和关联告警。
6. 修改或确认 TEMP/FAN、风扇转速、系统温度、CPU 温度阈值。
7. 确认接入信息并触发诊断。
8. 系统展示多 Agent 会诊动态过程。
9. Agent 完成后生成诊断结论。
10. 用户查看可能原因排序。
11. 进入步骤式检修向导。
12. 工程师按步骤逐项勾选检查项。
13. 图片随检查项切换，辅助工程师确认检查位置。
14. 完成恢复运行与验证。
15. 生成检修记录。
16. 后续可进入作业卡打印视图、专家审核和知识图谱。

## 17. 当前骨架大概要实现的功能定位

当前骨架应定位为：

“面向答辩演示的步骤式智能检修 MVP 骨架”。

它要展示的核心产品逻辑是：

1. 用户先描述问题。
2. 系统把自然语言问题转成结构化接入信息。
3. 系统模拟或调用智能诊断过程。
4. 诊断结论不只输出文字，而是转成可执行检修步骤。
5. 工程师按步骤勾选完成。
6. 每一步有图片辅助。
7. 最后形成检修记录和作业卡。
8. 知识图谱展示知识沉淀和赛题要求。

当前骨架重点是：

- 流程完整。
- 步骤清晰。
- 演示稳定。
- 可视化够用。
- 后续可接数据库和 API。

## 19. 当前结论

当前 MVP 已经完成了“从异常接入到诊断、检修、记录、知识图谱”的单场景闭环骨架。

它目前最适合承担：

- 比赛演示第一版。
- 产品流程验证。
- 和老师/队友讨论界面结构与交互逻辑。
- 后续接入模型、数据库、登录和更多设备场景的基础框架。

它还不是完整生产系统，关键限制是：流程预设、数据静态、Agent 模拟、无持久化、无真实模型能力、作业卡打印视图尚未完善。
