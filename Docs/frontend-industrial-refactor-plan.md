## 一、老师意见解析：他真正想表达什么

把聊天记录里的 8 条意见归并，老师实际只说了 **4 件事**：

### 1.1 「平铺功能 + 不像专用工具 + 与 LLM 搜索同质化」= 同一件事：**首页形态错了**

老师连发的三条（"把这些功能平铺开"、"界面与目前 LLM 搜索界面太像了"、"不像一个专用工具"），配的截图是 Quark 和通用 AI 搜索首页——他的意思非常具体：**一个居中大输入框 + 附件图标的首页，无论标题写什么，视觉基因就是 ChatGPT/Quark**。评委见过一百个这样的套壳项目，3 秒内就会归类为"通用问答换皮"。

他要的不是"加点工业风配色"，而是**信息架构层面的翻转**：打开系统第一眼看到的应该是"这个工具管什么设备、能处理什么故障、检修流程长什么样、最近有哪些工单"——输入框只是工作台上的一个入口，而不是整个首页。

### 1.2 「查 AB / 中控 / 霍尼韦尔 / 施耐德大型工控型号」= **设备对象太单薄，行业代表性不足**

老师看到你的研华小工控机图片后的反应是"查一下大型工控的型号"。他的潜台词：油气场站的真实主角是 **DCS 控制系统、大型 PLC 控制柜、成排的控制器机架和 I/O 卡件**，一台 4U 工控机撑不起"面向油气管道场站"的叙事。注意他没有说"换掉研华"，而是说"后面工控机图片查一下"——**补充行业代表性素材、证明系统可扩展到主流大厂设备**，主演示线仍然可以是研华（它的资料你已经吃透了，阈值、图号、备件号全有，换主线等于自杀）。

### 1.3 「没体现 Agent 如何参与 + 提问之后有思考过程」= **过程可见性不足**

老师认可检修步骤输出（"展现的检修步骤基本可以"），不满的是**从提问到结论之间是个黑箱**。他举的例子"现在提问之后，会有一个思考过程"，参照物是主流 LLM 产品的推理过程展示——但你不能照抄隐藏思维链，正确的翻译是：**可审计的诊断轨迹**。每个 Agent 是谁、收到什么输入、产出什么摘要、引用了哪条资料、处于什么状态——这是"多 Agent 协同"这个赛题亮点唯一能被评委看见的方式。你现在的三个 Agent 卡片只有进度条和流式文字，动画放完就消失，**既没有输入输出结构，也不可回看**，等于亮点白做。

### 1.4 「列出参考资料」= **证据链要成为一等公民**

每个判断、每个步骤、每张作业卡都要能回答"依据是什么"：手册名、页码、章节、设备型号、历史案例、专家经验、安全规范。你的知识条目 KB-001~KB-010 已经存在，缺的只是**把来源字段（手册名/页码/章节/审核状态）显性化，并绑定到结论、步骤和作业卡上**。

**一句话总结老师的诉求**：把"一个会聊天的 AI"改造成"一张摊开的检修工作台"——设备摆在台面上、流程贴在墙上、会诊过程有病历、每个结论有出处。

## 一·B 第二轮截图意见的精确拆解与对应方案

老师随后给出的三张截图（首页红框图、诊断结论页红字批注图、千问思考过程参照图）把第一轮意见落到了**具体元素级**。本节逐一拆解他真正指的是什么，并给出产品层面的对应方案（技术实现交由工程侧另行细化）。

### B.1 首页红框：老师圈的不是输入框，而是两个"套壳签名元素"

**老师的意思**：红框精准圈住了两样东西——输入框底部的附件图标行（添加材料/图片/视频/语音/发送）和下方"或试试这些常见问题"的提问 chips。这两样恰好是 ChatGPT/Quark 首页的两个签名特征。"把这些功能平铺开"因此有非常具体的所指：**这两处元素承载的能力要从"聊天附件/搜索建议"的形态里解放出来，升格为独立的业务模块**。

**对应方案（两处升格 + 一处顺手改）**：

1. **附件图标行 → "现场采集"业务模块**。图片/视频/语音在本项目里本来就是赛题核心的"多模态接诊能力"，却被压成输入框角落的三个通用小图标。改法：从输入框中拿出来，变成接诊卡旁的一组大尺寸采集入口卡，每张卡带**业务语义副标题**——不写"上传图片"，写"拍摄面板照片 · 识别指示灯与告警状态"；不写"视频"，写"现场视频 · 记录环境与风道遮挡"；不写"语音"，写"异响录音 · 辅助判断风扇异响特征"；不写"添加材料"，写"检修资料 · 附随工单归档备查"。**同样的预留入口，换个说法就从通用附件变成了领域能力**——这正是"结合行业需求、有领域特色"的字面执行。点击行为与现状一致，功能零改动。
2. **提问 chips → 故障任务卡**。现有四个 chips 的选题本身没问题，坏在问句措辞（"…怎么办？""…如何排查？"是向 AI 提问的口吻）和小胶囊形态（搜索建议词的视觉）。改法：升格为带图标、故障编号（FT-TEMP-01 式）、接入状态标（"已接入·支持完整闭环"/"模板建设中"）的任务卡；引导语从"或试试这些常见问题"改为"**快速接诊 · 按故障类型发起**"。点击行为不变。
3. **"早上好李师傅" → 值班信息条**。问候语 + 大输入框是消费级 AI 产品的开场白；工业系统的开场白是岗位上下文。改为一行值班信息条：`李师傅 · 一线检修 ｜ 某输气场站 · 站控区域 ｜ 在册设备 1 台 ｜ 待审核工单 1`，输入框标题由"智能诊断入口"改为"现场异常接诊"。

改后首屏结构：值班信息条 → 一行高的接诊输入框（无附件图标）→ 现场采集四宫格 → 故障任务卡 → 底部闭环流程条。与第四节工作台布局完全兼容。

### B.2 千问截图：老师要抄的是"可折叠思考组件"的骨架，内容全部换成工业语义

**老师的意思**：他圈的是千问回答前那个可展开的思考过程区块——折叠头（"已完成思考，参考了 10 篇材料"）、加粗分析小标题 + 一句推理、检索词 chips、资料卡列表。他要的不是照搬网页搜索结果，而是这种**"先见过程、再见结论，且过程可折叠回看"**的信息组织方式。

**对应方案：会诊依据折叠组件（千问骨架的工业化翻译）**：

| 千问的层 | 千问内容 | 本系统的工业化翻译 |
|---|---|---|
| 折叠头 | 已完成思考，参考了 10 篇材料 ∨ | **已完成会诊 · 引用 4 条检修依据 ∨**（可展开/收起） |
| 分析小节 | 加粗小标题 + 一句推理（"我会先查询…再补充…"） | 加粗小标题「研判散热异常方向」+ 一句业务推理「现象含风扇异响与温度告警，优先核对转速与温度判据，再排查风道与滤网」 |
| 检索词 chips | 🔍 搜索关键词若干 | **命中判据 chips**：KB-003 风扇<500rpm、KB-004 系统>55°C、KB-001 散热结构（等宽编号，点击联动证据卡） |
| 资料卡列表 | 带网站图标的网页结果 | **证据卡迷你态**：来源类型竖条 + 手册名 + §页码，来源类型图标（手册/规程/案例/专家）替代网站图标 |

该组件与第六节 AgentTimeline 是同一套数据的两种视图：每个 Agent 节点即一个"分析小节"。**与千问的关键差异要在文案上主动亮出来**（这也是答辩加分点）：千问引用的是网页，本系统引用的是带页码的手册判据和已审核知识条目——所以折叠头绝不写"参考了 N 篇材料"（搜索感），写"引用 N 条检修依据"（检修系统感）。

### B.3 红字批注"随着检修步骤推进，在右侧实时展现思考过程"：这是第二轮最重要的增量需求

**老师的意思**：注意批注的位置（指向诊断结论页右栏）和措辞（"随着检修步骤推进"）。这说明他要的**不是只在会诊阶段播一次的过程动画**，而是一条**贯穿全流程的右侧思考轨迹**：从接诊、会诊，到进入检修向导之后的每一步、每一个检查项勾选，右侧都应持续出现"系统此刻在想什么、依据是什么"。这是对"体现 Agent 参与"的升级——从"一段过程"升级为"全程陪同"。

**对应方案：全流程会诊轨迹流（右栏常驻）**：

- **位置**：右栏"诊断流程"列表下方，取代现有辅助对话的视觉位置（对话降级为底部折叠入口）。
- **触发逻辑**：用户的每个关键动作都对应一条预制轨迹——描述现象后出"结构化接诊"轨迹；确认阈值后出"判据核对"轨迹；向导中每勾选一个检查项，0.5 秒后滑入一条对应轨迹（如勾选"风扇滤网积尘"→ 轨迹"核对滤网状态判据：积尘不透光即判定堵塞，需更换风扇滤网 · 依据 KB-008 维护规程第 5 章"）。
- **每条轨迹的信息结构**：触发动作 + 思考标题 + 一句业务推理 + 引用依据编号（可点击联动证据卡）+ 状态。
- **分组折叠**：轨迹按阶段分组，每组头部即 B.2 的折叠头形态（"步骤 4 · 已核对 4 项判据 · 引用 3 条依据 ∨"），全程可回看。
- **演示效果**：工程师在中间勾检查项，右侧思考持续流动且每条带出处——"系统在陪着工程师干活"变成持续可见的事实。B.2 和 B.3 两条意见由同一套组件语言解决。
- **规模预估**：全流程约 25-30 条预制轨迹数据（每个检查项 1 条 + 各阶段节点若干），内容可直接从现有知识条目和向导步骤派生。

### B.4 "查大型工控机型号、想改成大型工控机的图片"：分层换图，换两层、保一层

**老师的意思**：他看到步骤里的研华小工控机图片后，觉得画面里的设备不够有油气行业代表性，希望换成 AB/中控/霍尼韦尔/施耐德那种大型工控系统的视觉。

**必须先识别的坑**：如果把检修步骤里的图直接换成大型 PLC/DCS 设备，**整条证据链会当场断裂**——本项目的故障闭环（TEMP/FAN 灯、<500 rpm 判据、门滤网备件号、FAN1/FAN2 接线、手册图号）全部锚定在研华手册上；换成 ControlLogix 机架后图与判据对不上，且大型 PLC 机架多为无风扇对流散热，连"风扇滤网"这个故障对象都不存在。评委一句"这个滤网备件号出自哪本手册"就会露馅。

**对应方案：三层换图策略**：

| 层 | 处理 | 说明 |
|---|---|---|
| ① 场景层（场站态势图、机房定位图） | **换** | 升级为"站控机房 + 成排大型控制柜"的视觉——这是评委对"油气场站"的第一印象层，必须有大型设备感 |
| ② 柜体层（站控柜 A01） | **换（关键一招）** | 把站控柜 A01 画成真实比例的大型控制柜：柜内自上而下为 PLC 机架层（ControlLogix/S7-1500 风格的机架 + 卡件线稿）、交换机层、**研华工控机（4U）层**、电源层。每次定位推近，评委看到的是大型工控柜，而检修对象只是柜内一层——**"大型"的观感有了，研华证据链一根没断**；且"工控机是柜内与 PLC 并存的数据采集终端"这一叙事本身更符合真实站控柜构成 |
| ③ 步骤细节层（向导第 1/4 步的前面板、滤网、风扇、接线图） | **保持不动** | 老师原话"检修步骤基本可以"，这一层锚定官方手册图号与备件号，是全项目最硬的资产 |

**版权提醒**：大厂设备实拍图版权敏感，柜体层建议自绘线稿（柜框 + 示意化机架卡件，参考公开产品图的形态但不照抄），研华层直接嵌入已有前面板图。

**向老师确认的口径（建议换图前先说这句话）**："老师，我计划把场站和控制柜的图升级成大型 DCS/PLC 控制柜（参考 AB、中控等），检修步骤的细节图仍锚定当前已核实手册的设备，保证每条依据可查——这样可以吗？"这句话本身就同时展示了"听懂了意见"和"守住了证据链"。

## 二、当前 MVP 最大问题判断

先回答你的五个追问，再给问题表。

**1. 为什么看起来像普通 LLM 搜索？** 因为首页的视觉主角是"居中大输入框 + 添加图片/语音/视频的附件图标行"——这是 ChatGPT/Quark 的标志性构图。人对界面的归类靠构图基因而不是文字内容，标题写"站控慧眼"也救不回来。

**2. 为什么不像专用工控检修工具？** 真实的工业运维系统（SCADA 上位机、DCS 操作站、EAM 检修系统）的首屏永远是**对象和状态**：设备列表、告警面板、工单队列、流程看板。你的首屏是"一个问题的入口"，没有对象、没有状态、没有队列——专用工具感 = 对象感 + 状态感 + 流程感，三者你首屏一个都没有。

**3. 哪些页面最容易被认为是通用 AI 套壳？** 按风险排序：① 首页诊断入口（最像，构图即 GPT）；② 多 Agent 会诊页（进度条 + 流式文字 = "AI 正在思考"的通用表达，且无结构化输入输出）；③ 右侧辅助对话框（悬浮聊天框是所有套壳的标配）。反而检修向导五步页是你最不像套壳的页面——老师也确认了这一点。

**4. 哪些功能做了但没体现出来？** 这是最可惜的部分：
- **知识条目有 KB 编号却没有页码/章节/来源类型** → RAG 证据链的说服力损失 80%；
- **三个 Agent 有分工却没有输入输出摘要** → 多 Agent 协同看起来像一个进度条动画；
- **异常接入四步是标准的结构化接诊** → 但入口是聊天框，评委以为你就是个聊天产品；
- **阈值确认页有官方判据** → 但没标注"来源：用户手册 Ed.6 p23"，白白浪费了你核实过的官方资料；
- **专家回流有完整闭环** → 但埋在流程最后，首页毫无露出。

**5. 如何不推倒重做地变专业？** 三个杠杆，成本从低到高：**改文案**（AI 词汇→行业词汇，半天）；**改首页**（输入框中心→工作台，1 天）；**改 Agent 展示**（进度条→结构化时间线，1 天）。三者做完，后端一行不用动。

### 4.3 推荐颜色与视觉风格

- **主色**：深蓝灰 `#1E2A3A`（顶部栏/左栏底色）+ 石墨白 `#F5F7F9`（内容区底色）。
- **强调色仅三档语义色**：告警红 `#D64545`、正常绿 `#2F9E6E`、待确认琥珀 `#D9822B`。禁止紫色渐变、禁止玻璃拟态、禁止大面积渐变。
- **卡片**：圆角 ≤8px、1px 边框 `#DDE3EA`、无重投影；标题左侧加 3px 语义色竖条（工业系统的经典视觉语言）。
- **字体**：数值与编号用等宽字体（`ui-monospace`），如「<500 rpm」「KB-003」「RX-0709-01」——等宽数字是仪表感的最廉价来源。
- **状态点**：所有对象/工单/Agent 前加 8px 圆点状态灯（红/绿/琥珀/灰），呼应工控面板指示灯语言。

### 4.4 如何避免 LLM 搜索感（构图层面的四条铁律）

1. **输入框绝不居中、绝不超过一行高、绝不带附件图标行**——附件图标挪进接诊流程第一步内部。
2. **首屏文字密度要高**：LLM 首页的特征是"空旷 + 一个框"，工业系统的特征是"密集的对象、编号、状态"。让首屏出现至少 10 个具体名词（设备型号、工单号、KB 编号、阈值数字）。
3. **出现队列和状态**："待审核 1 单""已闭环""建设中"——有队列 = 有业务，有业务 = 专用工具。
4. **底部流程条常驻**：评委扫一眼就知道这是七步闭环系统而不是问答框。

### 4.5 如何体现行业专用工具感

- 顶部栏常驻「某输气场站 · 站控区域 · 站控柜 A01」位置链——**位置链是运维系统的身份证**。
- 设备对象用**台账语言**：「在册 1 台」「资产编号 A01-IPC-01」。
- 任务卡用**故障代码化命名**：卡片角落加「FT-TEMP-01」式编号。
- 所有判据带出处角标：「风扇 <500 rpm · 判据源自原厂手册」。

## 六、Agent 参与过程展示方案（AgentTimeline 组件）

### 6.1 设计原则

不是聊天记录、不是隐藏思维链，而是**可审计的处理轨迹**：每个节点 = 一个业务模块，有输入摘要、输出摘要、引用、状态、耗时。演示态逐个点亮（复用现有定时器），完成态可整体回看。

### 6.2 Agent 阵容（从 3 个扩到 6 个，纯前端数据，成本≈0）

现有"分析诊断/操作合规/知识检索"三个 Agent 拆分重命名为六个，正好对应老师点名的分工，且顺序符合业务逻辑（检索先于诊断）：

接诊 → 设备识别 → 知识检索 → 工控专家 → 安全合规 → 作业卡

> 节奏控制：前两个各 1.5s（它们在接诊阶段其实已完成，进入会诊时快速点亮补录），后四个各 2.5-3s，总时长 12-14s，不冷场。

### 7.3 三处绑定关系（让证据"长"在业务上而不是堆在列表里）

1. **绑定结论**：结论页每条原因后跟证据徽标（`依据 KB-001 KB-003`）；
2. **绑定检修步骤**：向导每步底部现有"来源依据"行升级为 1-2 张迷你证据卡（只显示来源名 + 页码，点击展开）；
3. **绑定作业卡**：作业卡底部新增「本卡引用依据」区块，列出全部命中证据的编号 + 来源 + 页码 + 审核状态——打印视图同样输出，**这是评委翻作业卡时的决定性细节**。

### 7.4 与 RAG 结果绑定（数据层最小改动）

`knowledge_items.json` 每条追加 7 个字段（source_type / source_name / location / quote / fault_tags / trust_status / used_in），已有字段不动；`/api/knowledge/evidence` 接口无需改（透传即可）。前端 EvidenceCard 对缺字段做兜底显示，保证兼容。

### 7.5 如何体现"不是普通搜索结果"

普通搜索结果 = 标题 + 摘要 + 链接；你的证据卡 = **来源类型 + 精确页码 + 引用片段 + 审核状态 + 业务绑定**。答辩口径："系统的每条结论可以下钻到手册页码，专家修正的知识带审核标识回流——这是检修知识库，不是网页搜索。"
